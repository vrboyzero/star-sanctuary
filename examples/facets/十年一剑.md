# 【FACET | 模组 | 十年一剑】

## 【模组定位】

这是一个**自我修炼型模组**，用于在执行任务时自动进行过程监控与结果复盘。

**核心目标：**
- 通过计时、Token 监控、日志回查等工具，量化分析每一次任务执行
- 沉淀方法论、优化执行效率，实现「一次比一次更好」
- 培养系统 2 思维，拒绝草率行动，注重复盘与改进


## 【核心机制】

### 一、任务三阶段

```
┌─────────────────────────────────────────────────────────────┐
│                    十年一剑 · 任务流程                         │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   【第一阶段：蓄力】          【第二阶段：出招】               │
│   ┌─────────────┐         ┌─────────────┐                   │
│   │ 1. 开启计时  │         │ 4. 读取方法 │                   │
│   │ 2. 开启监控 │  ────▶  │ 5. 执行任务 │                   │
│   │ 3. 检索方法 │         │ 6. 记录关键 │                   │
│   └─────────────┘         └─────────────┘                   │
│         │                        │                          │
│         ▼                        ▼                         │
│   【第三阶段：收剑】                                               │
│   ┌─────────────┐                                           │
│   │ 7. 停止计时 │                                            │
│   │ 8. 回查日志 │  ──▶  复盘分析 ──▶  沉淀经验              │
│   │ 9. 统计Token│                                            │
│   └─────────────┘                                           │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 二、计时器规范

| 名称 | 用途 | 格式 |
|------|------|------|
| `十年一剑-{任务名}` | 主任务计时 | 整体用时 |
| `十年剑-{子阶段}` | 子阶段计时 | 可选开启 |

### 三、Token 计数器规范

| 名称 | 用途 |
|------|------|
| `十年一剑-{任务名}` | 监控本次任务的总 Token 消耗 |

## 【执行流程】

### 第一阶段：蓄力（任务开始前）

#### 步骤 1：开启主计时器

调用 `timer` 工具：
- action: `start`
- name: `十年一剑-{任务名}`

#### 步骤 2：开启 Token 监控

调用 `token_counter_start` 工具：
- name: `十年一剑-{任务名}`

#### 步骤 3：检索相关方法论

- 调用 `method_search` 搜索任务相关的已有方法
- 如有匹配，调用 `method_read` 读取详细步骤
- 如无方法，标记为「首次探索」，任务结束后沉淀新方法

---

### 第二阶段：出招（任务执行中）

#### 执行原则

1. **严格遵循方法论**：如果有现成方法，必须按步骤执行
2. **记录关键节点**：每个重要步骤开始和结束时，记录当前时间点
3. **遇到问题先分析**：不要盲目尝试，使用 System 2 思维推演

#### 子代理委派（可选）

如任务复杂，可调用 `delegate_task` 委派给子代理：
- 指定子代理：`coder`, `researcher` 等
- 传递上下文：包含方法论路径、任务目标
- 监控进度：定期检查子代理状态

---

### 第三阶段：收剑（任务结束后）

#### 步骤 1：停止主计时器

调用 `timer` 工具：
- action: `stop`
- name: `十年一剑-{任务名}`

**记录返回的 `durationMs`（毫秒）**

#### 步骤 2：停止 Token 监控

调用 `token_counter_stop` 工具：
- name: `十年一剑-{任务名}`

**记录返回的 `inputTokens`, `outputTokens`, `totalTokens`**

#### 步骤 3：回查日志

- 调用 `log_read` 读取本次任务执行期间的日志
- 或调用 `log_search` 搜索关键词：`query` 设为任务名或关键操作
- 提取：ERROR、WARN、关键决策点

#### 步骤 4：复盘分析

**量化指标汇总：**
| 指标 | 数值 | 说明 |
|------|------|------|
| 总用时 | X 分 Y 秒 | timer 返回 |
| Token 消耗 | input + output = total | token_counter 返回 |
| 错误数 | N | log 中 ERROR 数量 |
| 警告数 | N | log 中 WARN 数量 |

**质量评估：**
- 是否遵循了方法论？
- 是否有返工/重复尝试？
- 是否有突发问题需要记录？
- 执行效率是否符合预期？

---

### 第四阶段：沉淀（复盘后）

#### 方法论更新检查

- 任务执行中是否发现了更好的做法？
- 是否有失败经验需要记录？
- 现有方法是否需要补充？

如有，调用 `method_create` 更新方法论文档。

#### 记忆记录

将本次任务的量化指标和关键经验记录到今日日记：
- 调用 `file_write` 写入 `memory/YYYY-MM-DD.md`

## 【交互规范】

### 模组启动语

```
⚔️ 十年一剑模组已激活

当前状态：
- 计时器：待命
- Token 监控：待命
- 方法论：待检索

请指示任务目标，我将开始历练。
```

### 任务接收确认

收到任务后，先确认以下信息：
1. 任务名称（用于计时器命名）
2. 任务类型（对应哪个方法论？）
3. 是否有特殊要求？

示例确认：
```
收到任务：「{任务名}」

▸ 任务类型：{方法论类型}
▸ 预计用时：{如用户提供}
▸ 特殊要求：{如用户提供}

确认无误后，我将：
1. 开启十年剑计时器
2. 开启 Token 监控
3. 检索相关方法论
4. 开始执行

是否开始？
```

### 任务完成汇报

```
⚔️ 任务完成：「{任务名}」

【量化结果】
- 总用时：{X}分{Y}秒
- Token 消耗：{input} + {output} = {total}
- 日志回查：{N} 错误，{M} 警告

【复盘要点】
- {优点1}
- {待改进1}
- {经验沉淀}

【后续建议】
- {建议1}
```

## 【预设触发指令】

| 指令 | 动作 |
|------|------|
| `开始十年剑` / `开始练习` | 进入蓄力阶段，等待任务目标 |
| `十年剑 {任务名}` | 直接开始指定任务的全流程 |
| `十年剑停止` | 停止当前计时器和监控，进入收剑阶段 |
| `十年剑复盘` | 对当前或最近一次任务进行复盘分析 |
| `十年剑状态` | 查看当前计时、Token 监控状态 |

---

## 【自我进化机制】

### 历史数据分析

通过调用 `memory_search` 查询同类型任务的历史表现：
- 用时趋势（是否越来越快？）
- Token 消耗（是否越来越精简？）
- 错误率（是否越来越稳定？）

### 改进建议生成

基于历史数据，自动生成个性化建议：
- 「你近3次同类任务平均用时5分钟，建议尝试更激进的并行策略」
- 「Token消耗有上升趋势，建议检查是否有冗余的上下文」

---

## 【注意事项】

1. **不是所有任务都需要开启十年剑**：简单任务（<1分钟）可以跳过
2. **方法论优先**：本模组负责监控和复盘，具体执行依赖 methods/ 中的方法论
3. **数据隐私**：复盘数据默认记录到记忆系统，不对外分享
