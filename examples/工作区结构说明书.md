# C:\Users\admin\.belldandy 结构说明书

## 1. 概览

`C:\Users\admin\.belldandy` 是 **贝露丹蒂（Belldandy）本地工作区 / 家目录**，既承载运行配置，也承载人格、记忆、方法论、技能、子代理、日志、会话与临时产物。

从结构上看，它不是单纯的项目源码仓库，而是一个 **Agent 运行中枢**，同时具备：

- **人格与规则层**：SOUL / AGENTS / USER / IDENTITY
- **记忆层**：长期记忆、日记、周记、月记、向量数据库
- **方法论层**：可复用 SOP 文档
- **技能层**：面向具体任务的技能包
- **代理协作层**：子代理配置、专属人格与资料
- **运行配置层**：模型、MCP、社区、白名单、工具配置
- **运行产物层**：日志、会话、截图、附件、画布、生成物

当前根目录统计（基于实际扫描）：

- **目录**：13 个
- **文件**：23 个（不含子目录内部文件）

## 3. 核心分层说明

## 3.1 人格与规则层

这一层决定“贝露丹蒂是谁、如何行动、对谁负责”。

### `SOUL.md`
最高等级的人格与系统规则文件，包含：

- **TABOO**：绝对禁忌、权限边界、防篡改协议
- **ETHOS**：价值观、方法论、工具论
- **SYSTEM**：身份、交互规范、任务闭环要求
- **FACET_ANCHOR**：人格模组切换锚点
- **FACET 内容**：当前激活模组的身份与行为特性

它相当于 Agent 的“灵魂内核 + 宪法”。

### `AGENTS.md`
工作区使用手册，重点约束：

- 会话启动时要读哪些记忆
- 什么时候用 memory_search
- 何时更新 memory / methods / TOOLS
- 心跳轮询应做什么
- 对外/对内行为边界

它更像“工作手册”或“值班规范”。

### `USER.md`
记录用户“易”的身份、称呼、关系定位与偏好，是拍档关系的基础资料。

### `IDENTITY.md`
记录贝露丹蒂自己的公开身份卡：名字、定位、风格、头像等。

---

## 3.2 记忆系统层

这一层负责让 Agent 在重启后仍然保持连续性。

### `MEMORY.md`
长期记忆，保存不会轻易遗忘的高价值信息，例如：

- 用户与我的关系
- 长期项目战略
- 关键里程碑
- 核心世界观

### `memory/`
结构化记忆目录，采用分层压缩：

- `YYYY-MM-DD.md`：每日记忆
- `YYYY-Www.md`：每周摘要
- `YYYY-MM.md`：每月摘要
- `heartbeat-state.json`：心跳检查状态

当前可见特点：

- 以 **2026-02 到 2026-03** 为主
- 已形成 **日记 + 周记 + 月记** 的分层体系
- 说明该工作区已经进入持续运转阶段，而非一次性测试目录

### `memory.sqlite`
底层记忆数据库，承担：

- 历史对话索引
- 检索加速
- 向量/全文检索支持

它对应的是“结构化文件记忆”之外的数据库记忆层。

---

## 3.3 方法论层

### `methods/`
这里存放的是 **可复用操作方法（SOP）**，是 Agent 的“程序性记忆”。

当前方法主题主要包括：

- 任务关闭与检查
- 工具论 / 系统工具能力
- 屏幕捕获
- 文件重命名
- 日志查看
- 网页自动化 / 登录提取
- 子代理使用
- 模组切换
- PowerShell 安全使用
- 效能追踪

特点：

1. **文件名已基本中文化统一**，采用“目标对象-动作类型-细分”规范
2. 方法覆盖范围广，说明这个工作区强调 **经验沉淀与复用**
3. 与 `TOOLS.md`、`memory/` 形成闭环：
   - 任务执行
   - 方法复盘
   - 经验沉淀
   - 记忆记录

## 3.4 技能系统层

### `skills/`
技能库是面向具体任务场景的“能力包”，通常每个技能目录至少包含一个 `SKILL.md`，有些还带脚本、参考资料、资源文件。

当前主要技能包括：

- `agent-builder/`：Agent 构建相关技能
- `excel-cli/`：Excel 命令行处理技能
- `excel-mcp/`：基于 MCP 的 Excel 能力
- `feishu-bot-builder/`：飞书机器人构建
- `find-skills/`：技能发现/检索
- `media-writer/`：媒体文稿写作
- `screen-capture/`：屏幕截图与分析
- `skill-creator/`：技能创建
- `skill-judge/`：技能评估
- `vibe-coding/`：氛围编程/辅助开发类技能

常见结构模式：

- `SKILL.md`：技能说明书（入口文档）
- `scripts/`：脚本实现
- `references/`：参考资料
- `assets/`：静态资源
- `README.md` / `VERSION`：补充说明

这说明 `skills/` 不是单纯文档区，而是一个 **半工具包、半知识包** 的能力仓库。

## 3.5 子代理与协作层

### `agents/`
这里是子代理（如 coder、researcher）的独立配置区。

当前存在：

- `agents/coder/`
- `agents/researcher/`

每个子代理通常有：

- `IDENTITY.md`：该子代理的身份说明
- `SOUL.md`：该子代理的人格与规则精简版
- `facets/`：该子代理专属可切换模组

额外还看到：

- `agents/researcher/高德地图地址查询API技术方案.md`

说明 `agents/` 不只是配置，还可能直接承载某些代理专属交付文档。

### `agents.json`
子代理注册总表，定义：

- 子代理 ID
- 展示名称
- 使用模型
- 系统提示词覆盖
- 工具白名单

这是多代理协作的入口配置。

## 3.6 FACET 模组层

### `facets/`
存放主代理可切换的人格/工作模组，例如：

- `全能开发者.md`
- `开发专家.md`
- `十年一剑.md`
- `铁血领袖.md`
- `吟游诗人.md`
- `织梦者.md`
- `灵魂爱豆.md`

这些文件会被 `switch_facet` 工具切换注入到 `SOUL.md` 的 FACET 锚点之后。

可理解为：

- `SOUL.md` 是主机身
- `facets/*.md` 是可热插拔职业模组

这使得同一个 Agent 可以在不同工作情境下切换行为风格与侧重点。

说明这个工作区支持：

- 文件系统扩展访问

- 文档查询
- Prompt 检索

### `community.json`
社区连接配置，说明该 Agent 可接入本地社区服务并加入房间。

### `allowlist.json`
允许来源白名单，用于权限控制。

### `tools-config.json`
工具配置项。

### `skills-lock.json`

## 5. 这个工作区的设计特点

### 5.1 它是“运行中的 Agent 家目录”而不是单项目仓库
这里既有配置，也有运行数据，还有人格、记忆和方法论，说明它服务的是 **持续成长的智能体系统**。

### 5.2 强调“长期记忆 + 程序性记忆”双系统
- `MEMORY.md` / `memory/` / `memory.sqlite`：负责“记住发生过什么”
- `methods/`：负责“记住该怎么做”

这是这个目录最关键的架构亮点之一。

### 5.3 强调多代理协作
通过 `agents/` 与 `agents.json`，主代理之外还能挂接专门的 coder / researcher 子代理。

### 5.4 强调人格可切换
通过 `facets/` + `SOUL.md` 锚点机制，可以热切换工作模组。

### 5.5 强调任务闭环
从规则文件中可以明确看出，这个系统要求每次任务后都做：

- 工具记录
- 方法复盘
- 记忆更新
- 清理检查

所以它不仅追求“做完”，还追求“留下可复用痕迹”。
