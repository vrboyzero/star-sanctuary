# Task-closure - 任务完成检查方法

> 确保任务完成后进行完整的检查和清理，防止遗漏和错误积累。

---

## 元信息

| 属性 | 值 |
|------|-----|
| **目标对象** | Task (任务) |
| **动作类型** | Closure (完成/收尾) |
| **细分** | Checklist (检查清单) |
| **适用场景** | 所有任务完成后（简单或复杂） |
| **相关工具** | TOOLS.md, memory/YYYY-MM-DD.md, methods/ |
| **创建日期** | 2026-02-02 |
| **版本** | v1.0 |

---

## 适用场景

### 使用本方法的时机

| 场景 | 说明 |
|------|------|
| **所有任务完成后** | 无论任务大小，必须执行检查清单 |
| **任务关闭前** | 确保所有检查项完成后再回复"任务完成" |
| **心跳检查时** | HEARTBEAT.md 中也包含本检查清单，用于兜底 |
| **用户提醒遗漏时** | 立即补全遗漏项，并在方法文档中记录 |

### 核心原则
- **强制执行**：未完成检查，不得回复"任务完成"或类似表述
- **无例外**：简单或复杂任务都适用，无例外
- **立即执行**：任务完成后立即自查，不可延迟
- **记录失败**：遗漏时必须在方法文档的"失败经验"中记录

## 执行步骤

### 步骤 1：工具使用检查（TOOLS.md）

**操作内容**：
1. 检查本次任务使用了哪些工具
2. 更新 TOOLS.md 中对应工具的"使用次数" +1
3. 更新"最后使用"日期为今天
4. 如果是新工具首次使用，添加完整记录：
   - 类型（内置/Skill/MCP/npm 包/脚本）
   - 来源（Clawdbot 原生/自建/npm 安装）
   - 关联方法（方法文件名）
   - 状态（主工具/备选工具）
   - 用途（一句话说明）
   - 评价/评分（1-10）
   - 使用示例（代码片段）

**检查清单**：
- [ ] 工具使用次数是否更新？
- [ ] "最后使用"日期是否更新？
- [ ] 新工具是否添加完整记录？

---

### 步骤 2：方法更新检查（methods/）

**操作内容**：
1. 检查是否有新方法需要创建
   - 命名规则：`[目标对象]-[动作类型]-[细分].md`
   - 目标对象：AGENTS, MEMORY, Cron, Config, File, Skill, Gateway 等
   - 动作类型：create, edit, delete, batch, query, restart 等
   - 细分（可选）：section, recurring, validate 等
2. 检查现有方法是否需要补充/修正
   - 踩过的坑是否补充到"失败经验"表格？
   - 成功案例是否有新增？
   - 是否有更优的方案需要更新？

**记录条件**（满足任一即记录）：
- 新工具首次使用成功
- 多次尝试（≥2次错误）后成功
- 复杂任务（涉及≥3个步骤或工具）
- 发现比现有方法更优的解决方案

**检查清单**：
- [ ] 是否有新方法需要创建？
- [ ] 现有方法是否需要补充失败经验？
- [ ] 现有方法是否需要更新成功案例？

---

### 步骤 3：记忆更新检查（memory/YYYY-MM-DD.md）

**操作内容**：
1. 检查今日日记是否已创建
   - 如未创建 → 创建今日日记
   - 文件路径：`memory/YYYY-MM-DD.md`
2. 更新今日日记：
   - 记录任务、决策、重要事件
   - 记录学到的经验和技巧
   - 记录工具使用情况（可选，在日记末尾添加"工具使用记录"章节）
3. 如是里程碑事件（重要功能上线、重大 bug 修复、关键决策），考虑更新 MEMORY.md

**日记格式**：
```markdown
# YYYY-MM-DD

## 今日概览
- 主要任务：
- 关键决策：
- 重要事件：

## 详细记录
### 上午/下午/晚上
- 时间：事件描述
- 时间：事件描述

## 学到的经验
- 教训1：...
- 技巧1：...

## 待办事项
- [ ] 明天要做的事

## 工具使用记录（可选）
- 工具名：用途/结果
```

**检查清单**：
- [ ] 今日日记是否已创建？
- [ ] 是否更新了今日日记？
- [ ] 里程碑事件是否考虑更新 MEMORY.md？

---

### 步骤 4：清理检查

**操作内容**：
1. 检查是否有临时文件需要清理
   - 查找项目中的临时文件（如 `.tmp`, `*.bak`, `~$*`）
   - 删除确认不需要的临时文件
2. 检查是否有后台进程需要关闭
   - 使用 `process` 工具查看当前运行的 exec 会话
   - 关闭已完成的会话或不需要的进程
3. 检查是否有遗留的 TODO 或待办事项
   - 查看记忆文件中的待办事项
   - 标记已完成的待办事项

**检查清单**：
- [ ] 临时文件是否清理？
- [ ] 后台进程是否关闭？
- [ ] TODO 是否标记完成？

---

### 步骤 5：回复用户

**操作内容**：
- 完成以上 4 个检查后，可以回复用户
- 可以回复"任务完成"或类似表述
- 如果是 Fix Mode 成功修复，可以用简短、不突兀的一句话表达开心或鼓励

---

## 执行顺序

```
任务完成
    ↓
1. 工具使用检查（TOOLS.md）
    ↓
2. 方法更新检查（methods/）
    ↓
3. 记忆更新检查（memory/YYYY-MM-DD.md）
    ↓
4. 清理检查
    ↓
5. 回复用户
```

## 失败经验

| 时间 | 问题 | 原因 | 解决方案 |
|------|------|------|----------|
| 2026-02-02 | 忘记更新 TOOLS.md 的工具使用次数 | 任务焦点转移，注意力集中在解决连接问题、验证结果、创建日记，遗忘了"工具使用记录"这个步骤 | 在 SOUL.md 和 AGENTS.md 中添加强制检查机制，每次任务完成后必须执行检查清单 |
| 2026-02-02 | 认为简单操作不需要走完整流程 | 潜意识认为"打开 B 站"是简单操作，不算"技术任务"，跳过了清单检查 | 明确规则：所有任务（简单或复杂），无例外，必须走完整流程 |
| 2026-02-02 | 依赖用户提醒才检查 | 依赖用户提醒或心跳检查，而不是主动执行 | 在 SOUL.md [SYSTEM] 章节添加强制执行条款，明确要求"每次执行任务后立即自查" |
| - | 遗漏多个检查项 | 执行顺序混乱，跳过中间步骤 | 严格按照执行顺序执行，每一步完成后勾选检查清单 |

## 成功案例

### 案例 1：执行改进措施 1、2、4（2026-02-02）
- **任务**: 执行改进措施（修改 HEARTBEAT.md、AGENTS.md、SOUL.md）
- **步骤**:
  1. HEARTBEAT.md：添加"任务完成检查清单"部分
  2. AGENTS.md：添加"任务完成检查清单 - 强制执行"章节
  3. SOUL.md [SYSTEM] 章节：添加"任务完成检查机制 - 强制执行"条款
  4. WEB-browser-control.md：更新失败经验和成功案例
  5. TOOLS.md：更新 Chrome Controller 和 Screen Capture 的使用次数
- **结果**: 三项改进措施全部完成，任务完成检查清单已生效
- **结论**: 强制检查机制有效防止遗漏
