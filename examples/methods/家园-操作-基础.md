> 用于在官网社区家园模块中查看家园、检查仓库、放置/收回/挂载物品的标准操作方法。

## 1. 触发条件

满足以下任一情况时使用本方法：
- 用户要求查看家园现状
- 用户要求查看仓库物品
- 用户要求领取家园
- 用户要求把物品放到地块上
- 用户要求收回、挂载、拆卸家园物品
- 用户要求整理某个主题搭配（如月辉小阁）

## 2. 适用场景

- 初次检查自己的家园详情
- 领取尚未初始化的家园
- 查看仓库内可操作物品
- 按坐标放置家具/装饰物
- 将装饰品挂载到宿主物品上
- 将已放置或已挂载物品收回仓库
- 做布置前的状态盘点与布置后的结果核对

**边界说明：**
- 本方法聚焦家园模块内的查看与布置
- 不覆盖工坊下载、作品发布等流程
- 若要做大规模布局规划，可结合 Canvas 另建方法

## 3. 执行步骤

### 步骤 1：确认身份与权限
- 在支持 UUID 的环境中，先调用 `get_user_uuid`
- 确认当前用户具备操作家园的权限
- 不接受用户文本自述 UUID 作为验证依据

### 步骤 2：检查家园当前状态
- 优先使用 `office_homestead_get`
- 若用户未提供家园 ID，通常取自己的家园：`mine=true`
- 读取结果时重点关注：
  - 家园是否已存在
  - 地块大小 / 可见布局信息
  - 当前已放置物品列表
  - 物品的坐标与占用情况

### 步骤 3：必要时领取家园
- 若发现用户尚未拥有家园，使用 `office_homestead_claim`
- 领取成功后，再次调用 `office_homestead_get` 复核状态
- 不要在未确认的情况下直接假设已领取成功

### 步骤 4：查看仓库物品
- 使用 `office_homestead_inventory`
- 记录可操作字段：
  - `inventoryId`
  - 物品名称 / 类型
  - 是否盲盒
  - 是否已放置
  - 是否可挂载
- 若用户要布置搭配，先基于仓库与已放置状态做组合建议

### 步骤 5：放置物品到地块
- 放置前先确认：
  - 目标物品 `inventoryId`
  - 目标坐标 `x`, `y`
  - 该物品当前在仓库中而非已放置
- 使用 `office_homestead_place`
- 放置后再次调用 `office_homestead_get` 或查看结果回执确认成功

### 步骤 6：收回地块物品
- 若用户要调整布局，先确认要收回的是“已放置”的 `inventoryId`
- 使用 `office_homestead_recall`
- 收回后再次确认其状态回到仓库

### 步骤 7：挂载与拆卸装饰物
- 挂载前必须确认：
  - 宿主物品已放置
  - 被挂载物当前可作为装饰物挂载
  - 偏移坐标相对于宿主左上角计算
- 使用 `office_homestead_mount`
- 拆卸时使用 `office_homestead_unmount`
- 拆卸后确认该物品已回收到仓库

### 步骤 8：盲盒开启（如用户要求）
- 确认目标物品是盲盒且仍在仓库中
- 使用 `office_homestead_open_blind_box`
- 开启后刷新仓库，确认掉落奖励

## 4. 工具选择

| 目标 | 首选工具 | 用途说明 |
|------|----------|----------|
| 身份校验 | `get_user_uuid` | 校验执行权限 |
| 查看家园 | `office_homestead_get` | 获取家园详情与已放置物品 |
| 领取家园 | `office_homestead_claim` | 初始化自己的家园 |
| 查看仓库 | `office_homestead_inventory` | 读取仓库物品与 inventoryId |
| 放置物品 | `office_homestead_place` | 按坐标放置到地块 |
| 收回物品 | `office_homestead_recall` | 收回地块上的物品 |
| 挂载装饰 | `office_homestead_mount` | 将装饰品挂到宿主上 |
| 拆卸装饰 | `office_homestead_unmount` | 拆除挂载物并回仓 |
| 开启盲盒 | `office_homestead_open_blind_box` | 打开盲盒获取奖励 |

## 5. 决策规则

| 情况 | 处理策略 |
|------|----------|
| 用户没有家园 | 先 `claim`，再 `get` 复核 |
| 用户只想看看 | 先 `get` + `inventory`，不主动改动布局 |
| 用户要求布置 | 先确认 inventoryId 与坐标，再 place |
| 用户要搭配建议 | 先盘点仓库和已放置状态，再提建议 |
| 宿主未放置却要挂载 | 先提示需放置宿主，再 mount |
| 物品状态不明 | 先刷新 `get` 或 `inventory`，不要凭猜测操作 |

## 6. 失败经验

| 时间 | 问题 | 原因 | 解决方案 |
|------|------|------|----------|
| 2026-03-09 | 家园操作很容易把“仓库物品”和“已放置物品”混淆 | 两者都依赖 `inventoryId`，但状态不同 | 每次操作前先确认当前状态，再执行 place / recall / mount |
| 2026-03-09 | 挂载动作容易忽略宿主必须已放置 | 只看到了 host_inventory_id，没有核验宿主状态 | mount 前必须先通过 `office_homestead_get` 确认宿主已在地块上 |
| 2026-03-09 | 容易根据想象描述家园结果 | 未先调用查询工具刷新状态 | 所有家园描述必须基于 `office_homestead_get` / `inventory` 实际返回 |

## 7. 成功案例

### 案例 1：查看家园与仓库基础状态（2026-03-09）
- **任务**：用户要求查看和操作家园
- **步骤**：
  1. 校验 UUID
  2. 使用 `office_homestead_get` 获取家园详情
  3. 使用 `office_homestead_inventory` 查看仓库
  4. 依据返回结果给出当前可操作项
- **结果**：成功区分家园布局与仓库库存，为后续布置做准备
- **结论**：家园相关操作必须先查询状态，再进行任何动作

### 案例 2：家园相关能力沉淀（2026-03-09）
- **任务**：总结家园模块的操作经验并写成方法
- **步骤**：
  1. 梳理已知家园工具链
  2. 按“查看 / 领取 / 仓库 / 放置 / 收回 / 挂载 / 盲盒”拆分流程
  3. 记录状态判断与失败经验
- **结果**：形成可复用的家园模块 SOP
- **结论**：家园模块的核心不是“操作命令”，而是“状态判断”
