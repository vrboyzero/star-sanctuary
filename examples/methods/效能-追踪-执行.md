## 1. 为什么需要这套机制

### 1.1 起源：惨痛的教训

本方法源于多次任务执行后的疏忽遗漏：

| 时间 | 任务 | 遗漏内容 | 后果 |
|------|------|---------|------|
| 2026-02-15 | 日志分析功能 | 未更新 `TOOLS.md` | 工具使用记录断层 |
| 2026-02-20 | 定时任务配置 | 未检查 `memory/` | 当日记忆未记录 |
| 2026-02-22 | 子代理测试 | 未清理后台进程 | 资源泄漏 |

**核心洞察**：任务"功能完成" ≠ 任务"真正完成"。遗漏收尾工作会导致：
- 知识无法沉淀（方法/经验丢失）
- 记忆断层（上下文断裂）
- 资源浪费（进程/文件残留）

### 1.2 设计目标

1. **零遗漏**：确保每个任务的收尾工作都被执行
2. **可度量**：用数据说话，追踪真实效率
3. **可改进**：通过复盘持续优化流程

## 2. 三维效能指标体系

### 2.1 时间维度 (Time)

| 指标 | 定义 | 工具 | 目标 |
|------|------|------|------|
| 任务耗时 | 从开始到结束的总时间 | `timer` | 越短越好 |
| 思考时间 | 规划/分析阶段的耗时 | 估算 | 合理即可 |
| 执行时间 | 实际操作阶段的耗时 | `timer` | 高效稳定 |

### 2.2 错误维度 (Error)

| 指标 | 定义 | 追踪方式 | 目标 |
|------|------|---------|------|
| 报错次数 | 执行过程中的错误/异常 | `log_search` + 手动记录 | 0 |
| 回滚次数 | 撤销/重做的操作次数 | 手动记录 | 0 |
| 修正次数 | 发现错误后的修复次数 | 手动记录 | 0 |

### 2.3 资源维度 (Token)

| 指标 | 定义 | 工具 | 优化方向 |
|------|------|------|---------|
| Input Tokens | 输入消耗的 Token 数 | `token_counter` | 精简提示 |
| Output Tokens | 输出消耗的 Token 数 | `token_counter` | 控制长度 |
| Total Tokens | 总消耗 | `token_counter` | 整体优化 |

### 2.4 效能公式

```
效能评分 = (1 / 耗时) × (1 / (错误次数 + 1)) × (1 / Token消耗)
```

**解读**：
- 耗时越短，效能越高
- 错误越少，效能越高（+1 避免除零）
- Token 消耗越少，效能越高

## 3. 执行流程（四阶段）

### 阶段一：任务开始前

1. **启动计时器**
   ```bash
   # 示例
   timer start task_name
   ```

2. **启动 Token 计数器**
   ```bash
   # 示例
   token_counter_start task_name
   ```

3. **明确任务边界**
   - 什么是"完成"？（见第4节 三级完成标准）
   - 预计耗时？
   - 涉及哪些工具/方法？

### 阶段二：任务执行中

1. **实时记录错误**
   - 每次报错立即记录：时间、错误信息、解决方案
   - 使用 `log_search` 追踪系统日志

2. **保持专注**
   - 避免多任务切换
   - 遇到阻塞立即标记，不要硬撑

### 阶段三：任务完成后（强制检查清单）

> ⚠️ **禁止条件**：未完成以下检查，不得回复"任务完成"

| 检查项 | 具体内容 | 检查方式 |
|--------|---------|---------|
| **工具使用检查** | 更新 `TOOLS.md` 中对应工具的"使用次数"+1，更新"最后使用"日期 | 文件编辑 |
| **方法更新检查** | 检查 `methods/` 是否需要创建/更新方法文档，补充失败经验 | 目录检查 |
| **记忆更新检查** | 检查/创建 `memory/YYYY-MM-DD.md`，记录任务、决策、经验 | 文件检查 |
| **清理检查** | 清理临时文件、关闭后台进程、处理遗留 TODO | 系统检查 |

### 阶段四：效能对账（数据归档）

1. **停止计时器**
   ```bash
   timer stop task_name
   ```

2. **停止 Token 计数器**
   ```bash
   token_counter_stop task_name
   ```

3. **填写效能对账单**

   **效能对账单模板**：
   ```markdown
   ### 任务：XXX
   - **时间**：2026-03-07 14:30 ~ 15:45（耗时 75 分钟）
   - **错误**：
     - 报错 1 次：`log_search` 忘记填 query 参数
     - 修正 1 次：重新执行
   - **Token**：Input 2,500 / Output 1,200 / Total 3,700
   - **效能评分**：中等（有改进空间）
   - **经验**：
     - ✅ 成功经验：提前规划减少了返工
     - ❌ 失败经验：`log_search` 参数检查不仔细
   ```

## 4. 三级完成标准

| 级别 | 标准 | 适用场景 |
|------|------|---------|
| **Level 1 - 功能完成** | 代码/功能可用，测试通过 | 快速验证 |
| **Level 2 - 文档完成** | + 方法/记忆已更新 | 常规任务 |
| **Level 3 - 效能归档** | + 效能对账单已填写 | 重要任务 |

**默认要求**：所有技术任务至少达到 **Level 2**，重要任务必须达到 **Level 3**。

## 5. 失败经验与改进

### 5.1 常见失败模式

| 失败模式 | 原因 | 预防措施 |
|---------|------|---------|
| "我以为删了" | 口头确认但未实际执行 | 执行后立即验证（如 `ls` 检查） |
| "清单太长不想做" | 检查项过多，产生抗拒 | 精简清单，聚焦核心（4项） |
| "做完就忘了记录" | 没有及时记录，事后遗忘 | 执行中实时记录，不要事后补 |
| "Token 计数器忘开了" | 流程执行不严格 | 制作检查清单，逐项勾选 |

### 5.2 元经验：如何改进这套机制本身

1. **定期复盘**：每月回顾效能对账单，寻找模式
2. **简化流程**：如果某一步总是"无内容可记"，考虑删除
3. **工具化**：考虑将检查清单制作成自动化脚本
