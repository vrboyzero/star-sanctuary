## 📖 标准操作流程（SOP）

### 阶段一：连接与准备

```
Step 1: 确认扩展连接状态
- 首次尝试获取页面信息
- 如果失败，坦诚告知用户连接问题
- 询问用户是否已安装扩展并点击 "Attach Tab"
- 等待用户确认后再继续
```

**关键原则**：**连接问题要坦诚告知，不要假装在操作**

### 阶段二：登录操作

#### Step 2.1: 访问登录页

```bash
browser action=navigate url="http://目标网站.com/login"
```

#### Step 2.2: 识别页面元素

```bash
browser action=snapshot profile=chrome maxChars=50000
```

**必须识别的元素**：
- 账号输入框 ID
- 密码输入框 ID
- 验证码图片
- 登录按钮 ID

#### Step 2.3: 填写登录信息

```bash
browser action=act profile=chrome request={"kind":"type","ref":"账号框ID","text":"用户名"}
browser action=act profile=chrome request={"kind":"type","ref":"密码框ID","text":"密码"}
browser action=act profile=chrome request={"kind":"type","ref":"验证码框ID","text":"验证码"}
browser action=act profile=chrome request={"kind":"click","ref":"登录按钮ID"}
```

#### Step 2.4: 确认登录成功

```bash
browser action=snapshot profile=chrome maxChars=50000
```

### 阶段三：数据提取

#### Step 3.1: 导航到数据页面

```bash
browser action=navigate url="http://目标网站.com/data-page"
# 或
browser action=act profile=chrome request={"kind":"click","ref":"菜单ID"}
```

#### Step 3.2: 获取数据并分析

```bash
browser action=snapshot profile=chrome maxChars=50000
```

**提取维度**：
- 销量/金额排行（TOP N）
- 类目分布
- 时间趋势
- 关键洞察

## ❌ 失败经验与教训

### 经验一：首次连接失败处理

**正确做法** ✅：
```
1. 坦诚告知："Star Sanctuary Browser Relay 当前未连接"
2. 解释原因并询问用户是否已连接
3. 等待用户确认后再继续
```

**错误做法（避免）** ❌：
```
1. 不告知用户，反复尝试失败操作
2. 模糊地说"网络问题"而不解释原因
```

### 经验二：验证码识别

**正确做法** ✅：
```
1. 获取登录页快照
2. 从快照中识别验证码
3. 告知用户识别结果，确认后再填写
```

**可以改进** ⚠️：
```
应该明确说："识别到的验证码是：XXX，是否正确？"
让用户确认后再填写，避免识别错误
```

### 经验三：登录后页面确认

**本次可改进** ⚠️：
```
登录后应该先获取完整页面快照，再汇报状态
而不是仅根据URL判断页面状态
```

**改进后的流程**：
```
1. 登录操作后，立即获取页面快照
2. 从快照确认：是否登录成功、当前页面内容
3. 向用户汇报：登录成功，当前显示XX页面
4. 再询问下一步操作
```

### 经验四：数据提取深度

**本次做对** ✅：
- 提取TOP 20商品完整信息
- 分析类目分布
- 计算关键洞察
- 提供经营建议

**可以改进** ⚠️：
- 数据量大时考虑分页处理
- 询问用户是否需要导出原始数据
- 提供更多维度分析（时间趋势、同比环比）

## 🎯 最佳实践

### DO（应该做）

| 实践 | 说明 |
|------|------|
| ✅ 坦诚告知工具问题 | 明确说明原因和解决方案 |
| ✅ 验证码确认 | 告知识别结果，确认后再填写 |
| ✅ 登录后确认状态 | 获取快照确认页面内容 |
| ✅ 主动探索功能入口 | 发现菜单入口主动获取数据 |
| ✅ 数据结构化整理 | 整理成表格、图表、报告 |
| ✅ 提供洞察建议 | 不仅罗列数据，还分析建议 |
| ✅ 分步汇报进度 | 每个阶段完成后汇报 |

### DON'T（不应该做）

| 实践 | 说明 |
|------|------|
| ❌ 隐瞒工具问题 | 不告知用户反复尝试失败操作 |
| ❌ 假设验证码正确 | 不告知用户直接填写 |
| ❌ 仅依赖URL判断 | 不获取快照确认页面内容 |
| ❌ 只罗列原始数据 | 不进行整理和分析 |
| ❌ 忽略分页 | 数据量大时不处理分页 |
| ❌ 不问用户需求 | 不询问是否需要导出 |

## 🔧 故障排查

### 验证码识别失败

**现象**：验证码图片模糊无法识别

**解决方案**：
1. 截图告知用户无法识别
2. 请用户手动输入
3. 或使用其他识别服务

### 登录后跳转异常

**现象**：未跳转到预期页面

**解决方案**：
1. 获取快照确认实际页面
2. 检查登录失败提示
3. 尝试直接导航目标URL

### 数据表格过长

**现象**：数据表有几百行

**解决方案**：
1. 询问需要前几名
2. 使用筛选功能
3. 分页获取数据
