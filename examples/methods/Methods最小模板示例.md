# Methods 最小模板示例

这个文件提供一份可直接复制的 Methods 模板，适合在使用 `method_create` 新建方法文档时作为参考。

## 命名规则

- **推荐**：优先使用英文 slug，格式为 `[target]-[action]-[suffix].md`
- **兼容**：也支持安全的任意中文 `.md` 文件名
- **推荐示例**：`web-browser-automation.md`、`file-read-basic.md`、`memory-query-recall.md`
- **中文示例**：`网页自动化基础.md`、`飞书机器人接入排查.md`
- **避免**：路径分隔符、Windows 非法字符、前后空白、隐藏文件名

## 推荐字段说明

以下 frontmatter 字段可被当前方法论工具识别并用于列表展示、搜索排序与阅读提示：

| 字段 | 是否推荐 | 说明 |
|------|----------|------|
| `summary` | 推荐 | 方法摘要，建议 1 句话 |
| `status` | 推荐 | 例如 `draft` / `verified` / `deprecated` |
| `version` | 可选 | 方法版本 |
| `created_at` | 可选 | 创建时间 |
| `updated_at` | 推荐 | 最近更新时间 |
| `last_verified` | 可选 | 最近验证时间 |
| `read_when` | 推荐 | 适合在什么任务前优先阅读 |
| `tags` | 推荐 | 检索标签 |

## 可直接复制的模板

```md
---
summary: "一句话说明这条方法解决什么问题"
status: "verified"
version: "1.0"
created_at: "2026-03-07"
updated_at: "2026-03-07"
last_verified: "2026-03-07"
read_when:
  - "开始相关任务前"
  - "遇到同类报错时"
tags:
  - "tag1"
  - "tag2"
---

# 方法标题（可用中文）

## 适用场景
- 这条方法适用于什么任务
- 哪些前置条件满足时优先使用

## 前置检查
- 环境 / 依赖是否已就绪
- 需要哪些权限、配置、文件路径或外部服务

## 输入与变量
- 关键输入参数是什么
- 哪些路径、端口、环境变量、账号信息会影响执行

## 执行步骤
1. 第一步做什么
2. 第二步做什么
3. 如需命令、工具调用或关键参数，在这里写清楚

## 验证方式
- 如何判断执行成功
- 成功时应看到什么结果
- 失败时通常会出现什么信号

## 工具选择
- 主工具：优先使用什么工具，为什么
- 备选工具：主工具不可用时如何兜底

## 失败经验
- 常见坑点
- 常见报错及原因
- 如何规避或快速恢复

## 成功案例
- 记录至少一个真实使用场景
- 简述背景、结果和关键决策

## 快速参考
- 只保留最常用的命令、参数、判断条件

## 更新记录
- 2026-03-07：初版
```

## 最小可用版本

如果当下时间紧，只写最小必需信息，至少保留以下部分：

1. 标题
2. `summary` / `status` / `read_when` / `tags`
3. `适用场景`
4. `执行步骤`
5. `验证方式`
6. `失败经验`

## 编写建议

- 标题面向人类阅读，文件名面向稳定检索；两者不必完全一致
- 优先写“怎么做”和“怎么判断对了”，少写空泛原则
- 如果是修 bug 得出的经验，务必写清楚报错现象、根因线索和验证办法
- 如果方法会随环境变化，建议维护 `updated_at` 和 `last_verified`
- 如果正文很长，`summary` 要尽量短，方便 `method_list` / `method_search` 快速判断

## 示例文件名组合

- 英文推荐：`web-browser-automation.md`
- 英文推荐：`config-env-setup.md`
- 英文推荐：`memory-query-recall.md`
- 中文兼容：`网页自动化基础.md`
- 中文兼容：`数据库迁移排查.md`
- 中文兼容：`飞书消息转发故障处理.md`

## 说明

- 新方法默认按这个模板或其精简版编写
- 旧方法文件暂不强制迁移，可在后续维护时逐步补齐 frontmatter 和章节
- 当前方法论工具会检索文件名、标题、摘要、标签、章节名和正文，因此建议把关键信息写在前面
