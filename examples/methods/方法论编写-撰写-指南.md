## 1. 触发条件（什么时候写方法论）

满足**任一**条件即应创建/更新方法论：

| 条件 | 说明 | 示例 |
|------|------|------|
| **新工具首次使用成功** | 掌握了一个新工具的使用方法 | 第一次成功使用 `token_counter` |
| **多次尝试后成功** | ≥2 次错误后才找到正确路径 | 尝试了 3 种方式才实现 WSL 代理 |
| **复杂任务** | 涉及 ≥3 个步骤或工具 | 需要协调 timer + log_search + token_counter |
| **发现更优方案** | 现有方法有改进空间 | 发现比现有方法快 50% 的新路径 |
| **同一任务第二次出现** | 临时方案应固化为方法 | 第二次执行相同类型的文件备份 |

## 2. 命名规范

### 2.1 中文命名（推荐）
```
[目标对象]-[动作类型]-[细分].md
```

| 部分 | 长度限制 | 示例 |
|------|----------|------|
| 目标对象 | ≤7 字 | 效率追踪、文件备份、任务关闭 |
| 动作类型 | ≤4 字 | 创建、编辑、删除、查询、重启 |
| 细分（可选） | ≤4 字 | 安全、批量、定时、分段 |

**命名示例**：
- ✅ `效率追踪-v2.md`（简洁+版本号）
- ✅ `文件备份-工作区.md`（目标+细分）
- ✅ `任务关闭-检查清单.md`（目标+细分）
- ❌ `Log-methodology.md`（英文，不符合中文规范）

### 2.2 版本管理
- **首次创建**：无需版本号
- **重大升级**：创建 `xxx-v2.md`，保留原文件作为历史
- **小修正**：直接编辑原文件，在"更新记录"中注明

## 4. 编写流程

### 阶段一：需求确认（Prevent Misunderstanding）

**必须明确的问题**：
1. [ ] 这个方法论解决什么问题？
2. [ ] 目标用户是谁？（自己 / 其他 Agent / 拍档）
3. [ ] 命名是否符合规范？（中文？长度？）
4. [ ] 是创建新文件还是升级现有文件？

**失败教训**：
> 拍档说"改为中文"，我创建了 `Efficiency-Tracking-v2.md`，但实际上这算"英文+版本号"，不是纯中文命名。
> 
> **教训**：需求确认时要更精确，"中文"≠"包含中文"，可能是"纯中文"。

---

### 阶段二：内容编写（Content Creation）

**核心原则**：
- **可执行性**：每个步骤都能被直接执行，不模糊
- **可验证性**：有明确的完成标准和检查清单
- **可演进性**：预留"失败经验"和"成功案例"章节

**必含元素**：
| 元素 | 说明 | 示例 |
|------|------|------|
| 工具调用示例 | 具体的命令/代码 | `timer start <task_name>` |
| 检查清单 | 可勾选的步骤 | `- [ ] 工具使用次数更新？` |
| 失败经验表格 | 时间/问题/原因/解决方案 | 见下方"失败经验"章节 |
| 版本历史 | 记录演进过程 | v1.0 → v2.0 的变更 |

---

### 阶段三：版本决策（Version Management）

**决策树**：
```
是否已存在同名方法论？
    ├── 否 → 创建新文件（无需版本号）
    └── 是 → 变更是否重大？
            ├── 是（结构/核心逻辑改变）→ 创建 v2，保留 v1
            └── 否（补充/修正）→ 编辑原文件，更新记录
```

**失败教训**：
> 创建了 v2 后，没有立即决定如何处理 v1，实际上应该一开始就明确：是覆盖？还是保留历史版本？
>
> **教训**：版本升级时，文件生命周期管理要在一开始就确定。

---

### 阶段四：自举验证（Self-Bootstrap）

**关键步骤**：用新方法论追踪"创建它"的这次任务本身！

**操作**：
1. 启动 `timer` 和 `token_counter`
2. 执行方法论编写
3. 结束后填写"效能对账单"
4. 如果发现问题，立即修正方法论

**失败教训**：
> 方法论设计完，没有立即用一个真实任务来验证。
>
> **教训**：方法论写完要立即"自举验证"，用新方法来追踪创建它的过程。

## 5. 失败经验

| 时间 | 问题 | 原因 | 解决方案 |
|------|------|------|----------|
| 2026-03-07 | 文件命名不符合"纯中文"要求 | 对"中文命名"理解不精确，以为是"包含中文"即可 | 明确规范：优先纯中文，英文仅用于版本号（v2） |
| 2026-03-07 | 旧文件处理犹豫 | 创建 v2 后未立即决定 v1 的命运 | 在编写前就确定版本策略：小改编辑，大改新建 |
| 2026-03-07 | 缺少自举验证 | 方法论写完后没有立即用其追踪自身 | 强制要求：用新方法论追踪"创建它"的这次任务 |
| 2026-03-07 | 遗漏"方法论编写方法"本身 | 有多种方法论，但没有"如何编写方法论"的方法 | 创建本文件，填补元方法空白 |

## 6. 成功案例

### 案例 1：效率追踪方法论 v2.0（2026-03-07）
- **任务**: 将 Log-methodology 升级为三维效能模型
- **步骤**:
  1. 确认需求：纳入错误量、计时、Token 消耗
  2. 确认命名：中文优先，版本号后缀
  3. 设计三维模型：时间 + Token + 错误
  4. 编写完整文档，包含元信息、执行流程、效能对账单
  5. 创建 v2，保留 v1 作为历史
  6. 更新 TOOLS.md 和记忆日记
- **结果**: 成功创建 `Efficiency-Tracking-v2.md`，三维模型清晰可用
- **结论**: 需求确认充分，版本管理清晰，但缺少自举验证环节
