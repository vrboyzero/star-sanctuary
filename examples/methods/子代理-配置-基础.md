# 子代理配置方法

## 概述

本文档说明如何配置和管理子代理（Sub-Agent）。子代理配置存储在 `~/.star_sanctuary/agents.json` 文件中。

## 配置文件位置

> **⚠️ 重要**：配置文件位于 Star Sanctuary 状态目录根部，**不是** `agents/` 目录！

```
~/.star_sanctuary/
├── agents.json          # ← 主配置文件（JSON 格式）
├── agents/              # ← Agent 专属工作区目录
│   ├── coder/           # coder 的自定义人格
│   │   ├── SOUL.md     # 覆盖 default 的 SOUL
│   │   ├── IDENTITY.md # 覆盖 default 的 IDENTITY
│   │   └── facets/     # coder 专属的 FACET 模组
│   └── researcher/
│       └── SOUL.md     # researcher 的自定义人格
├── SOUL.md              # default Agent 的人格
└── ...
```

## 配置结构（agents.json）

### 完整字段说明

```json
{
  "agents": [
    {
      "id": "coder",
      "displayName": "代码专家",
      "model": "primary",
      "systemPromptOverride": "你是一个严谨的代码专家，擅长 TypeScript 和系统设计。",
      "workspaceDir": "coder",
      "toolsEnabled": true,
      "toolWhitelist": ["file_read", "file_write", "run_command", "web_fetch"],
      "maxInputTokens": 100000
    }
  ]
}
```

### 字段说明

| 字段 | 必填 | 说明 | 示例 |
|------|------|------|------|
| `id` | ✅ | 唯一标识 | `"coder"`, `"researcher"` |
| `displayName` | ❌ | 显示名称（默认等于 id） | `"代码专家"` |
| `model` | ✅ | 模型引用 | `"primary"` 或 `"deepseek-chat"` |
| `systemPromptOverride` | ❌ | 追加到系统提示词的额外内容 | `"你擅长..."` |
| `workspaceDir` | ❌ | Agent 专属工作区目录名（默认等于 id） | `"coder"` |
| `toolsEnabled` | ❌ | 是否启用工具（默认 true） | `true` |
| `toolWhitelist` | ❌ | 可用工具白名单 | `["file_read", ...]` |
| `maxInputTokens` | ❌ | 最大输入 token 数覆盖 | `100000` |

### model 字段说明

| 值 | 含义 |
|-----|------|
| `"primary"` | 使用环境变量配置（BELLDANDY_OPENAI_*） |
| 其他值 | 引用 `models.json` 中对应 id 的条目 |

## 创建新子代理

### 步骤 1: 编辑 agents.json

> **路径**：`~/.star_sanctuary/agents.json`

在 `agents` 数组中添加新代理配置：

```json
{
  "agents": [
    {
      "id": "writer",
      "displayName": "文案专家",
      "model": "primary",
      "systemPromptOverride": "你是一位专业的文案专家，擅长各种类型的文案创作，包括广告文案、产品描述、社交媒体内容等。",
      "toolsEnabled": true,
      "toolWhitelist": ["file_read", "file_write"]
    }
  ]
}
```

### 步骤 2:（可选）创建专属工作区

如需为 Agent 创建自定义人格，在 `agents/` 目录下创建：

```
~/.star_sanctuary/agents/writer/
├── SOUL.md      # 自定义人格（可选，会覆盖 default 的 SOUL）
└── facets/      # 专属 FACET 模组（可选）
```

**继承规则**：
- 对每个可继承文件（SOUL.md、IDENTITY.md、USER.md、AGENTS.md、TOOLS.md、MEMORY.md）
- 优先从 `agents/{id}/` 读取
- 不存在则自动 fallback 到根目录的同名文件

### 步骤 3: 重启服务

修改配置后必须重启服务才能生效：

```bash
service_restart reason="加载新的子代理配置"
```

### 步骤 4: 测试验证

```bash
# 测试新代理
delegate_task agent_id="writer" instruction="写一段介绍文案"
```

## 失败经验

| 错误 | 原因 | 解决方案 |
|------|------|----------|
| 代理不响应 | 修改配置后未重启 | 执行 `service_restart` |
| Agent not found | agents.json 格式错误或 agent_id 不存在 | 检查 JSON 语法和 agent_id |
| 工具权限不足 | toolWhitelist 未包含所需工具 | 在配置中添加工具到 toolWhitelist |
| 上下文丢失 | context 参数未传递 | 明确在 delegate_task 中传递 |
| 配置文件找不到 | 在 agents/ 目录下找 | 正确路径是 ~/.star_sanctuary/agents.json |
| model 无效 | model 值不在 primary 或 models.json 中 | 确认模型配置正确 |

## 最佳实践

1. **职责分离**：每个子代理应有明确的专业领域
2. **工具最小化**：通过 toolWhitelist 只授予必要的工具权限
3. **提示词优化**：使用 systemPromptOverride 根据任务类型定制
4. **测试验证**：每次配置后立即测试
5. **监控追踪**：通过 `sessions_history` 跟踪执行
6. **工作区隔离**：需要人格隔离时，创建 `agents/{id}/` 目录
