# 模型配置调用改造实施方案

## 1. 背景与目标

### 1.1 背景
- 当前 Belldandy 已支持：
  - 全局 `BELLDANDY_OPENAI_WIRE_API`
  - 按模型 `wireApi`（`models.json`）
  - fallback 默认 `chat_completions`，primary 按全局或模型覆盖
- 现象：`gpt-5.3-codex` 在部分网络环境出现 `fetch failed` / `503`，fallback 正常。
- 对比参考：cc-switch 在 Codex 路径下具备更完整的代理/转发层、网络策略与健康治理。

### 1.2 目标
- 提升模型调用稳定性，尤其是第三方代理场景。
- 建立“可分阶段交付”的改造路径，避免一次性重构风险。
- 保证中断后可续做：每阶段都有明确产物、验收与回滚方案。

---

## 2. 分级改造总览

| 级别 | 定位 | 可行性 | 工作量 | 风险 |
|---|---|---|---|---|
| 一级（小改） | 现有架构增强（网络策略+健康探测） | 高 | 1-2 天 | 低 |
| 二级（中改） | 引入统一 Provider Router/Adapter 层 | 中高 | 4-7 天 | 中 |
| 三级（大改） | 完整本地代理化（类 cc-switch） | 中低 | 2-4 周 | 高 |

---

## 3. 一级方案（小改，优先落地）

## 3.1 改造范围
- 在 `models.json` 增加网络策略字段（按模型）：
  - `requestTimeoutMs`
  - `connectTimeoutMs`（可选，若实现拆分）
  - `maxRetries`
  - `retryBackoffMs`
  - `proxy`（可选：每模型代理）
- 在 `FailoverClient` 生效上述策略。
- 增加“primary 预热健康检查”与快速降级机制。

## 3.2 关键实现点
1. 扩展模型配置结构
- 文件：`packages/belldandy-agent/src/failover-client.ts`
- `ModelProfile` 与 `ModelConfigFile` 增加可选网络字段。
- `loadModelFallbacks()` 解析并校验范围（防止异常值）。

2. 请求执行策略
- 文件：`packages/belldandy-agent/src/failover-client.ts`
- `fetchWithFailover()` 使用 profile 级 timeout/retry。
- 对网络错误（TLS 握手失败、连接重置）归类为可重试并触发 failover。

3. 启动预热检查
- 文件：`packages/belldandy-core/src/bin/gateway.ts`
- 启动后对 primary 执行轻量探测（按 wire api 走 `/responses` 或 `/chat/completions`）。
- 失败仅告警，不阻塞启动；请求时优先快速切 fallback。

4. 可观测性
- 日志中增加字段：
  - `profile_id`
  - `wire_api`
  - `timeout_ms`
  - `retry_index`
  - `error_category`

## 3.3 验收标准
- primary 不可用时，首轮请求 failover 总耗时显著下降（目标 < 3s，视网络环境可调整）。
- 日志能清晰区分网络错误/HTTP错误/格式错误。
- 不影响现有 `wireApi` 与 `protocol` 行为。

## 3.4 回滚方案
- 保留旧逻辑分支开关（如 `BELLDANDY_FAILOVER_PROFILE_NETOPTS=false`）。
- 若出现异常，关闭新开关后回退到现有稳定逻辑。

## 3.5 风险与缓解
- 风险：重试过多导致延迟拉长。
- 缓解：限制 `maxRetries` 上限，且只对可重试错误生效。

---

## 4. 二级方案（中改，统一调用层）

## 4.1 改造范围
- 在 Belldandy 内新增统一的 `ProviderRouter + Adapter` 层：
  - 负责 endpoint/wire_api 选择、URL 构建、认证注入、可选请求体适配。
- `OpenAIChatAgent` 与 `ToolEnabledAgent` 调用路径收敛到统一层，减少重复逻辑。

## 4.2 关键实现点
1. 新增抽象
- 建议新目录：`packages/belldandy-agent/src/provider-router/`
- 组件：
  - `adapter.ts`：接口定义（extract base url、build endpoint、auth headers）
  - `openai-adapter.ts`：openai 协议适配（含 responses/chat_completions）
  - `anthropic-adapter.ts`：anthropic 协议适配
  - `router.ts`：依据 profile 选择 adapter 与 endpoint

2. 收敛调用
- 修改：
  - `packages/belldandy-agent/src/openai.ts`
  - `packages/belldandy-agent/src/tool-agent.ts`
- 目标：请求构造逻辑只保留“业务字段组织”，网络路由走统一层。

3. 统一错误语义
- 网络层错误标准化：
  - `network_connect`
  - `network_tls`
  - `timeout`
  - `http_4xx`
  - `http_5xx`
  - `payload_format`

## 4.3 验收标准
- 两个 Agent 共享同一套路由策略，行为一致。
- 新增/切换模型协议时，只改 Adapter 层即可。
- 原有用例与构建通过。

## 4.4 回滚方案
- 保留旧调用构造代码路径（feature flag 切换）。
- 如线上异常，回切旧路径并保留新代码待修。

## 4.5 风险与缓解
- 风险：SSE 与工具调用解析回归。
- 缓解：补充回归测试（non-stream/stream/tool-calls/failover 四类）。

---

## 5. 三级方案（大改，完整代理化）

## 5.1 改造范围
- 引入本地代理服务（类似 cc-switch）：
  - 统一入口路由（如 `/responses`、`/chat/completions`）
  - 转发、重试、熔断、健康状态、可选 takeover
  - 可选请求日志与流量审计

## 5.2 关键实现点
1. 代理服务进程/模块
- 可选形态：
  - 内置同进程模块（Node）
  - 独立服务进程（Node/Rust）
- 明确与 Gateway 的边界与通信方式。

2. 路由与治理
- 路由层统一识别 app/protocol/wire_api。
- 内置熔断器与健康检查缓存，支持动态 failover 队列。

3. 管理面
- 配置管理（模型、优先级、网络策略、代理策略）。
- 观测面（成功率、延迟、错误分类、切换次数）。

## 5.3 验收标准
- 关键场景可用性显著提升（上游抖动场景下稳定可切换）。
- 调用链与配置解耦，扩展新 provider 成本降低。

## 5.4 回滚方案
- 保留“直连模式”，代理层可开关。
- 灰度放量：先开发环境，再小流量，再全量。

## 5.5 风险与缓解
- 风险：系统复杂度和维护成本大幅上升。
- 缓解：仅在一级/二级效果不足时再启动三级立项。

---

## 6. 推荐执行顺序

1. 先落地一级方案（低风险高收益）。
2. 观察 3-7 天稳定性指标（primary 成功率、failover 次数、首包延迟）。
3. 若问题仍明显，再立项二级。
4. 三级仅在中长期有统一代理平台需求时执行。

## 8. 已落地实现明细（2026-03-06）

### 8.1 代码改造清单
- `packages/belldandy-agent/src/failover-client.ts`
  - `ModelProfile` 增加：`wireApi`、`requestTimeoutMs`、`maxRetries`、`retryBackoffMs`、`proxyUrl`
  - `loadModelFallbacks()` 解析并透传上述字段
  - `fetchWithFailover()` 增加按模型重试/退避与可观测日志
- `packages/belldandy-agent/src/agent-profile.ts`
  - 配置类型新增 `wireApi`，支持从配置文件按模型覆盖
- `packages/belldandy-agent/src/openai.ts`
  - OpenAI 调用支持按 `wireApi` 路由到 `responses` 或 `chat_completions`
  - 引入“多模态/大文本附件”自适应超时下限（120s/300s）
- `packages/belldandy-agent/src/tool-agent.ts`
  - 工具调用链同步支持 `responses` 与 `chat_completions`
  - 与聊天链路对齐：支持文本附件统计驱动的自适应超时下限
- `packages/belldandy-core/src/server.ts`
  - 文本附件统一按单文件 200000 字符截断并注入 `meta.attachmentStats`
  - 上报 `textAttachmentCount/textAttachmentChars/textAttachmentTruncatedCharLimit`
- `packages/belldandy-core/src/bin/gateway.ts`
  - 增加 primary 启动预热检查与失败冷却注入
- `packages/belldandy-core/src/cli/commands/doctor.ts`
  - 诊断输出补充 `wire_api` 相关信息
- `E:\\project\\belldandy\\.env.example`
  - 新增默认重试、退避、primary 代理、预热参数说明

### 8.2 实现效果
- 支持“按模型单独线路协议”：
  - `primary` 可走全局 `BELLDANDY_OPENAI_WIRE_API`
  - fallback 默认为 `chat_completions`
  - 可在 `models.json` 单独设置 fallback 的 `wireApi`
- 支持“按模型单独网络策略”：
  - 每个 fallback 可覆盖超时、重试、退避、代理
- 支持“primary 启动预热 + 失败冷却”：
  - 避免启动后首轮请求长时间卡在不稳定上游
- 支持“文本附件稳态策略”：
  - 文本附件大输入自动提高请求超时下限，避免 15s/30s 误超时
  - 文本附件字符统计进入 Agent `meta`，策略与模型调用链解耦

### 8.3 推荐配置（主稳 + gpt fallback）

`.env.local`（主线路建议稳定通道）：

```env
BELLDANDY_AGENT_PROVIDER=openai
BELLDANDY_OPENAI_BASE_URL=https://api.minimax.chat/v1
BELLDANDY_OPENAI_API_KEY=sk-xxx
BELLDANDY_OPENAI_MODEL=MiniMax-M2.5
BELLDANDY_OPENAI_WIRE_API=chat_completions

BELLDANDY_OPENAI_MAX_RETRIES=1
BELLDANDY_OPENAI_RETRY_BACKOFF_MS=300
BELLDANDY_PRIMARY_WARMUP_ENABLED=true
BELLDANDY_PRIMARY_WARMUP_TIMEOUT_MS=8000
BELLDANDY_PRIMARY_WARMUP_COOLDOWN_MS=60000

BELLDANDY_MODEL_CONFIG_FILE=~/.belldandy/models.json
```

`~/.belldandy/models.json`（将 `gpt-5.3-codex` 放 fallback）：

```json
{
  "fallbacks": [
```

### 8.4 验收与观察点
- 验收结论：fallback 线路已可用，`wireApi` 可按模型生效。
- 文本附件策略结论：
  - 单附件 200000 字符截断已生效
  - 大文本附件自动延时已生效（>=12000 字符: 120s；>=30000 字符: 300s）
  - 普通文本请求时延策略保持不变
- 观察重点：
  - `primary` 失败后切换耗时是否稳定在可接受范围（建议目标 < 3s）
  - `gpt-5.3-codex` 在代理网络抖动时是否以 fallback 身份稳定接管
  - 日志是否包含 `wire_api/timeout/retry` 关键字段，便于快速定位
