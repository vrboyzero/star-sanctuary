# 改名方案

## 重要执行原则（当前确认）

在本轮改名实施中，先明确以下执行原则，并作为后续所有操作的前提：

- **采用方案 A**：先在当前 `E:\project\belldandy` 仓库内，原地实施第一、二阶段的品牌展示层改名。
- **不动 WSL 备份副本**：暂时不要把改名中的中间状态同步到 `\\wsl.localhost\Ubuntu\home\vrboyzero\workspace\Belldandy`。
- **保留 WSL 备份作为原版本兜底**：该目录继续作为备份与回滚副本保留，确保一旦当前仓库改名过程中出现问题，仍然有一份可参考、可恢复的原始版本。
- **先本地完成、再决定是否同步备份**：只有当当前仓库的第一、二阶段改名完成，并经过人工检查确认无明显问题后，再单独决定是否将最新版本同步到 WSL 备份目录。
- **当前不新建 `Star_Sanctuary` 项目目录**：避免在“只改展示层、不改内部技术标识”的阶段，同时维护两套高度相似但命名不一致的项目副本，造成路径、脚本、文档和后续维护混乱。

一句话概括：

**当前就在主仓原地改；WSL 副本只保留为原版本备份，不参与本轮改名实施。**

## 目标

将项目对外展示名称从 **Belldandy（贝露丹蒂）** 调整为 **Star Sanctuary（星辰圣所）**，优先解决研发期品牌暴露与潜在版权联想问题。

本次评估的重点不是“全仓技术标识重构”，而是判断：

- 哪些内容适合优先修改
- 哪些内容暂时不应修改
- 改动的可行性、工作量、风险如何
- 可以采用怎样的分阶段实施方案

## 总体结论

结论很明确：

**现在做“表层品牌改名”是可行且推荐的；现在做“深层技术标识全改”不推荐。**

原因如下：

1. 项目中大量 `Belldandy / belldandy / 贝露丹蒂` 命中，既包括展示文案，也包括内部技术契约。
2. 展示层改名可以显著降低对外品牌暴露，且改动集中、风险较低。
3. 如果连环境变量、包名、状态目录、CLI、工作区路径一起改，就会从“品牌替换”升级为“兼容性重构”，风险和工作量都会明显上升。

因此，最合理的做法是：

- **第一阶段**：只改用户看得到、容易暴露品牌的表层内容
- **第二阶段**：根据新仓库名 / 域名 / 镜像名是否确定，再扩展到发布展示层
- **第三阶段**：如果未来确实有必要，再考虑深层技术标识迁移

## 命中内容分类

为了便于决策，可以把本次改名相关内容分成三层：

### A. 展示层（建议第一阶段优先修改）

这类内容通常是用户、合作方、测试人员、截图、演示视频里第一眼能看到的地方。

典型位置包括：

- 根文档与说明文档
- Web 页面标题、欢迎语、按钮文案、Logo alt 文案
- 浏览器扩展名称与描述
- 启动脚本输出的 Banner
- 产品介绍、使用手册、渠道接入说明

这部分改动的特点：

- 改后收益高
- 不影响核心逻辑
- 风险较低
- 回滚容易

### B. 发布展示层（可作为第二阶段）

这类内容主要出现在打包、部署、发布说明、外部链接、默认镜像名等位置。

典型位置包括：

- Docker / Compose 默认镜像名
- Nix 描述信息
- README 中的 GitHub / Release 链接
- 官方主页链接、工坊入口链接

这部分改动的特点：

- 对外观感影响较大
- 但常常依赖“新域名 / 新仓库 / 新组织名 / 新镜像名”是否已经确定
- 如果这些外部资源尚未确定，改一半容易造成信息不一致

### C. 内部技术标识层（当前不建议改）

这类内容属于程序内部契约，往往和兼容性、构建链路、运行数据直接相关。

典型位置包括：

- 环境变量前缀：`BELLDANDY_*`
- 用户状态目录：`~/.belldandy`
- 包作用域：`@belldandy/*`
- 工作区包目录：`packages/belldandy-*`
- 本地存储 key：`belldandy.webchat.*`
- CLI 入口：`bdd`
- 测试数据、快照、样例文本中的技术命名

这部分改动的特点：

- 表面看是“名字替换”
- 实际上是“内部契约重构”
- 很容易牵动兼容层、迁移脚本、文档、测试、部署流程

## 当前不建议修改的内容

以下内容**建议暂时不改**，至少不要放在第一阶段。

### 1. 环境变量前缀 `BELLDANDY_*`

例如：

- `.env`
- `.env.example`
- `docker-compose.yml`
- 大量 `packages/*` 代码中的 `process.env.BELLDANDY_*`

原因：

- 这是运行时配置契约
- 一旦修改，就要同时处理兼容读取、旧配置迁移、文档同步、Docker / Nix / 脚本联动
- 很容易引发“项目能启动但某些功能悄悄失效”的隐性问题

结论：

- **第一阶段不要改**
- 如果未来要改，建议做“双前缀兼容期”而不是一次性硬切

### 2. 状态目录 `~/.belldandy`

例如：

- 日志目录
- methods / skills / plugins / memory / agents / facets 等目录

原因：

- 这已经是当前项目的数据根目录约定
- 一旦修改，必须处理旧用户数据迁移
- 如果迁移不完整，容易出现“历史数据丢失”或“新旧目录混用”问题

结论：

- **第一阶段不要改**
- 如果未来要改，应设计自动迁移或兼容回退

### 3. 包名与目录名

例如：

- `@belldandy/core`
- `@belldandy/skills`
- `packages/belldandy-core`
- `packages/belldandy-agent`

原因：

- 这直接牵涉 workspace 依赖、import 路径、构建链、锁文件、测试、脚本、IDE 索引
- 这是典型的“深层重构”，不是简单的表层改名

结论：

- **当前不建议动**

### 4. CLI 名称与内部存储 key

例如：

- `bdd`
- `belldandy.webchat.auth`
- `belldandy.webchat.clientId`

原因：

- CLI 名会影响现有使用方式与文档命令
- 本地存储 key 会影响浏览器端状态与旧缓存兼容

结论：

- **第一阶段不改**

## 风险评估

### 方案一：只做表层品牌替换

风险等级：**低**

风险主要在于：

- 个别文案漏改，导致前后不一致
- 文档标题改了，但正文仍残留旧品牌
- 某些截图或说明文本仍写旧名字

优点：

- 不动运行时契约
- 不影响环境变量
- 不影响状态目录
- 不影响 workspace / import / build
- 很适合当前阶段先落地

### 方案二：表层 + 发布展示层

风险等级：**中低**

新增风险：

- 对外链接、镜像名、主页地址可能与实际资源不一致
- 如果新 GitHub 仓库或域名未定，容易出现“文档写新名，链接还是旧名”的过渡混乱

优点：

- 对外品牌一致性更强
- 更适合准备公开演示或对外发布前统一口径

### 方案三：深层技术标识全改

风险等级：**高**

新增风险：

- 环境变量兼容问题
- Docker / Nix / 脚本失效
- 本地数据目录兼容与迁移问题
- 包作用域与 workspace 依赖断裂
- 旧文档命令、测试、自动化脚本全面联动

结论：

- 当前不建议做

## 最终建议

**建议采用“第一阶段 + 第二阶段联合推进，但第二阶段按资源确认拆分执行”的方式。**

- 能较快把对外可见的 `Belldandy` 形象替换为 `Star Sanctuary`

- 能把高风险的“技术标识迁移”继续延后

- 更换仓库名
- 更换官网域名
- 更换 Docker 镜像名
- 更换技术命名体系

再决定是否进入第二阶段或第三阶段。
