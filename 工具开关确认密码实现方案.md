## 目标

为 `BELLDANDY_AGENT_TOOL_CONTROL_MODE=confirm` 增加一个可选的确认密码配置项：

- 新增配置项：`BELLDANDY_AGENT_TOOL_CONTROL_CONFIRM_PASSWORD`
- 未配置密码时，保持现有确认方式不变：
  `批准工具设置变更 <requestId>`
- 已配置密码时，改为要求用户输入自定义确认口令
- Agent、服务端日志、聊天记录、前端事件中都不得回显该密码

这个方案主要解决社区聊天室、群聊、多用户共享会话场景下，其他人伪造确认文本去控制 Agent 工具开关的问题。

---

## 现状

当前 `confirm` 模式的关键实现位于：

- `packages/belldandy-skills/src/builtin/tool-settings-control.ts`
- `packages/belldandy-core/src/bin/gateway.ts`
- `packages/belldandy-core/src/server.ts`
- `apps/web/public/index.html`
- `apps/web/public/app.js`

当前流程：

1. Agent 调用 `tool_settings_control`，`action="apply"`。
2. 若模式为 `confirm`，工具返回提示文案：
   `如确认，请回复：批准工具设置变更 <requestId>`
3. 用户下一条消息进入会话历史。
4. Agent 再次调用 `action="confirm"`。
5. 工具通过读取最近一条用户消息，校验是否精确匹配：
   `^批准工具设置变更\\s+([A-Z0-9-]+)$`

当前风险：

- 口令本身是公开文本，群聊里别人能直接照抄。
- 如果把“自定义密码”直接替换进去，但仍然从最近一条用户消息读取，就会把密码暴露给会话历史和 Agent 上下文。
- 用户消息目前会先写入 `conversationStore`，再传给 agent 的 `history`，所以“只改匹配文本，不改消息链路”不满足安全目标。

结论：

- 这个需求方向是对的。
- 但不能只改 `批准工具设置变更 <requestId>` 这句文案。
- 必须把“密码验证”从普通聊天内容中分离出来。

## 目标行为

### 1. 默认兼容行为

当未配置 `BELLDANDY_AGENT_TOOL_CONTROL_CONFIRM_PASSWORD` 时：

- 保持现有 `confirm` 模式不变
- 继续使用：
  `批准工具设置变更 <requestId>`

### 2. 密码模式行为

当已配置 `BELLDANDY_AGENT_TOOL_CONTROL_CONFIRM_PASSWORD` 时：

- `apply` 不再提示 `批准工具设置变更 <requestId>`
- 改为提示用户输入“已配置的确认口令”
- 服务端收到这条消息后，只做隐藏验证，不把密码写入会话历史
- Agent 看不到密码原文
- `confirm` 不再依赖“最近一条用户消息正文”，而是依赖服务端记录的“已确认状态”

### 3. 安全约束

必须满足以下约束：

- `config.read` 返回该配置时必须脱敏
- `tool_result` 不能输出密码
- `chat.final` 不能输出密码
- 服务器日志不能打印密码
- 会话历史不能保存密码原文
- Agent 的输入 `history` / `userInput` / `promptText` 中不能包含密码原文

涉及文件：

3. `config.read`

### 二、工具层

### 现有问题

当前 `tool-settings-control.ts` 的 `confirm` 分支依赖：

- `getLatestUserMessage(context)`
- `extractApprovalRequestId(message)`

这套逻辑只适合公开口令，不适合密码口令。

### 推荐改法

不要让工具直接读取密码文本。

改为把工具控制确认拆成两种确认源：

- `request_text`
  现有模式，使用 `批准工具设置变更 <requestId>`
- `password`
  新模式，使用服务端隐藏校验结果

### 建议的数据结构调整

扩展 `PendingToolControlRequest`，增加字段：

```ts
type PendingToolControlRequest = {
  requestId: string;
  conversationId: string;
  requestedByAgentId?: string;
  changes: ToolControlChanges;
  createdAt: number;
  expiresAt: number;
  passwordApprovedAt?: number;
};
```

或等价地新增独立 approval store。两种都可以，但优先推荐直接挂在 pending request 上，改动更小。

### `tool-settings-control.ts` 需要修改的点

1. `AgentToolControlDeps` 新增：

```ts
getHasConfirmPassword?: () => boolean;
```

这里只传布尔值，不传密码原文，避免工具层误输出。

2. `apply` 分支

当 `mode === "confirm"` 时：

- 未配置密码：保持原文案
- 已配置密码：改为输出类似文案

建议文案：

```text
本次请求尚未执行。
<变更摘要>
这是全局调用设置变更，会影响当前 Gateway 的其他会话。
如确认，请让用户回复已配置的工具开关确认口令。
收到确认后，再次调用 tool_settings_control，参数使用 action="confirm" 与 requestId="<requestId>"。
```

注意：

- 绝不能把密码本身拼到返回文案里

3. `confirm` 分支

改成双通道校验：

- 未配置密码：继续走 `extractApprovalRequestId(latestUserMessage)`
- 已配置密码：检查 pending request 是否已经被服务端标记为 `passwordApprovedAt`

建议错误文案：

```text
尚未检测到有效确认口令。请让用户回复已配置的工具开关确认口令。
```

注意：

- 不要返回“正确口令是什么”
- 不要提示密码长度或部分片段

### `ToolControlConfirmationStore` 需要修改的点

建议新增方法：

```ts
getLatestByConversation(conversationId: string): PendingToolControlRequest | undefined;
markPasswordApproved(requestId: string, approvedAt?: number): PendingToolControlRequest | undefined;
```

用途：

- 服务端收到密码消息时，可以找到当前会话最近的待确认请求
- 标记该请求已经通过密码确认

### 三、服务端消息链路

这是本方案最关键的部分。

### 核心原则

密码消息不能按普通用户聊天消息处理。

如果仍然执行当前逻辑：

1. `ctx.conversationStore.addMessage(conversationId, "user", parsed.value.text, ...)`
2. 再把这条消息带进 `history`

那么密码会直接暴露给 Agent。

所以必须在 `message.send` 入口做前置拦截。

### 推荐流程

在 `packages/belldandy-core/src/server.ts` 的 `message.send` 处理分支中增加一段前置判断：

前提同时满足：

- `BELLDANDY_AGENT_TOOL_CONTROL_MODE === "confirm"`
- `BELLDANDY_AGENT_TOOL_CONTROL_CONFIRM_PASSWORD` 已配置
- 当前会话存在未过期的待确认工具请求
- 当前用户消息精确等于配置密码

命中后执行：

1. 不把原始密码写入 `conversationStore`
2. 不把原始密码传给 Agent
3. 调用 `confirmationStore.markPasswordApproved(requestId)`
4. 将用户输入替换为一个安全占位文本，再继续本轮消息处理

建议占位文本：

- 用户侧占位：
  `【已提交工具开关确认口令】`

也可以不写入会话历史，但这样 Agent 可能不知道用户刚完成确认。为了兼顾安全和可用性，推荐写入占位文本，而不是原文密码。

### 为什么推荐“占位文本”而不是直接吞掉这条消息

如果完全吞掉密码消息：

- Agent 不会收到任何新的用户输入
- 本轮可能不会继续调用 `action="confirm"`

如果写入占位文本：

- Agent 能知道“用户已经做了确认动作”
- 但看不到密码本身

### 建议增加的服务端上下文

当前 `startGatewayServer` / `handleReq` 并不知道 `toolControlConfirmationStore`。

因此需要把 store 注入到服务端上下文，至少能在 `message.send` 阶段访问：

- `packages/belldandy-core/src/server.ts`
- `packages/belldandy-core/src/bin/gateway.ts`

建议扩展：

```ts
toolControlConfirmationStore?: ToolControlConfirmationStore;
getAgentToolControlMode?: () => "disabled" | "confirm" | "auto";
getAgentToolControlConfirmPassword?: () => string;
```

这样服务端才能在消息入口完成密码拦截与标记。

## 边界与风险

### 1. 不要把密码模式做成新的 mode

不建议新增 `BELLDANDY_AGENT_TOOL_CONTROL_MODE=password`。

原因：

- 语义上它仍然属于 `confirm`
- 改动 UI、分支逻辑、兼容性都更大
- 当前需求只是给 `confirm` 增加一种更安全的确认方式

### 2. 不要在工具层直接读取密码明文

工具层只需要知道：

- 是否启用了密码确认
- 当前 request 是否已经通过密码确认

工具层不应该持有密码值。

### 3. 不要在聊天里输出“请回复：<密码>”

这是最重要的泄露点之一。

### 4. 不要在没有 pending request 时拦截密码文本

否则会误吞正常对话，尤其是在社区聊天室里。

### 5. 不要只靠 requestId

密码模式下仍然建议保留 `requestId` 参数和 pending request 结构：

- 便于明确是哪一次变更请求被批准
- 便于沿用现有超时、会话隔离、去重逻辑

## 建议文案

### `apply` 成功但待确认

未配置密码：

```text
本次请求尚未执行。
<变更摘要>
这是全局调用设置变更，会影响当前 Gateway 的其他会话。
如确认，请回复：批准工具设置变更 <requestId>
收到确认后，再次调用 tool_settings_control，参数使用 action="confirm" 与 requestId="<requestId>"。
```

已配置密码：

```text
本次请求尚未执行。
<变更摘要>
这是全局调用设置变更，会影响当前 Gateway 的其他会话。
如确认，请让用户回复已配置的工具开关确认口令。
收到确认后，再次调用 tool_settings_control，参数使用 action="confirm" 与 requestId="<requestId>"。
```

### `confirm` 失败

未配置密码：

```text
尚未检测到精确确认口令。请让用户回复：批准工具设置变更 <requestId>
```

已配置密码：

```text
尚未检测到有效确认口令。请让用户回复已配置的工具开关确认口令。
```

### `confirm` 成功

两种模式都可以保留现有成功文案：

```text
已根据用户确认执行全局工具设置变更。
```

## 验收标准

满足以下条件即可认为方案完成：

1. 未设置密码时，旧流程完全可用
2. 设置密码后，用户只需输入密码，不需发送 requestId 口令
3. 密码不会出现在：
   - 会话历史
   - Agent 输入
   - tool_result
   - chat.final
   - 配置读取结果
4. 只有当前会话存在待确认请求时，密码才会生效
5. 密码确认成功后，工具开关变更仍正常广播到前端
