# 自动填充记忆优化实施方案

## 目标

本方案解决两个问题：

1. `BELLDANDY_CONTEXT_INJECTION` 开启后把最近原始会话记忆直接注入，导致 Agent 误把“历史已执行动作”当作“当前待执行指令”，出现重复重启服务、重复执行命令、重复执行已完成任务。
2. 记忆调用缺少重要性分层，当前只有“最近”概念，没有“长期有效 / 高价值 / 低价值”的区分。

本版 patch 的原则：

- **默认不再自动注入 raw session memory**
- **自动注入改为：高价值记忆 + 最近任务摘要**
- **增加重复执行守卫（按工具风险做分级去重）**
- **重要性只做 3 档：high / medium / low**
- **尽量基于现有 `task memory / task summary / category / memoryType` 能力扩展，不做大重构**

## 改动摘要

### 核心行为变化

1. `BELLDANDY_CONTEXT_INJECTION` 的默认注入来源，从“最近 chunk”改为“筛选后的 context injection memories”。
2. 默认排除 `memoryType=session`，除非显式开启 `BELLDANDY_CONTEXT_INJECTION_INCLUDE_SESSION=true`。
3. 自动追加 `recent-task-summary` 块，注入最近成功/部分成功的任务摘要，而不是回放完整对话原文。
4. 在 `before_tool_call` 增加重复执行守卫：
   - 若最近 N 分钟内已经成功执行过相同 `toolName + actionKey`
   - 当前已改为分级处理：
     - 高风险工具直接阻止
     - 中风险工具跳过本次执行，并把重复动作提示以 synthetic tool result 回注给 Agent
     - 具体分级可通过 `BELLDANDY_TASK_DEDUP_MODE` 与 `BELLDANDY_TASK_DEDUP_POLICY` 覆盖
5. 给记忆注入增加简单重要性评分：
   - `core` / `shared` / `decision|fact|preference|entity` 得分更高
   - `session` 原始会话得分更低
   - 越近的记忆会得到适度加分

这部分最终落地实现，已经比最初 patch 草案更进一步，核心点如下：

1. `gateway` 自动注入时不再直接拉最近 chunk，而是调用 `mm.getContextInjectionMemories()` 与 `mm.getRecentTaskSummaries()`。
2. 工具动作指纹 `buildToolActionKey()` 已下沉到 `task-dedup.ts`，并扩展支持：
   - `service_restart`
   - `switch_facet`
   - `method_create`
   - `run_command`
   - `file_write`
   - `file_delete`
   - 以及一组通用 fallback 参数指纹
3. `before_tool_call` 不再是“命中即一律 block”，而是先根据
   - `BELLDANDY_TASK_DEDUP_MODE`
   - `BELLDANDY_TASK_DEDUP_POLICY`
   解析当前工具的实际去重模式。
4. 当模式为 `warn-only` 时，当前实现会：
   - `skipExecution: true`
   - 返回 `syntheticResult`
   - 让 Agent 在当前轮次看到“这个动作最近已执行过”的提示，而不是静默吞掉或只写日志。
5. 当模式为 `hard-block` 时，当前实现会：
   - `block: true`
   - 返回明确 `blockReason`
   - 要求显式使用 `retry=true`、`force=true` 或 `allowDuplicate=true` 才允许重做

当前实际逻辑可概括为：

```ts
const mode = resolveToolDedupMode(event.toolName, {
  globalMode: taskDedupGlobalMode,
  policy: taskDedupPolicy,
});

if (mode === "off") return;

const duplicated = mm.findRecentDuplicateToolAction(...);
if (!duplicated) return;

if (mode === "warn-only") {
  return {
    skipExecution: true,
    syntheticResult: buildWarnOnlyDuplicateNotice(...),

  block: true,
  blockReason: "...若确需重试，请显式传入 retry=true / force=true / allowDuplicate=true",

```

默认内置分级策略为：

- `service_restart` => `hard-block`
- `switch_facet` => `hard-block`
- `method_create` => `hard-block`
- `run_command` => `warn-only`
- `file_write` => `warn-only`
- `file_delete` => `warn-only`
- 其他工具 => `off`

## 方案说明

### 1. 为什么默认不再注入 session

因为当前 `session` 是从 `jsonl` 会话原文抽出来的：

- `User: ...`
- `Assistant: ...`

这里面会天然包含：

- 已执行命令
- 已完成动作
- 失败重试过程
- “请重启服务”“刚刚已经写文件”等动作语句

如果按 `updated_at DESC` 直接注入，这些内容会非常像当前轮次的显式指令。

所以本 patch 让 `session` **默认只参与检索，不参与自动注入**。

### 2. 为什么引入 recent-task-summary

任务摘要比原始会话更适合自动注入，因为它的语义是：

- 发生过什么
- 状态是什么
- 已完成还是失败
- 产物是什么

而不是“把原来的命令原文再重复给模型看一遍”。

### 3. 为什么重复执行守卫放在 before_tool_call

因为这层是最后一道安全闸：

- 即使模型还是误判了
- 在真正执行副作用工具前仍可拦住

这比只靠 prompt 提示更稳。

### 4. 为什么 actionKey 只先支持少数工具

第一版建议只覆盖最容易产生事故的副作用工具：

- `service_restart`
- `run_command`
- `file_write`

后续如果效果稳定，再扩展到：

- `file_delete`
- `db_migrate`
- `deploy` 类工具
- 其他带明显副作用的自定义 MCP 工具

## 最小验证清单

### Case 1：重复重启服务回归

1. 打开：
   - `BELLDANDY_CONTEXT_INJECTION=true`
   - `BELLDANDY_CONTEXT_INJECTION_INCLUDE_SESSION=false`
   - `BELLDANDY_CONTEXT_INJECTION_TASK_LIMIT=3`
   - `BELLDANDY_TASK_DEDUP_GUARD_ENABLED=true`
2. 让 Agent 成功执行一次服务重启。
3. 在下一轮只问一个普通问题，不要求重启。
4. 预期：
   - 注入中只出现任务摘要，不出现原始“restart service ...”片段
   - Agent 不会再次触发 `service_restart`

### Case 2：普通知识/偏好记忆仍可自动注入

1. 在 `MEMORY.md` 或 `daily memory` 中写入一条：
   - 用户偏好 / 项目决策 / 重要事实
2. 发起普通对话。
3. 预期：
   - 该条记忆仍可进入 `<recent-memory>`
   - 因为 category 和 memoryType 高分而被保留

### Case 3：显式要求重跑时允许 bypass

1. 让 Agent 刚成功执行一次 `run_command`。
2. 再次明确要求“请再执行一次”。
3. 调用工具时传 `force=true`。
4. 预期：
   - dedup guard 不拦截
   - 工具允许执行

### Case 4：原始 session 仍能被搜索

1. 自动注入保持 `includeSession=false`。
2. 手动调用 `memory_search` 搜索前面对话里提到的内容。
3. 预期：
   - session 记忆仍然能检索出来
   - 只是默认不自动注入

## 风险点

### 风险 1：有些 session 内容其实有价值

确实如此，所以本 patch 不是删除 session，而是：

- **默认不自动注入**
- **保留显式检索能力**
- **保留开关可重新放开**

### 风险 2：actionKey 规则过粗可能误拦截

例如：

- 两次 `run_command` 文本相近，但上下文目的不同

所以第一版：

- 只在短时间窗口内生效
- 只拦截有副作用的高风险动作
- 保留 `force=true` 作为显式绕过

### 风险 3：category 白名单会漏掉一部分有用信息

这属于预期权衡。

自动注入应该偏保守；不在白名单内的内容仍可通过 `memory_search` 找回。

## 建议默认配置

```env
BELLDANDY_CONTEXT_INJECTION=true
BELLDANDY_CONTEXT_INJECTION_LIMIT=5
BELLDANDY_CONTEXT_INJECTION_INCLUDE_SESSION=false
BELLDANDY_CONTEXT_INJECTION_TASK_LIMIT=3
BELLDANDY_CONTEXT_INJECTION_ALLOWED_CATEGORIES=preference,fact,decision,entity
BELLDANDY_TASK_MEMORY_ENABLED=true
BELLDANDY_TASK_SUMMARY_ENABLED=true
BELLDANDY_TASK_DEDUP_GUARD_ENABLED=true
BELLDANDY_TASK_DEDUP_WINDOW_MINUTES=20
```

## 结论

这版 patch 的关键不是“让记忆更多”，而是：

- **让自动注入的记忆更像“状态”和“事实”**
- **而不是“历史动作原文回放”**

这样能明显降低 Agent 把历史任务重新执行一遍的概率，同时保留记忆系统的上下文价值。
