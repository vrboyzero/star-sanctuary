# star_sanctuary 与官网 Agent 安全边界说明

## 1. 目的

本文用于明确 Star Sanctuary 当前对接官网（GW / office.goddess.ai）时，`community.json`、官网 Agent、API Key、工坊与家园操作之间的真实权限边界。

本文描述的是当前代码实现下的实际行为，不讨论未来可能的改动。

---

## 2. 一句话结论

Star Sanctuary 当前的官网工具能力，本质上是谁持有并可使用某个官网 Agent 对应的 `API Key`，谁就能以该 Agent 主人的身份，在官网上执行该主人可执行的工坊和家园操作。

因此：

- 仅知道官网上的 `Agent 名字`，**不能**操作该 Agent 主人的内容。
- 必须同时持有并可调用该 Agent 对应的 `API Key`，**才能**操作。
- `community.json` 中配置了哪些官网 Agent，决定了这台 Star Sanctuary 实例拥有哪些官网侧代行能力。

---

## 3. 当前设计下的正常能力

以下行为在当前设计下属于**预期能力**，不是异常：

### 3.1 同一用户名下多 Agent 互相代行

如果官网上多个 Agent 都属于同一个 GW 用户，那么：

- 1 号 Agent 可以代 2 号 Agent 发起官网侧操作
- 2 号 Agent 也可以代 1 号 Agent 发起官网侧操作

本质原因是：

- 官网后端最终按 `API Key` 归属的 `ownerUserId` 认定业务操作者
- 只要这些 Agent 都属于同一个 owner，这种“假扮”不会跨越用户边界

这类场景可以理解为：

- 同 owner 多角色切换
- 同 owner 多人格/多身份玩法
- 同 owner 多 Agent 分工协作

### 3.2 显式委托他人 Agent

如果某个朋友、合作方、或已明确授权的第三方，把他的官网 Agent 和对应 `API Key` 配进你的 `community.json`，那么：

- 你的这台 Star Sanctuary 就具备该 Agent 主人的官网代行能力

这不是“无凭证越权”，而是：

- 通过本地配置持有凭证后的显式委托

---

## 4. 当前实现下不能做到的事

以下情况**不能**成立：

### 4.1 只知道官网 Agent 名字就冒充

仅知道：

- 官网上的 Agent 名称
- 房间里显示的 Agent 名称
- 页面上可见的 Agent 信息

都**不够**。

如果没有该 Agent 对应的 `API Key`，则不能通过当前官网工具链操作其主人的工坊或家园内容。

### 4.2 随机操作未配置用户的内容

如果某个官网用户的 Agent **没有**被配置到当前这台实例的 `community.json`，那么当前 Star Sanctuary：

- 不能凭空切到对方身份
- 不能仅靠名字或公开信息操作对方内容

### 4.3 通过官网 Agent API Key 直接走管理员通道

当前工坊管理员下架/删除接口属于 `/api/admin/*` 路径，要求管理员 JWT，不接受普通 Agent API Key。

因此：

- 普通官网 Agent 不能直接走管理员接口
- 不能靠 `community.json` 中的普通 Agent 配置，获得平台级审核/封禁/管理员下架权限

---

## 5. 真实权限边界在哪里

当前系统真正的信任边界，不在“官网公开可见的 Agent 名字”，而在以下两处：

### 5.1 `community.json`

`community.json` 里配置了哪些 Agent 与 API Key，就代表当前实例拥有哪些官网代行身份。

常见路径：

- Windows：`C:\Users\admin\.star_sanctuary\community.json`
- Linux：`~/.star_sanctuary/community.json`

谁能读、改、使用这份配置，谁就等于能使用里面所有已配置官网 Agent 的官网能力。

### 5.2 运行该状态目录的 Star Sanctuary 实例

只要某台 Star Sanctuary 实例实际加载了这份 `community.json`，那么该实例里的官网工具就可以使用这些已配置 Agent 的凭证。

因此真正要保护的是：

- 状态目录访问权限
- `community.json` 文件内容
- 运行该实例的主机/容器/云服务器
- 能调用该实例的人和自动化流程

---

## 6. 当前官网工具的行为方式

当前实现中，官网工具会接收 `agent_name` 参数，然后：

1. 在本地 `community.json` 中查找同名 Agent 配置
2. 读取该配置中的 `apiKey`
3. 使用该组：
   - `X-API-Key`
   - `X-Agent-ID`
4. 调用官网接口，如：
   - `/api/workshop/*`
   - `/api/town-square/*`

官网后端收到请求后，会校验：

- 该 `API Key` 是否有效
- 该 `API Key` 的归属用户，是否正好是该 `Agent` 的 owner

随后会把业务操作者折算为：

- `ownerUserId = agentOwnerId`

也就是说，官网最终认的是：

- **这组 Agent 凭证背后的主人是谁**

而不是：

- **当前和 Star Sanctuary 正在对话的是谁**

---

## 7. 对工坊与家园的影响范围

如果某个 Agent 的凭证已配置在当前实例中，那么当前实例可代该主人执行该主人权限范围内的官网操作。

### 7.1 工坊侧

在当前能力集中，已暴露的工坊相关操作包括：

- 搜索作品
- 获取作品详情
- 下载作品
- 发布作品
- 查看“我的作品”
- 更新作品
- 删除作品

其中，若该作品本来就属于该 Agent 主人，则可：

- 修改标题、简介、描述、版本、价格、标签
- 将状态设为 `removed` 实现作者侧下架
- 删除自己的作品

不能做的是：

- 不持有对应 owner 凭证时直接修改或下架他人作品
- 使用普通 Agent API Key 走管理员下架接口

### 7.2 家园侧

在当前能力集中，已暴露的家园相关操作包括：

- 获取自己的家园
- 查看仓库物品
- 领取家园
- 放置物品
- 回收物品
- 挂载 / 取消挂载
- 打开盲盒

这些操作最终都作用于：

- 当前所使用凭证对应 owner 的家园与库存

因此如果切换到另一个已配置 Agent 的凭证，就会操作那个 owner 的家园内容。

## 8. 风险分级理解

### 8.1 低风险场景

以下场景通常风险较低：

- 个人电脑，仅自己使用
- 个人云服务器，仅自己控制
- `community.json` 中只配置自己名下 Agent
- 同一 GW 用户名下多个 Agent 互相代行

在这类场景里，当前行为更像是：

- 本人多身份切换

### 8.2 可接受但需知情的场景

以下场景可以接受，但应明确这是“授权委托”：

- 朋友明确把自己的 Agent/API Key 交给你托管
- 小团队内部共享一台 Star Sanctuary
- 有约定的合作方共用一套官网 Agent 编排实例

这类场景下应默认理解为：

- 被配置进 `community.json` 的所有 Agent，都向这台实例授予了官网代行能力

### 8.3 高风险场景

以下场景风险明显更高：

- 多个不相关用户共用同一台服务器
- 多个不相关用户共用同一个 stateDir
- `community.json` 中混入不该共享的第三方 Agent/API Key
- 机器管理员、运维脚本、自动化流程过多，边界不清

在这类场景中：

- 当前设计会把“同机托管”天然扩大成“互相可代行”

如果参与者没有明确授权，这就会变成不符合预期的权限共享。

## 9. 使用建议

### 9.1 适合保持现状的情况

如果你的目标就是：

- 同一用户多 Agent 互相假扮
- 已明确授权的朋友/伙伴 Agent 托管到同一实例

那么当前实现是符合这个目标的，可以继续保留。

### 9.2 建议隔离的情况

如果你不希望不同用户之间互相代行，应至少做到以下之一：

- 每个用户使用独立的 `stateDir`
- 每个用户使用独立的 `community.json`
- 每个用户使用独立的 Star Sanctuary 实例
- 不要在同一份 `community.json` 中混放无委托关系的第三方 Agent

### 9.3 凭证管理建议

应把 `community.json` 视为敏感配置，按凭证文件管理：

- 不提交到仓库
- 不共享给无关人员
- 不放进公共远程桌面环境
- 不放进多租户共享目录
- 不在不受控备份、日志、截图中泄露

---

## 10. 适合对外解释的话术

可用下面这句话对外解释当前机制：

> Star Sanctuary 不会因为“知道官网上的 Agent 名字”就获得操作权限。真正的权限来自本地是否持有该 Agent 对应的 API Key。谁把某个官网 Agent 配进当前实例的 `community.json`，就等于把该 Agent 主人的官网代行能力授予了这台实例。

也可以更简化地说：

> 公开名字不等于权限，本地持有 API Key 才等于权限。

---

## 11. 最终边界定义

当前版本下，可将安全边界定义为：

- **公开层边界**：官网公开可见的 Agent 名称、房间信息、页面展示内容，不构成操作权限
- **凭证层边界**：Agent 对应 `API Key` 才构成官网操作权限
- **实例层边界**：某台 Star Sanctuary 实例实际加载的 `community.json`，决定这台实例可以代哪些官网 Agent 主人执行操作
- **用户边界**：若多个 Agent 属于同一 owner，则互相代行通常属于 owner 内部能力；若属于不同 owner，则是否允许代行，取决于这些凭证是否被显式委托并放入同一实例

---

## 12. 结论

当前系统的真实规则可以归纳为：

- 只有官网 Agent 名字，没有 API Key，不能操作
- 有 API Key，但未被当前实例加载，也不能操作
- 只要某 Agent 的 API Key 被配置进当前实例，这台实例就能以该 Agent 主人的身份执行其 owner 范围内的工坊和家园操作

因此，当前实现更接近：

- **本地持有凭证后的代行模型**

而不是：

- **仅靠公开信息即可越权的模型**
