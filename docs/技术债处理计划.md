# WebChat 技术债处理计划

## 1. 背景与目标

当前 [app.js](/E:/project/star-sanctuary/apps/web/public/app.js) 已达到高风险体量区间，单文件同时承载了聊天、WebSocket 通信、附件、设置、文件树/编辑器、记忆查看、长期任务、Canvas 桥接等多个子系统。继续在该文件内直接叠加功能，会明显增加回归风险、性能波动和问题定位成本。

本计划的核心目标只有一个：**在保证当前项目功能正常运行的前提下，渐进式拆分 `app.js`，并在拆分过程中同步完成性能收敛与问题复核。**

## 2. 基线

在开始正式拆分前，已确认当前仓库基线可工作：

- `corepack pnpm test` 通过
- `corepack pnpm build` 通过
- `node --check apps/web/public/app.js` 通过
- [index.html](/E:/project/star-sanctuary/apps/web/public/index.html) 已使用 `type="module"` 加载 [app.js](/E:/project/star-sanctuary/apps/web/public/app.js)，可直接沿浏览器原生 ES Modules 路线拆分

## 3. 约束原则

- 功能稳定优先于技术债治理；一旦出现冲突，以保持现有功能正常运行为准。
- 不引入 React/Vue/Vite/Webpack 等新框架或新构建链。
- 拆分方式采用 **Feature-based**，但允许保留极薄的共享层用于承载公共状态、DOM 引用和 RPC 桥接。
- 每完成一个拆分阶段，必须先回归验证，再进入下一阶段。
- 性能优化只做“行为不变”的收敛，不做大范围交互改造。
- 安全与漏洞复核贯穿每个阶段，不单独等待到最后一次性处理。

## 4. 目标结构

目标是逐步把 `apps/web/public` 的前端逻辑演进为如下结构：

- `app.js`
  - 仅保留 bootstrap、全局协调、兼容暴露接口
- `app/features/*`
  - 按功能域拆分模块
- `app/shared/*`
  - 仅放置少量共享状态、通用工具、RPC/事件桥接

初步模块边界如下：

- `auth-workspace`
- `prompt-composer`
- `voice`
- `attachments`
- `settings`
- `tool-settings`
- `file-tree`
- `editor`
- `chat-network`
- `memory-viewer`
- `goals`
- `canvas-bridge`

## 5. 分阶段实施计划

### Phase 0：基线与回归护栏

目标：

- 固化当前可工作的前端基线
- 为后续模块化拆分补最小回归护栏

实施项：

- 增加 WebChat 模块结构校验脚本
- 校验 `index.html` 的模块入口与本地模块文件存在性
- 对 `app.js` 及后续拆分出的模块执行语法校验

风险点：

- 仅依赖手工验证，后续拆分时容易出现“模块加载成功但事件未绑定”的隐性回归

验收：

- `corepack pnpm verify:webchat` 通过
- `corepack pnpm test` 通过
- `corepack pnpm build` 通过

### Phase 1：低耦合模块拆分

目标：

- 优先抽离不依赖聊天主流程的低耦合逻辑
- 建立 `app/features` 的目录边界

范围：

- `auth/workspace` 本地持久化与恢复
- `prompt/composer` 输入框高度与提交交互
- `voice shortcut` 与录音控制的边界梳理

风险点：

- 本地存储 key 行为变化
- 快捷键与输入框事件次序改变

验收：

- 刷新页面后 Auth / UUID / Workspace Roots 保持原行为
- 输入框回车发送、Shift+Enter 换行、自动高度保持原行为
- 语法校验和仓库构建通过

### Phase 2：中耦合模块拆分

目标：

- 进一步压缩 `app.js` 的 UI 杂糅代码

范围：

- `attachments`
- `settings`
- `tool-settings`
- `file-tree`
- `editor`

风险点：

- 附件大小限制与图片压缩边界
- `.env` 编辑和工具设置确认弹窗行为漂移
- 文件树与编辑器状态同步失效

验收：

- 附件上传、图片压缩、语音附件发送正常
- 设置读写、工具设置读写、编辑器打开保存正常

### Phase 3：高耦合主链路拆分

目标：

- 收缩 `app.js` 为入口协调器

范围：

- `chat-network`
- `memory-viewer`
- `goals`
- `canvas-bridge`

风险点：

- `handleEvent` 主链路回归
- WebSocket 请求-响应映射失效
- Goal / Memory / Canvas 三者联动故障

验收：

- 连接、发送消息、流式消息展示正常
- Memory Viewer、Goals、Canvas 入口与联动保持原行为

### Phase 4：性能收敛与问题复核

目标：

- 在模块边界清晰后做不改变行为的性能优化
- 复核新增路径是否引入错误、问题或漏洞

重点项：

- 减少重复 DOM 查询与重复事件绑定
- 减少重复读取与重复解析
- 评估非首屏逻辑的延迟初始化空间
- 复核 WebChat、Gateway、Goal/Memory/Canvas 相关接口边界

验收：

- 无新增功能回归
- 输出剩余风险清单与后续建议

## 6. 验证矩阵

每个阶段至少回归以下关键链路：

- WebSocket 连接与重连
- 发送消息与流式回复
- Auth / UUID / Workspace Roots 恢复
- Settings / Tool Settings
- 文件树 / 编辑器

- Memory Viewer

## 7. 回滚策略

- 采用渐进式拆分，不做一次性大迁移。
- 每一批拆分控制在可单独回退的范围内。
- 若某阶段出现功能异常，优先回退该阶段的模块接入，不连带影响其他阶段。

## 13. Phase 4 结果（2026-03-21）

### 已落地收敛

- 已修复 `message.send` 错误分支中直接引用未定义 `botMsgEl` 的前端错误，避免在 `pairing_required / config_required` 场景下触发运行时异常。
- 已补齐 pairing 提示链路中的 HTML 转义：
  - `sendMessage` 本地错误提示
  - `chat-events` 的 `pairing.required` 事件展示
- 已将高频状态展示节点改为启动期缓存引用，减少重复 DOM 查询：
  - token usage 区
  - task token usage 面板
  - restart overlay 倒计时区
- 已为聊天视频缩略图提取补 `try/catch` 防护，避免跨域或编解码限制导致 `canvas.toDataURL()` 失败后打断消息渲染。

### 评估结论

- 当前前端主路径未发现需要立即处理的新增高风险漏洞。
- `WebSocket / workspace / origin / attachment limit` 相关关键边界已有仓库测试覆盖，本轮复查未发现回归。
- `app.js` 仍有若干 `innerHTML + querySelectorAll + addEventListener` 的渲染后绑定模式，但主要集中在：
  - `memory-task-detail`
  - `goals` 审批与 checkpoint workflow
  - 若无真实性能瓶颈数据，当前不建议为了“代码形态更漂亮”而继续重写为统一事件委托。

### 剩余观察项

- `memory-task-detail` 仍是前端剩余最明显的性能/复杂度热点：
  - 每次 detail 渲染都会整块重绘并重新绑定多组按钮事件
  - 但其同时承担 Goal 跳转、candidate 展开、usage revoke 等动作链，继续收敛应以真实卡顿或事件重复问题为前提
- `goals` 主流程仍是当前最不适合硬拆的区域：
  - 审批动作
  - checkpoint workflow
  - 缓存/回写编排
  - 后续更适合基于行为回归测试或 profiling 数据定点优化，而不是继续结构性拆分
