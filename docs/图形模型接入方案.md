# 图形模型接入方案

PLAN

Goal
将 `gpt-image-2` 作为独立图片生成工具接入 Star Sanctuary，不复用主 Agent 的 `BELLDANDY_OPENAI_API_KEY / BELLDANDY_OPENAI_BASE_URL`；主 Agent 与子 Agent 都可以通过工具调用触发图片生成；生成结果在 WebChat 消息区可直接预览，并在预览下方显示保存路径，点击该路径打开生成图片所在目录。

Constraints
- 不把 `gpt-image-2` 当主 Agent 模型使用。主 Agent 仍走现有文本/工具调用模型链路。
- 图片生成能力走独立环境变量，避免和主模型线路、密钥、成本策略耦合。
- 保持最小改动，优先复用现有 `multimedia` 工具组、`/generated` 静态托管、WebChat 消息渲染能力。
- WebChat 里“图片预览”和“打开目录”是两个不同交互，不建议让同一个点击动作同时承担两者。
- 打开本地目录不能只靠前端完成，需要新增一个网关侧 query runtime 或 HTTP/RPC 动作，由本机进程调用系统资源管理器。

Steps
1. 为图片生成工具定义独立配置面：`BELLDANDY_IMAGE_*`。
2. 重写 `image_generate` 的运行逻辑，切到 `gpt-image-2`，解析 `b64_json` 并把图片落盘到 `generated/`。
3. 让工具输出稳定的消息格式，保证 WebChat 能直接显示图片预览。
4. 为 WebChat 在图片预览下方增加“保存路径”显示，并让该路径点击后触发网关侧本地目录 reveal。
5. 补测试与手测路径，覆盖未配置、生成成功、消息预览、路径点击打开目录四类场景。

Validation
- Unit: `image_generate` 未配置专用图片密钥时报错；拿到 `b64_json` 时能正确写文件并返回可访问路径。
- Integration: Agent 在启用 `multimedia` 组时可调用 `image_generate`，主 Agent/子 Agent 行为一致。
- Manual: WebChat 消息区可看到缩略图；点击缩略图可看大图；点击保存路径可在本机资源管理器打开目标目录或高亮目标文件。

## 1. 目标与边界

本方案的目标是把 `gpt-image-2` 接入为 **独立图片生成工具**，而不是把它混入主 Agent 模型配置。

当前仓库已经有以下基础能力：

- `image_generate` 已经是 builtin multimedia tool。
- `multimedia` 工具组已经接入 Gateway。
- `/generated` 已由网关静态托管，适合回传图片预览路径。
- WebChat 已能把消息中的 Markdown 图片渲染为可点击缩略图。

因此本次应沿着“工具能力”继续做，而不是改动主 Agent 模型通道。

## 2. 配置方案

### 2.1 新增专用环境变量

建议增加以下环境变量：

```env
BELLDANDY_IMAGE_PROVIDER=openai
BELLDANDY_IMAGE_OPENAI_API_KEY=
BELLDANDY_IMAGE_OPENAI_BASE_URL=https://api.openai.com/v1
BELLDANDY_IMAGE_MODEL=gpt-image-2
BELLDANDY_IMAGE_OUTPUT_FORMAT=png
BELLDANDY_IMAGE_TIMEOUT_MS=60000
BELLDANDY_IMAGE_ENABLED=true
```

可选扩展：

```env
BELLDANDY_IMAGE_OUTPUT_DIR=
BELLDANDY_IMAGE_QUALITY=auto
BELLDANDY_IMAGE_BACKGROUND=auto
BELLDANDY_IMAGE_MODERATION=auto
```

### 2.2 配置原则

- **不回退**到 `BELLDANDY_OPENAI_API_KEY / BELLDANDY_OPENAI_BASE_URL`
- **不回退**到旧的 `OPENAI_API_KEY / OPENAI_BASE_URL`
- 图片能力未配置时应明确报错，而不是静默复用主模型配置

这样做的好处：

- 成本隔离
- 权限隔离
- 线路隔离
- 更利于后续接入其他生图服务商

## 3. 工具接入设计

### 3.1 目标文件

- `packages/belldandy-skills/src/builtin/multimedia/image.ts`
- `packages/belldandy-skills/src/builtin/multimedia/index.ts`
- `packages/star-sanctuary-distribution/src/templates/default-env/runtime.env`

### 3.2 工具行为

`image_generate` 保持现有工具名不变，继续给 Agent 作为标准工具使用。

建议参数：

```json
{
  "prompt": "string, required",
  "size": "1024x1024 | 1536x1024 | 1024x1536 | auto",
  "quality": "low | medium | high | auto",
  "output_format": "png | jpeg | webp",
  "background": "transparent | opaque | auto"
}
```

说明：

- `prompt` 必填
- 其余参数可选，默认值从 `BELLDANDY_IMAGE_*` 推导
- 工具描述里应明确写“使用独立图片模型生成图片”

### 3.3 运行流程

1. 读取 `BELLDANDY_IMAGE_*`
2. 校验 `BELLDANDY_IMAGE_ENABLED`
3. 校验 `BELLDANDY_IMAGE_OPENAI_API_KEY`
4. 调用 `openai.images.generate()`，模型默认 `gpt-image-2`
5. 读取响应中的 `b64_json`
6. 将二进制写入 `<stateDir>/generated/` 或约定子目录
7. 构造 `/generated/...` Web 路径
8. 返回可被 WebChat 直接渲染的输出文本

### 3.4 输出格式

建议工具结果输出为：

```md
![Generated Image](/generated/images/<filename>.png)

保存位置：`generated/images/<filename>.png`

Prompt: ...
Model: gpt-image-2
```

但实际渲染时，建议不要只依赖一段纯代码块路径文本。更稳的方案是：

- 保持 Markdown 图片，确保缩略图能出现
- 在下方附加一条“保存位置”路径链接
- WebChat 在渲染后为该节点补事件

更推荐的输出载体是：

```html
<div class="generated-image-result">
  <img src="/generated/images/<filename>.png" alt="Generated Image" />
  <div>
    <span>保存位置：</span>
    <a href="#generated-image-reveal:/generated/images/<filename>.png">generated/images/&lt;filename&gt;.png</a>
  </div>
</div>
```

前提是保留在 WebChat sanitizer 白名单内，并由前端绑定点击事件。缩略图本身继续沿用现有大图预览逻辑，不承担 reveal 行为。

## 4. 文件落盘策略

### 4.1 保存路径

建议默认保存到：

```text
<stateDir>/generated/images/
```

对应 Web 路径：

```text
/generated/images/<filename>.png
```

理由：

- 与现有 `/generated` 托管一致
- 不污染工作区
- 能被 WebChat 直接访问
- 后续可对该目录做清理策略或审计

### 4.2 文件命名

建议：

```text
image-<YYYYMMDD-HHmmss>-<shortId>.<ext>
```

例如：

```text
image-20260427-154022-a1b2c3d4.png
```

### 4.3 元数据

可选在同目录写 sidecar json：

```text
image-20260427-154022-a1b2c3d4.json
```

包含：

- prompt
- revisedPrompt（若返回）
- model
- size
- quality
- createdAt
- absolutePath
- webPath

这不是首批必需，可作为 `defer` 项。

## 5. WebChat 预览方案

### 5.1 现状

当前 WebChat 已具备：

- Markdown/HTML 图片渲染
- 将消息内 `img` 转成缩略图
- 点击缩略图弹出大图预览

因此“消息区能看到预览”这部分是 **低成本** 的，只要工具结果里返回稳定的图片 URL 或 `<img>` 即可。

### 5.2 推荐交互

不建议把“点击图片缩略图”直接改成“打开目录”，因为这会覆盖现有的大图预览能力，交互上也不直觉。

推荐：

1. 消息区直接显示缩略图预览
2. 单击缩略图：打开大图预览
3. 在缩略图下方显示一条“保存位置：generated/images/...”
4. 单击该路径：打开生成图片所在目录

这样用户既能看图，也能迅速定位文件。

### 5.3 前端改动点

目标文件：

- `apps/web/public/app/features/chat-ui.js`
- 相关样式文件 `apps/web/public/styles.css`

建议实现：

## 6. “打开保存目录”实现方案

### 6.1 是否好实现

**可以实现，复杂度中等偏低。**

难点不在前端，而在于：

- 浏览器不能直接打开本机资源管理器目录
- 需要网关进程代替前端执行本地系统动作

所以这不是纯前端功能，而是一个“小型前后端联动改动”。

### 6.2 推荐方案

新增一个 query runtime 方法，例如：

```text
artifact.reveal
```

参数：

```json
{
  "path": "/generated/images/image-20260427-154022-a1b2c3d4.png"
}
```

网关侧负责：

1. 把 `/generated/...` 或相对 artifact 路径解析到本地绝对路径
2. 校验目标必须位于 `generatedDir` 下
3. 调用平台相关命令打开目录或高亮文件

### 6.3 平台动作建议

- Windows: `explorer.exe /select,<absolute-file-path>`
- macOS: `open -R <absolute-file-path>`
- Linux: 优先目录打开；高亮文件能力可降级为打开所在目录

### 6.4 安全约束

- 只允许 reveal `generatedDir` 下的文件
- 不允许任意路径
- 失败时返回明确错误消息
- 不通过 Agent 工具直接暴露任意“打开路径”能力

这一步虽然会新增本地副作用，但风险可控，因为目标路径边界很小且可校验。

## 9. 风险与决策

### 9.1 风险

- OpenAI 图片接口响应结构与老 `dall-e-3` 分支不同，不能沿用旧解析
- 若直接把图片点击行为改成打开目录，会破坏现有预览体验
- OS 目录打开存在平台差异，需要做降级兼容

### 9.2 技术债决策

- `fix_now`: 独立 `BELLDANDY_IMAGE_*` 配置、`gpt-image-2` 工具接入、WebChat 预览、artifact reveal
- `defer`: sidecar 元数据文件、历史图片面板、批量清理策略、多图片结果浏览
