# 摄像头功能改造

## Goal

为 Agent 提供一套相对通用、稳定、可扩展的摄像头调用方案，先覆盖当前最现实的浏览器回环场景，再为后续接入原生节点 / 外部设备 provider 留出统一接口。

补充目标：

- 必须把“本机只有外接摄像头设备”的桌面场景纳入正式方案，而不是只把它当成浏览器枚举的附带结果
- 最终需要支持本机外接 USB 摄像头 / 采集卡类设备的稳定调用
- Agent 看到的接口应保持统一，不能因为底层来自浏览器、桌面原生还是远端 node 而分裂成三套工具

## Why

当前仓库里已经有浏览器能力、截图能力和 `mirror.html`，但过去的摄像头调用更像一次性链路，不适合作为长期的 Agent 能力面：

- 设备发现、权限状态、采集状态没有统一 runtime 表达
- 参数层面缺少统一的 `facing / deviceId / width / height / fit / mirror`
- 工具层面缺少稳定的 `camera_list`
- 后续如果要接入 node / 原生摄像头 provider，会直接把实现绑死在浏览器路径上

## 参考输入

### 仓库现状

- `packages/belldandy-skills/src/builtin/multimedia/camera.ts`
- `packages/belldandy-skills/src/builtin/multimedia/camera-runtime.ts`
- `apps/web/public/mirror.html`

当前最适合先落地的路径，是通过浏览器打开镜像页，使用 `getUserMedia()` 建立视频流，再由 Agent 走元素级截图获取静态照片。

### OpenClaw 参考

`tmp/openclaw/openclaw-main/docs/nodes/camera.md` 给出的经验比较清晰：

- 能力面拆成 `camera.list` / `camera.snap` / `camera.clip`
- 参数统一包含 `facing`、`deviceId`、尺寸 / 时长等
- 明确权限 gating、前台约束、错误码和 payload 限制
- `camera.list` 先做设备发现，`camera.snap` / `camera.clip` 再做采集

这些点对 Star Sanctuary 也成立，只是我们的第一阶段 provider 不是原生 node，而是 `browser_loopback`。

另外，OpenClaw 的 `node.invoke + camera.list/snap/clip` 思路也说明了一点：

- Agent 不应该直接理解平台细节
- 平台差异应该落在 provider / node 适配层
- `camera.list` 必须先成为稳定能力，而不是仅靠临时拍照碰运气

### Context7 / MDN 参考

基于 `/mdn/content` 检索到的 Web Camera 主文档结论：

- 摄像头主入口应使用 `navigator.mediaDevices.getUserMedia()`
- 设备选择应同时支持 `facingMode` 偏好和 `deviceId` 精确选择
- 设备列表来自 `navigator.mediaDevices.enumerateDevices()`
- 必须显式处理 `NotAllowedError`、`OverconstrainedError` 等权限 / 约束失败
- 在浏览器里拍静态图时，`video.srcObject` + `canvas` / 元素截图是标准路径

## 目标能力面

### 对 Agent 暴露的工具

第一阶段：

- `camera_list`
- `camera_snap`

第二阶段：

- `camera_clip`
- 统一的摄像头诊断 / 权限状态输出

### 统一输入参数

- `facing`: `front | back`
- `deviceId`: 指定具体摄像头
- `deviceRef`: 面向长期使用的逻辑设备引用，后续优先于瞬时 `deviceId`
- `provider`: 显式指定 provider，避免多 provider 并存时路由不确定
- `nodeId`: 当 provider 为 `node_device` 时指定目标设备节点
- `width` / `height`: 期望采集分辨率
- `fit`: `cover | contain`
- `mirror`: 是否镜像预览
- `delayMs`: 拍摄前等待，用于对焦 / 曝光 / 页面稳定
- `readyTimeoutMs`: 等待 provider 进入 ready 的超时

`camera_clip` 额外补：

- `durationMs`
- `includeAudio`
- `format`

### 统一输出结构

- `provider`
- `mirrorUrl` 或 provider 会话标识
- `state`
- `device`
- 媒体产物路径
- 设备元信息
- 失败时的结构化错误信息

## 外接摄像头是一等场景

这部分不是可选优化，而是正式目标。

原因很直接：

- 很多桌面机器没有内置摄像头，只有外接 USB 摄像头
- 直播 / 录课 / 会议设备经常以外接摄像头或采集卡形式存在
- 浏览器 `deviceId` 不稳定、权限依赖强，不能作为外设接入的最终方案
- Agent 长期任务如果依赖“浏览器恰好还连着、恰好拿到那个 deviceId”，稳定性不够

因此最终方案必须支持：

- 枚举本机外接摄像头
- 用稳定设备引用重新找到该设备
- 在不依赖浏览器页面的情况下完成拍照 / 录像
- 将产物按统一 contract 返回给 Agent

## 方案设计

### 1. Provider Registry

摄像头能力不直接绑在某一个实现上，而是抽象成 provider：

- `browser_loopback`
- `native_desktop`
- `node_device`

统一对外暴露：

- `listDevices()`
- `captureSnapshot()`
- `captureClip()`
- `diagnose()`

这样后面即使接入手机节点、桌面原生摄像头，也不需要重写 Agent 的工具契约。

建议把 camera 能力做成和邮件 provider 类似的 registry，而不是继续把逻辑堆回单个 `camera-runtime.ts`。

### 1.1 Provider Contract 草案

建议抽成独立类型层，例如：

```ts
export type CameraProviderId =
  | "browser_loopback"
  | "native_desktop"
  | "node_device";

export type CameraDeviceSource =
  | "integrated"
  | "external"
  | "virtual"
  | "capture_card"
  | "unknown";

export type CameraDeviceDescriptor = {
  provider: CameraProviderId;
  deviceId: string;
  deviceRef: string;
  stableKey?: string;
  label: string;
  facing?: "front" | "back";
  source: CameraDeviceSource;
  transport: "browser" | "native" | "node";
  external: boolean;
  available: boolean;
  metadata?: Record<string, unknown>;
};

export type CameraProviderCapabilities = {
  list: boolean;
  snapshot: boolean;
  clip: boolean;
  audio: boolean;
  hotplug: boolean;
  background: boolean;
};

export interface CameraProvider {
  readonly id: CameraProviderId;
  diagnose(context: CameraProviderContext): Promise<CameraProviderDiagnostic>;
  listDevices(input: CameraListRequest, context: CameraProviderContext): Promise<CameraListResponse>;
  captureSnapshot(input: CameraSnapshotRequest, context: CameraProviderContext): Promise<CameraSnapshotResponse>;
  captureClip?(input: CameraClipRequest, context: CameraProviderContext): Promise<CameraClipResponse>;
}
```

这里最关键的不是字段多，而是两点：

- `provider` 和 `deviceRef` 必须进入统一 contract
- `external` / `source` 必须被显式表达，否则外接设备永远只是“某个 label”

### 1.2 DeviceRef 设计

不能只靠浏览器或系统临时生成的 `deviceId`。

建议区分：

- `deviceId`
  当前 provider 会话内可直接调用的瞬时 ID
- `deviceRef`
  对 Agent 暴露的逻辑设备引用
- `stableKey`
  provider 内部用于设备再识别的稳定键

推荐格式：

- `browser_loopback:rear-default`
- `native_desktop:usb-logitech-brio`
- `node_device:<nodeId>:front`

本机外接摄像头优先依赖 `stableKey + label + hardwarePath` 组合做稳定识别，再映射为对 Agent 友好的 `deviceRef`。

### 2. Browser Loopback Runtime

第一阶段先落浏览器 provider，原因是：

- 项目已经具备浏览器接线和截图能力
- 不需要新增本地原生依赖
- 调试链路短，适合先打通 Agent 侧体验

运行链路：

1. `camera_list` / `camera_snap` 归一化参数
2. runtime 生成镜像页 URL
3. BrowserManager 复用或打开 `mirror.html`
4. 镜像页通过 `getUserMedia()` 建流
5. 镜像页通过 `window.__BELLDANDY_CAMERA_MIRROR__` 暴露状态桥
6. runtime 等待 `ready`
7. `camera_list` 读取设备与状态
8. `camera_snap` 对视频元素截图并落盘

它仍然保留价值，但应该明确定位为：

- 快速可用 provider
- Web 权限受控 provider
- 不是最终唯一 provider
- 不负责解决“只有外接摄像头且浏览器不稳定”这一类长期问题

### 2.1 Browser Provider 的边界

更明确地说，`browser_loopback` 适合：

- 已经连接浏览器
- 需要快速拍一张图
- 允许浏览器弹权限框

不适合作为最终主方案的场景：

- 长时间后台驻留
- 设备被浏览器 / 会议软件争抢
- 外接摄像头需要稳定重连
- 无浏览器或浏览器会话不可控

### 3. Mirror Page 责任边界

`apps/web/public/mirror.html` 只负责浏览器侧的媒体生命周期：

- 解析 URL 参数
- 调用 `getUserMedia()`
- 维护当前 stream
- 暴露设备列表
- 响应 `devicechange`
- 输出 `booting / requesting-permission / ready / error`

镜像页不直接承担 Agent 业务语义，业务语义放在 skills runtime。

### 4. Runtime State Model

统一状态对象至少包含：

- `status`
- `selectedFacing`
- `selectedDeviceId`
- `devices`
- `videoWidth` / `videoHeight`
- `settings`
- `error`
- `startedAt` / `updatedAt`

这样 Agent 拿到的不是“拍到了 / 没拍到”的二元结果，而是可解释的运行时状态。

建议进一步把 provider 诊断状态也纳入统一模型：

- `providerStatus`: `available | unavailable | degraded`
- `permissionState`
- `selectionReason`
- `fallbackChain`
- `lastSuccessfulCaptureAt`

### 5. 错误模型

错误分三层：

- provider 不可用：浏览器未连接、镜像页不可达
- 权限 / 设备失败：`NotAllowedError`、`NotFoundError`、`OverconstrainedError`
- 采集失败：ready 超时、视频元素缺失、截图失败

工具输出应尽量保留原始错误名和可读 message，避免只返回“拍摄失败”。

对外接摄像头，还需要额外覆盖：

- 设备被拔出
- 设备被其他进程独占
- 驱动异常或能力协商失败
- provider helper 未启动
- helper 已启动但无权限或未就绪

## 原生节点 / 外部设备方案

### 6. 目标 provider 列表

建议正式收敛为三类：

- `browser_loopback`
  当前已实现，适合浏览器回环
- `native_desktop`
  面向本机桌面摄像头，优先覆盖你的“只有外接摄像头”的场景
- `node_device`
  面向未来的手机 / Companion / 远端设备节点

### 6.1 `native_desktop` 的实现策略

不建议把平台原生摄像头采集代码直接塞进主 Gateway 进程。

更稳妥的做法是：

- 由主进程维护统一 camera contract
- `native_desktop` 通过本地 sidecar / bridge helper 实现
- helper 负责和操作系统摄像头 API 交互
- 主仓库只和 helper 通过 `stdio`、本地 HTTP 或 domain socket 通讯

这样做的好处：

- 平台差异被隔离，不污染主仓库
- 原生依赖、驱动问题、崩溃影响面更小
- 将来可以先做 Windows，再做 macOS / Linux
- 更符合仓库里现有 bridge / MCP / wrapper 的总体思路

### 6.2 为什么优先 sidecar / bridge，而不是主进程直连

对于本机外接摄像头，真正麻烦的不是“拍一张图”，而是：

- 设备热插拔
- 驱动和采集栈差异
- 进程独占与抢占
- 长时间运行时的资源释放
- 原生依赖对构建链的污染

sidecar 更适合承担这些责任：

- 枚举设备
- 生成稳定设备键
- 管理打开 / 关闭 / 重连
- 拍照 / 录像
- 输出文件路径和元信息
- 上报诊断状态

### 6.3 Windows 本机外接摄像头优先方案

考虑到你当前机器是 Windows，且明确依赖外接摄像头，建议实现顺序是：

1. 先定义 `native_desktop` provider contract
2. 先做 Windows helper
3. 优先支持外接 USB 摄像头拍照
4. 再扩到录像、音频、热插拔监听

实现手段上，文档层面建议保持开放，但工程上优先顺序建议是：

- 首选：专用 helper，内部使用平台摄像头 API
- 过渡方案：helper 内部包一层现成采集能力
- 不建议：让 Gateway 直接依赖某个 Windows 专有 capture 库

这样即使后面 helper 的内部实现从一种采集方式切换到另一种，Agent 侧 contract 也不需要变。

### 6.4 `node_device` provider 的定位

`node_device` 不是本机外接摄像头的替代，而是第二条能力线。

它适合：

- 手机 / 平板摄像头
- 远端 companion 设备
- 专门挂在局域网或工位上的采集节点

它的 contract 应该与 `native_desktop` 对齐，只是底层调用来自 `node.invoke` 或未来的设备桥接层。

### 6.5 Provider 选择策略

建议统一路由规则：

1. 用户显式传 `provider` 时，严格按指定 provider 走
2. 用户传 `deviceRef` 时，优先解析其 provider 前缀
3. 如果检测到本机 `native_desktop` 可用，且存在外接设备，桌面场景默认优先 `native_desktop`
4. 如果浏览器已连接且用户未指定 provider，可回退 `browser_loopback`
5. 远端设备或显式 `nodeId` 时走 `node_device`

核心原则：

- 设备选择尽量稳定
- provider 选择尽量可解释
- 回退链路必须可见，不能静默切 provider

### 6.6 外接设备识别策略

外接摄像头需要显式识别，否则系统换口、重启、驱动刷新后就会漂。

建议 provider 内部至少采集：

- friendly name
- hardware path / instance path
- vendor / product 信息
- 是否外接
- 是否虚拟设备
- 当前能力集

然后生成：

- `stableKey`
- `deviceRef`

这样 Agent 可以记住“会议室主摄像头”或“桌面 USB 摄像头”，而不是记住一次性的 `deviceId`。

### 6.7 Artifact Contract

不论底层 provider 是浏览器、桌面 helper 还是 node，都应该尽量统一产物行为：

- 图片 / 视频先落本地文件
- 返回路径、尺寸、格式、时长、设备信息
- 大媒体默认不直接内联进模型上下文
- 需要时再通过附件或后续工具读取

这点对外接摄像头尤其重要，因为视频或高分辨率图片很容易放大上下文成本。

## 当前实现落点

本轮已经先把第一阶段核心骨架落下：

- `camera-runtime.ts` 作为浏览器摄像头 runtime
- `camera_list` 作为设备探测入口
- `camera_snap` 继续保留，并改成走统一 runtime
- `mirror.html` 暴露状态桥和设备列表

这意味着方案已经不是纯文档，而是“文档 + 第一阶段可运行骨架”。

但也要明确：

- 当前还没有 provider registry
- 当前还没有 `native_desktop`
- 当前外接摄像头仍主要依赖浏览器 provider 枚举
- 这对你当前机器来说只是过渡状态，不是终态

## 兼容性与约束

- 第一阶段依赖浏览器已连接并允许镜像页访问摄像头
- `enumerateDevices()` 在未授权前可能拿不到完整 label，这属于浏览器标准行为
- 浏览器 provider 更适合静态拍照；`camera_clip` 需要额外考虑录制生命周期和文件落盘
- 当前方案不引入新的原生依赖，优先保证仓库现有运行方式不被破坏

补充约束：

- `native_desktop` 需要明确 helper 部署方式
- Windows 外接摄像头可能被会议软件独占，需要设计可解释错误
- `deviceRef` 必须考虑设备重插、换 USB 口、驱动刷新后的重识别
- 多 provider 并存时，默认选择逻辑不能让用户“以为拍的是 A，实际拍的是 B”

## Validation

### Unit

- 参数归一化测试
- 状态桥读取测试
- ready / error 等待逻辑测试

### Manual

- 启动 Gateway
- 连接浏览器
- 调用 `camera_list`
- 调用 `camera_snap`
- 验证：
  - 镜像页能拉起
  - 摄像头权限提示正常
  - 能看到设备列表
  - 截图文件真实落盘

### Integration

- 在仅有外接摄像头的 Windows 机器上验证 `native_desktop`
- 验证外接摄像头拔插前后 `deviceRef` 是否还能正确重映射
- 验证同一台机器上 `browser_loopback` 与 `native_desktop` 共存时的默认选择和显式选择
- 验证 `node_device` 返回结构是否和本机 provider 一致

### Regression Focus

- 不影响现有 browser 工具
- 不影响现有截图目录结构
- `camera_snap` 在未授权场景下能返回可解释错误

## 技术债决策

- `fix_now`: 统一浏览器摄像头 runtime 和 `camera_list`
- `fix_now`: 把 `native_desktop` 和外接摄像头支持写进正式设计，不再停留在“以后再说”
- `split_task`: provider registry、Windows helper、node provider 分阶段落地
- `defer`: `camera_clip` 和更完整的诊断 UI
- `record_only`: 更完整的 WebChat 诊断 UI，等摄像头能力面稳定后再补
