# 长期任务使用指南

> 适用范围：Star Sanctuary 当前已落地的超长期任务 / Long-term Goal 能力  
> 对应实现主线：`docs/超长期任务系统实现方案.md`

---

## 0. 先说结论

### 0.1 最近新增能力有没有引入新的“必填配置”？

没有。

最近补完的这几项：

- `cron` 自动 approval scan
- reviewer / template / SLA policy 运行态
- 更细粒度 notification dispatch / outbox 运行态
- unified approval panel

**都没有新增必须填写的环境变量。**

新增内容里，只有下面这个文件属于“用户可选配置”：

- `~/.star_sanctuary/governance/review-governance.json`

其余新增文件大多是**自动生成的运行态文件**，用户通常**不需要手工创建**：

- `~/.star_sanctuary/cron-jobs.json`
- `<goal.runtimeRoot>/review-notifications.json`
- `<goal.runtimeRoot>/review-notification-dispatches.json`
- `<goal.runtimeRoot>/suggestion-reviews.json`
- `<goal.runtimeRoot>/publish-records.json`

## 1. 长期任务系统整体结构

长期任务系统分成两层：

1. **全局状态目录**
2. **单个 goal 的文档目录 + 运行态目录**

### 1.1 全局状态目录

默认位置：

- `~/.star_sanctuary`

可通过环境变量覆盖：

- `BELLDANDY_STATE_DIR`

全局目录下与你最相关的文件/目录：

- `~/.star_sanctuary/goals/index.json`
  - 所有长期任务注册表
- `~/.star_sanctuary/docs/long-tasks/`
  - 所有长期任务文档目录
- `~/.star_sanctuary/governance/review-governance.json`
  - 组织级 reviewer / template / reminder / SLA / notification policy
- `~/.star_sanctuary/cron-jobs.json`
  - 定时任务注册表
- `~/.star_sanctuary/goals/cross-goal-flow-patterns.json`
  - 跨 goal 流程模式归并结果
- `~/.star_sanctuary/docs/long-tasks/cross-goal-flow-patterns.md`
  - 跨 goal 流程模式文档视图
- `~/.star_sanctuary/methods/`
  - 正式发布的方法资产
- `~/.star_sanctuary/skills/`
  - 正式发布的技能资产

### 1.2 单个 Goal 的两套目录

每个长期任务都会同时有：

- **文档目录**：位于 `~/.star_sanctuary/docs/long-tasks/<slug>/`
- **运行态目录**：默认位于 `~/.star_sanctuary/goals/<goalId>/`

如果你在创建 goal 时指定了 `goalRoot`，那么：

- `runtimeRoot = goalRoot`
- 文档目录仍然在 `~/.star_sanctuary/docs/long-tasks/<slug>/`

## 2. 环境变量与配置：哪些是必需，哪些是可选

## 2.1 最小必需配置

如果你希望长期任务具备真实 AI 能力，至少建议配置：

```env
BELLDANDY_AGENT_PROVIDER=openai
BELLDANDY_OPENAI_BASE_URL=https://api.openai.com/v1
BELLDANDY_OPENAI_API_KEY=<your-api-key>
BELLDANDY_OPENAI_MODEL=<your-model>
```

说明：

- `BELLDANDY_AGENT_PROVIDER=openai`
  - 使用真实模型
- `BELLDANDY_OPENAI_BASE_URL`
  - OpenAI 或兼容 OpenAI 的服务地址
- `BELLDANDY_OPENAI_API_KEY`
  - 模型 API Key
- `BELLDANDY_OPENAI_MODEL`
  - 模型名

如果不配这些，系统仍可创建/浏览长期任务骨架，但很多“生成、分析、编排、建议”能力会受限。

## 2.2 常用但非必需配置

```env
# 可选：覆盖默认状态目录
BELLDANDY_STATE_DIR=C:\Users\admin\.star_sanctuary

# 可选：允许 Agent 在对话中自动调用 goal / cron / file 等工具
BELLDANDY_TOOLS_ENABLED=true

# 可选：启用 cron 调度器
BELLDANDY_CRON_ENABLED=true

# 可选：启用 embedding，提升经验检索/记忆检索质量
BELLDANDY_EMBEDDING_ENABLED=true
BELLDANDY_EMBEDDING_OPENAI_API_KEY=<your-embedding-key>
BELLDANDY_EMBEDDING_OPENAI_BASE_URL=https://api.openai.com/v1
BELLDANDY_EMBEDDING_MODEL=<your-embedding-model>

# 可选：允许 capability plan 结合 MCP 能力
BELLDANDY_MCP_ENABLED=true
```

### 这些配置分别影响什么？

| 配置项 | 是否必需 | 用途 |
|---|---:|---|
| `BELLDANDY_STATE_DIR` | 否 | 覆盖默认状态目录 |
| `BELLDANDY_AGENT_PROVIDER` | 建议必配 | 决定是否使用真实模型 |
| `BELLDANDY_OPENAI_BASE_URL` | provider=openai 时必需 | 模型服务地址 |
| `BELLDANDY_OPENAI_API_KEY` | provider=openai 时必需 | 模型鉴权 |
| `BELLDANDY_OPENAI_MODEL` | provider=openai 时必需 | 模型名 |
| `BELLDANDY_TOOLS_ENABLED` | 否，但强烈建议 | 让 Agent 能自动调用 goal / cron / workspace 工具 |
| `BELLDANDY_CRON_ENABLED` | 否 | 启用 cron 自动调度 |
| `BELLDANDY_EMBEDDING_ENABLED` | 否 | 提升经验/记忆检索效果 |
| `BELLDANDY_MCP_ENABLED` | 否 | 让 capability plan / 执行时可接 MCP |

## 3. 用户需要关心的配置文件

## 3.1 `.env.local`（推荐）

建议在项目根目录维护 `.env.local`。

示例：

```env
BELLDANDY_AGENT_PROVIDER=openai
BELLDANDY_OPENAI_BASE_URL=https://api.openai.com/v1

BELLDANDY_TOOLS_ENABLED=true
BELLDANDY_CRON_ENABLED=true

# 可选
BELLDANDY_STATE_DIR=C:\Users\admin\.star_sanctuary
BELLDANDY_EMBEDDING_ENABLED=true
BELLDANDY_EMBEDDING_OPENAI_API_KEY=<your-embedding-key>
BELLDANDY_EMBEDDING_OPENAI_BASE_URL=https://api.openai.com/v1
BELLDANDY_EMBEDDING_MODEL=<your-embedding-model>
```

## 3.2 `review-governance.json`（可选，但强烈建议）

路径：

- `~/.star_sanctuary/governance/review-governance.json`

用途：

- 组织级 reviewer 目录
- 审批模板
- reminder / SLA 默认策略
- notification channel / route 默认策略

当前**没有在线编辑面板**，需要直接编辑这个文件。

最小示例：

```json
{
  "version": 1,
  "reviewers": [
    {
      "id": "producer",
      "name": "制作人",
      "reviewerRole": "owner",
      "channels": ["reviewer_inbox", "im_dm"],
      "tags": ["default", "release"],
      "active": true
    },
    {
      "id": "tech-lead",
      "name": "技术负责人",
      "reviewerRole": "tech_lead",
      "channels": ["reviewer_inbox"],
      "active": true
    }
  ],
  "templates": [
    {
      "id": "tpl_suggestion_quorum",
      "title": "Suggestion 双人会签",
      "target": "suggestion_review",
      "enabled": true,
      "mode": "quorum",
      "reviewers": ["tech-lead", "producer"],
      "minApprovals": 2,
      "slaHours": 24,
      "reminderMinutes": [240, 60],
      "escalationMode": "manual",
      "escalationReviewer": "producer"
    },
    {
      "id": "tpl_checkpoint_single",
      "title": "Checkpoint 单审",
      "target": "checkpoint",
      "enabled": true,
      "mode": "single",
      "reviewers": ["producer"],
      "slaHours": 12,
      "reminderMinutes": [120, 30],
      "escalationMode": "manual",
      "escalationReviewer": "producer"
    }
  ],
  "defaults": {
    "suggestionTemplateId": "tpl_suggestion_quorum",
    "reminderMinutes": [240, 60, 15],
    "notificationChannels": ["goal_detail", "reviewer_inbox"],
    "notificationRoutes": {
      "im_dm": "im://review/{recipient}",
      "webhook": "webhook://review/{recipient}",
      "org_feed": "org://review-feed"
    }
  },
  "updatedAt": "2026-03-21T00:00:00.000Z"
}
```

### `channels` 与 `notificationChannels` 当前可用值

- `goal_detail`
- `goal_channel`
- `reviewer_inbox`
- `org_feed`
- `im_dm`
- `webhook`

注意：

- 当前 `im_dm / webhook` **只写运行态 outbox**
- 不会直接发到真实 IM / webhook
- 对应 dispatch 状态通常会是 `pending`

## 5. 一个长期任务从创建到治理的完整使用流程

下面按“用户实际怎么用”来写。

## 5.1 第一步：初始化环境

推荐顺序：

1. 配置 `.env.local`
2. 运行 `bdd doctor`
3. 启动系统
4. 在 WebChat 设置中确认：
   - Agent provider 已配置
   - Tools 已启用（如需要 Agent 自动调用工具）
   - Cron 已启用（如需要自动扫描）

---

## 5.2 第二步：创建长期任务

### 方式 A：WebChat / Goals 面板

适合普通用户。

创建后系统会自动生成：

- goal 注册表项
- 文档脚手架
- 运行态目录
- 默认 goal conversation

### 方式 B：Tool

工具名：

- `goal_init`

示例：

```json
{
  "title": "Phase 6 审批治理补完",
  "objective": "把 suggestion review、checkpoint、cron approval scan 与 notification dispatch 串成闭环",
  "slug": "phase6-review-governance"
}
```

### 方式 C：RPC

方法名：

- `goal.create`

示例：

```json
{
  "method": "goal.create",
  "params": {
    "title": "Phase 6 审批治理补完",
    "objective": "把 suggestion review、checkpoint、cron approval scan 与 notification dispatch 串成闭环"
  }
}
```

### 创建后会出现哪些文件？

文档目录中会自动创建：

- `00-goal.md`
- `01-target-solution.md`
- `02-design.md`
- `03-architecture-and-implementation.md`
- `04-split-plan.md`
- `05-task-list.md`
- `06-retrospective.md`
- `07-method-candidates.md`
- `08-skill-candidates.md`
- `09-flow-patterns.md`
- `NORTHSTAR.md`
- `tasks.json`
- `progress.md`
- `handoff.md`

运行态目录会自动创建：

- `state.json`
- `runtime.json`
- `checkpoints.json`
- `capability-plans.json`
- `method-candidates.json`
- `skill-candidates.json`
- `flow-patterns.json`
- `suggestion-reviews.json`
- `publish-records.json`
- `review-notifications.json`
- `review-notification-dispatches.json`
- `board-ref.json`
- `runs/`
- `artifacts/`

## 5.3 第三步：拆任务图

你可以先写 `05-task-list.md`，但真正的机器可读任务图以 `tasks.json` / task graph 为准。

常用入口：

### Tool

- `task_graph_create`
- `task_graph_update`
- `task_graph_read`

示例：

```json
{
  "title": "补 unified approval panel",
  "node_id": "node_unified_approval_panel",
  "phase": "phase6",
  "acceptance": [
    "goal detail 内可查看 reviewer/template 摘要",
    "可查看 pending / overdue / notifications / dispatches"
  ],
  "checkpoint_required": true,
  "status": "ready"
}
```

### RPC

- `goal.task_graph.create`
- `goal.task_graph.update`
- `goal.task_graph.read`

---

## 5.4 第四步：进入 Goal 通道并执行节点

### 恢复 Goal / 进入基础通道

- Tool：`goal_resume`
- RPC：`goal.resume`

### 节点认领 / 状态推进

常用入口：

- `task_graph_claim`
- `task_graph_pending_review`
- `task_graph_validating`
- `task_graph_complete`
- `task_graph_block`
- `task_graph_fail`
- `task_graph_skip`

### 推荐做法：优先用 `goal_orchestrate`

`goal_orchestrate` 会：

1. 为节点生成 capability plan
2. 认领节点
3. 高风险时自动发起 checkpoint
4. 如配置允许，可触发最小子代理编排

常用入口：

- Tool：`goal_orchestrate`
- RPC：当前主要通过 tool 使用；底层依赖 `goal.task_graph.* + capabilityPlan`

如果你只想先看计划，不执行编排，也可以用：

- Tool：`goal_capability_plan`

---

## 5.5 第五步：Checkpoint 审批

当节点需要人工审批时，使用 checkpoint 流程。

常用入口：

- `goal_checkpoint_request` / `goal.checkpoint.request`
- `goal_checkpoint_approve` / `goal.checkpoint.approve`
- `goal_checkpoint_reject` / `goal.checkpoint.reject`
- `goal_checkpoint_expire` / `goal.checkpoint.expire`
- `goal_checkpoint_reopen` / `goal.checkpoint.reopen`
- `goal_checkpoint_escalate` / `goal.checkpoint.escalate`

示例：发起 checkpoint

```json
{
  "method": "goal.checkpoint.request",
  "params": {
    "goalId": "goal_phase6_review_governance",
    "nodeId": "node_unified_approval_panel",
    "title": "Approval Panel Checkpoint",
    "summary": "前端面板已接入 dispatch outbox 展示",
    "reviewer": "producer",
    "requestedBy": "main-agent"
  }
}
```

你也可以直接在现有 `goal detail` 面板中点击批准 / 拒绝 / 升级。

## 5.6 第六步：生成复盘、候选资产与模式摘要

当一个 goal 或某些节点完成后，建议执行：

- `goal_retrospect_generate` / `goal.retrospect.generate`
- `goal_method_candidates_generate` / `goal.method_candidates.generate`
- `goal_skill_candidates_generate` / `goal.skill_candidates.generate`
- `goal_flow_patterns_generate` / `goal.flow_patterns.generate`
- `goal_cross_goal_flow_patterns` / `goal.flow_patterns.cross_goal`
- `goal_experience_suggest` / `goal.experience.suggest`

推荐顺序：

1. retrospective
2. method candidates
3. skill candidates
4. flow patterns
5. cross-goal flow patterns
6. experience suggest

## 5.7 第七步：Suggestion Review 与 Publish

这是 Phase 6 的核心治理闭环。

### 先看 review 列表

- `goal_suggestion_review_list`
- `goal.suggestion_review.list`

### 配审批工作流

- `goal_suggestion_review_workflow_set`
- `goal.suggestion_review.workflow.set`

支持：

- `single`
- `chain`
- `quorum`

示例：

```json
{
  "method": "goal.suggestion_review.workflow.set",
  "params": {
    "goalId": "goal_phase6_review_governance",
    "reviewId": "review_method_candidate_node_unified_approval_panel",
    "mode": "quorum",
    "reviewers": ["tech-lead", "producer"],
    "minApprovals": 2,
    "slaHours": 24,
    "escalationMode": "manual",
    "escalationReviewer": "producer"
  }
}
```

### 做审阅决定

- `goal_suggestion_review_decide`
- `goal.suggestion_review.decide`

支持决定：

- `accepted`
- `rejected`
- `deferred`
- `needs_revision`

### 手动升级

- `goal_suggestion_review_escalate`
- `goal.suggestion_review.escalate`

### 发布正式资产

- `goal_suggestion_publish`
- `goal.suggestion.publish`

发布结果：

- method → `~/.star_sanctuary/methods/`
- skill → `~/.star_sanctuary/skills/`
- 同时写入 `publish-records.json`
- 同时同步到全局 `experience_candidates`

## 5.8 第八步：统一审批治理与通知

如果你要查看一个 goal 当前的审批全景，使用：

- `goal_review_governance_summary`
- `goal.review_governance.summary`

它会聚合：

- governance config
- reviewer / template 摘要
- suggestion review 状态
- checkpoint workflow
- publish records
- cross-goal focus patterns
- notifications
- notification dispatches / outbox

### 统一审批扫描

使用：

- `goal_approval_scan`
- `goal.approval.scan`

它会统一扫描：

- suggestion review workflow
- checkpoint workflow

并生成：

- overdue / reminder / auto escalation notifications
- notification dispatch outbox

### 生成后的关键文件

- `<goal.runtimeRoot>/review-notifications.json`
  - 事件层
- `<goal.runtimeRoot>/review-notification-dispatches.json`
  - 下发层 / outbox

当前 dispatch channel 支持：

- `goal_detail`
- `goal_channel`
- `reviewer_inbox`
- `org_feed`
- `im_dm`
- `webhook`

注意：

- 当前 `im_dm / webhook` 只写 outbox，不真实发送
- 因此这是“运行态可观察能力”，不是“外部通知通道已打通”

## 5.9 第九步：用 Cron 做自动 approval scan

如果希望系统定时扫描审批超时，需要：

1. 设置 `BELLDANDY_CRON_ENABLED=true`
2. 保持 Gateway / 主进程在运行
3. 创建 cron 任务

### 推荐方式：用 `cron` 工具创建

工具名：

- `cron`

#### 示例 1：每 30 分钟扫描单个 goal

```json
{
  "action": "add",
  "name": "approval scan: phase6",
  "payloadKind": "goalApprovalScan",
  "goalId": "goal_phase6_review_governance",
  "autoEscalate": true,
  "scheduleKind": "every",
  "everyMs": 1800000
}
```

#### 示例 2：每天早上 9 点执行一次

```json
{
  "action": "add",
  "name": "daily approval scan",
  "payloadKind": "goalApprovalScan",
  "goalId": "goal_phase6_review_governance",
  "autoEscalate": true,
  "scheduleKind": "at",
  "at": "2026-03-22T09:00:00+08:00"
}
```

#### 示例 3：扫描全部 goal

```json
{
  "action": "add",
  "name": "all goals approval scan",
  "payloadKind": "goalApprovalScan",
  "allGoals": true,
  "autoEscalate": true,
  "scheduleKind": "every",
  "everyMs": 3600000
}
```

### Cron 相关文件

- `~/.star_sanctuary/cron-jobs.json`

说明：

- 这个文件由系统自动维护
- 不建议手工编辑

## 5.10 第十步：前端里怎么用

当前遵守 `16.7` 的约束：

- 不新增长期任务独立 Dashboard
- 不新增长期任务专用前端 store
- 所有可视入口尽量复用现有 `goal detail / modal`

你在 WebChat 里可以使用：

1. **Goals 列表**
   - 创建 goal
   - 恢复 / 暂停
2. **Goal Detail**
   - 查看概要
   - 查看 task graph / checkpoints / handoff
   - 查看 review governance / unified approval panel
3. **Approval Panel**
   - 执行 approval scan
   - suggestion review 决策
   - checkpoint 决策
   - 查看 recent notifications
   - 查看 dispatch count / recent dispatches
   - 打开 `review-notifications.json`
   - 打开 `review-notification-dispatches.json`

## 6. 推荐的日常使用顺序

建议按下面顺序使用长期任务系统：

1. 配好 `.env.local`
2. 创建 goal
3. 拆 task graph
4. 对核心节点执行 `goal_orchestrate`
5. 做 checkpoint 审批
6. 节点完成后生成 retrospective / method / skill / flow patterns
7. 对 suggestion 进入 review workflow
8. accepted 后发布到 methods / skills
9. 定期执行 `goal.review_governance.summary`
10. 对需要持续治理的项目开启 `cron goalApprovalScan`

## 7. 常见误区

### 7.1 `review-notifications.json` 是不是要我手工写？

不是。  
这是系统自动生成的运行态文件。

### 7.2 `review-notification-dispatches.json` 是不是已经发出 IM / webhook 了？

不是。  
当前只是运行态 outbox。

### 7.3 不配置 `review-governance.json` 能不能用？

能用。  
系统会走默认配置，但你就没有组织级 reviewer / template / SLA / notification policy。

### 7.4 不开 `BELLDANDY_CRON_ENABLED` 能不能用 approval scan？

能。  
你仍然可以手工执行：

- `goal.approval.scan`
- `goal.suggestion_review.scan`

只是不会自动定时执行。

### 7.5 `cron-jobs.json` 要不要手工维护？

不要。  
请优先通过 `cron` 工具管理。
