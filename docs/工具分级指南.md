# 工具分级指南

## 目标

本指南用于给 Belldandy 当前可见工具建立一套稳定的分级体系，主要服务于以下目标：

1. 控制默认 Agent 暴露给模型的工具数量，降低 `tools schema` 带来的 input token 消耗。
2. 把“日常聊天”和“专项执行”拆开，避免所有工具长期对默认 Agent 可见。
3. 为后续的 `toolWhitelist`、运行时工具开关、Agent Profile 设计提供统一口径。

本指南不是最终开关配置文件，而是分级规则与编排模板。

---

## 核心原则

### 1. 分级看的不是“工具是否强大”，而是“是否值得默认暴露”

一个工具再强，如果不是高频、刚需、低副作用，就不应该进低级别。

### 2. 默认 Agent 只看低级别

建议默认聊天 Agent 仅暴露：

- `P0`
- `P1`
- 必要时少量 `P2`

其余工具改由专项 Agent 或显式开关处理。

### 3. 避免双栈重复暴露

以下能力默认不建议双栈并存：

- 文件：Builtin 文件工具 vs `mcp_filesystem_*`
- 浏览器：Builtin `browser_*` vs `mcp_chrome_devtools_*`
- 搜索：`web_fetch` vs `mcp_MiniMax_web_search`

同一能力若存在多套实现，默认只保留一套进入低级别。

### 4. Goal 专用工具单独处理

`goal_*`、`task_graph_*` 这类只在 Goal 通道有意义的工具，不建议混入普通会话的 P0-P4 分级口径。建议单独标记为 `P_goal`。

---

## 分级定义

### P0：生存与控制层

定义：

- 默认 Agent 在绝大多数场景都必须能用
- 数量必须极少
- 一旦缺失会显著影响自我管理、基础定位或最小操作能力

判定标准：

- 是否属于最小闭环必需能力
- 是否应该尽量始终保留
- 是否会被频繁用到
- 是否能帮助 Agent 自查、自救或继续工作

典型工具类型：

- 工具管理
- 基础读取
- 最小身份确认
- 基础目录浏览

### P1：核心工作层

定义：

- 日常高频使用
- 对开发、排查、整理信息有明显价值
- 适合长期对默认 Agent 暴露

判定标准：

- 是否属于高频刚需
- 是否能明显提升完成任务效率
- 是否对 token 成本仍然可接受

典型工具类型：

- 代码修改
- 命令执行
- 日志排查
- 方法与记忆检索

### P2：扩展执行层

定义：

- 有用，但不是每轮对话都该默认暴露
- 偶尔需要，适合按需开启或交给专用 Agent

判定标准：

- 是否属于中频能力
- 是否有一定复杂性或副作用
- 是否只在某些任务类型下才有意义

典型工具类型：

- 删除类动作
- 写记忆
- 技能查询
- 更深入的任务/经验操作

### P3：专项能力层

定义：

- 专门领域工具
- 通常只有明确进入某个工作模式时才需要

判定标准：

- 是否面向某个专项场景
- 是否 schema 较大
- 是否不适合长期暴露给默认 Agent

典型工具类型：

- 浏览器自动化
- Canvas
- 子代理
- 调度器
- 经验候选处理

### P4：场景/外部系统层

定义：

- 强依赖外部环境、外部服务、社区环境或特定配置
- 不适合默认暴露给通用聊天 Agent

判定标准：

- 是否依赖外部服务或远程环境
- 是否只在少数特定场景使用
- 是否不属于默认聊天闭环

典型工具类型：

- Office
- 社区房间
- 外部 MCP 专业服务
- 多媒体生成

### P_goal：Goal 专用层

定义：

- 仅在 Goal / 长期任务通道中成立
- 不建议纳入普通聊天 Agent 的默认工具面

典型工具类型：

- `goal_*`
- `task_graph_*`

---

## 判定矩阵

给每个工具打分时，可用下面 5 个问题做快速判断：

| 问题 | 是 | 否 |
|------|----|----|
| 这是默认 Agent 的高频刚需能力吗？ | 往 P0/P1 靠 | 往 P2-P4 靠 |
| 没有它会明显影响最小工作闭环吗？ | 往 P0 靠 | 往 P1-P4 靠 |
| 这个工具只在专项场景才用吗？ | 往 P3/P4 靠 | 往 P0-P2 靠 |
| 这个工具依赖外部环境或额外配置吗？ | 往 P4 靠 | 往 P0-P3 靠 |
| 这个工具的 schema 或能力面是否明显偏大？ | 降一级，避免默认暴露 | 可保留原级别 |

---

## 编排建议

### 默认聊天 Agent

建议只暴露：

- `P0`
- `P1`
- 少量真正高频的 `P2`

### 开发 / Coding Agent

建议暴露：

- `P0`
- `P1`
- 与编码强相关的 `P2`
- 少量浏览器或搜索能力

### 调研 / Research Agent

建议暴露：

- `P0`
- 搜索相关 `P2/P3`
- 极少量记忆与方法工具

### Goal Agent

建议暴露：

- `P0`
- `P1`
- `P_goal`

## 分级模板

### 模板一：按工具逐项登记

| 工具名 | 当前分类 | 建议等级 | 是否默认暴露 | 备注 |
|--------|----------|----------|--------------|------|
| 示例：`file_read` | 文件 | P0 | 是 | 基础读取能力 |
| 示例：`run_command` | 系统执行 | P1 | 是 | 开发高频能力 |
| 示例：`browser_open` | 浏览器 | P3 | 否 | 只在专项任务中开启 |

### 模板二：按工具簇登记

| 工具簇 | 代表工具 | 建议等级 | 默认策略 | 备注 |
|--------|----------|----------|----------|------|
| 文件读取/写入 | `file_read` `file_write` | P0/P1 | 默认保留 | 核心工作闭环 |
| MCP filesystem | `mcp_filesystem_*` | P4 | 默认关闭 | 与 Builtin 文件能力重复 |
| Chrome DevTools | `mcp_chrome_devtools_*` | P3 | 默认关闭 | schema 大，专项使用 |

### 模板三：口令映射模板

用于后续让 Agent 理解“关闭所有非 P0/P1 工具”这种说法。

| 等级 | 含义 | 允许的典型口令 |
|------|------|----------------|
| P0 | 最小生存层 | “只保留 P0” |
| P1 | 核心工作层 | “启用 P0+P1” |
| P2 | 扩展执行层 | “临时开启 P2” |
| P3 | 专项能力层 | “切到浏览器/委派模式” |
| P4 | 外部场景层 | “仅在需要时开启” |
| P_goal | Goal 专用层 | “进入 Goal 模式” |

## 当前工具生态的建议初稿

以下为结合当前 Star Sanctuary 工具生态给出的建议初稿，不是最终结论。

### P0 建议

建议只保留极少数工具：

- `tool_settings_control`
- `file_read`
- `list_files`
- `get_user_uuid`

说明：

- `tool_settings_control` 用于自我调节工具面
- `file_read` + `list_files` 组成最小读取闭环
- `get_user_uuid` 适用于身份确认场景

### P1 建议

建议作为默认聊天 / 默认工作 Agent 的核心层：

- `apply_patch`
- `file_write`
- `run_command`
- `memory_search`
- `method_list`
- `method_read`
- `method_search`
- `log_search`
- `task_recent`

说明：

- 这批工具足以覆盖大多数开发、排查、整理任务
- 对 token 有成本，但远低于全量工具集

### P2 建议

- `file_delete`
- `log_read`
- `memory_read`
- `memory_write`
- `skills_list`
- `skills_search`
- `skill_get`
- `task_search`
- `task_get`
- `timer`
- `token_counter_start`
- `token_counter_stop`
- `experience_usage_get`
- `experience_usage_list`
- `experience_usage_record`
- `experience_usage_revoke`

说明：

- 这些工具有价值，但不是每轮聊天都值得暴露

### P3 建议

- `browser_*`
- `mcp_chrome_devtools_*`
- `canvas_*`
- `sessions_spawn`
- `sessions_history`
- `delegate_task`
- `delegate_parallel`
- `method_create`
- `task_promote_method`
- `task_promote_skill_draft`
- `experience_candidate_*`
- `cron`
- `service_restart`
- `switch_facet`

说明：

- 这里通常是 schema 大户
- 适合专项 Agent，不适合默认 Agent

### P4 建议

- `mcp_filesystem_*`
- `mcp_context7_*`
- `mcp_prompts_chat_*`
- `mcp_MiniMax_*`
- `office_workshop_*`
- `office_homestead_*`
- `join_room`
- `leave_room`
- `get_room_members`
- `camera_snap`
- `image_generate`
- `text_to_speech`

说明：

- 外部依赖强
- 场景性明显
- 默认不建议暴露给普通聊天 Agent

### P_goal 建议

- `goal_*`
- `task_graph_*`

说明：

- 这类工具不建议纳入普通分级决策
- 应由 Goal 通道或 Goal Agent 专门负责

## 推荐落地顺序

### 第一步：先确定默认 Agent 的最低可用集合

建议先只讨论：

- P0
- P1

不要一开始就试图把全部工具一次性分完。

### 第二步：先压缩默认暴露面，再细化专项层

更合理的顺序是：

1. 先收缩默认 Agent
2. 再定义专项 Agent
3. 最后再考虑运行时分级口令

### 第三步：避免过度细分

如果某一层级内部差异太大，优先拆成“专项 Agent”，不要无限细化 P5、P6、P7。

---

## 审核清单

在你正式采用某个分级方案前，建议逐项确认：

1. 默认 Agent 是否仍具备最小工作闭环。
2. 是否存在重复暴露的双栈能力。
3. 是否把 Goal 工具误混进普通聊天层。
4. 是否把大量低频外部工具暴露给默认 Agent。
5. 是否能用一句口令表达常见切换需求。

---

## 后续可衔接项

本指南后续可衔接到以下实现方向：

1. `agents.json` 中为 `default` / `coder` / `researcher` 等 Agent 写入不同 `toolWhitelist`
2. 在 `tool_settings_control` 上增加“按分级批量开关”的语义层
3. 在 Web 设置页中增加“按分级切换工具集”的预设

当前阶段建议先完成文档分级，再决定是否把分级固化到代码与配置中。
