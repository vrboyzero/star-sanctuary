# H4 真实观察清单

## 1. 看哪里

1. `bdd doctor`
   看 `Runtime Resilience` 段。
2. WebChat -> `settings -> doctor`
   看 `Runtime Resilience` 卡片。
3. launch explainability / prompt 相关视图
   看是否带出最近一次 `fallback / degrade` 说明。

## 2. 重点看什么

1. 当前主路由是否符合预期
   主要看 `primary provider / model`。
2. 最近一次是否发生降级
   主要看 `latest status`、`degraded`、`headline`。
3. 降级类型是什么
   重点区分：
   - `same-profile retry`
   - `cross-profile fallback`
   - `cooldown skip`
   - `terminal fail`
4. 原因是否集中
   重点看 `reasons` 是否长期集中在：
   - `rate_limit`
   - `timeout`
   - `server_error`
   - `auth`
   - `billing`

## 3. 观察时机

1. 正常日常使用时顺带看
   不需要专门制造异常。
2. 发生“回复变慢、明显失败、疑似切备用模型”时再重点看一次。
3. 连续观察几天即可，不需要长时间挂着盯。

## 4. 什么时候需要继续开发 H4

1. 用户明显感知失败，但 `doctor` 看不到有效 fallback / degrade 证据。
2. `doctor` 长期显示大量 degrade，但真实使用并没有明显问题。
   这说明告警可能过敏。
3. 经常出现同一类问题：
   - 总是 `same-profile retry`
   - 总是 `cross-profile fallback`
   - 总是 `cooldown skip`
4. explainability 给出的原因过粗，已经不足以判断是否该调超时、冷却或 fallback 策略。

## 5. 当前默认决策

1. 只观察，不主动继续开发 `H4-2`。
2. 只有出现稳定误判、漏报或高频真实痛点时，才做最小修补。
