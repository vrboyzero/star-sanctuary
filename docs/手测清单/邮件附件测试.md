# 邮件附件测试手测清单

## 1. 目标

验证 Star Sanctuary 当前邮件链路在“带附件”场景下是否仍能稳定工作，重点覆盖：

- IMAP 收信时的附件识别
- MIME 解析在纯文本 / HTML / 内联图片 / 多附件场景下是否正常
- 邮件线程建链、整理、审计是否仍正确
- `send_email` 发信与线程内回复带附件时，SMTP 是否能真实送达
- 附件场景下是否引入新的超时、重复入站或线程串线问题

本清单默认你已经完成：

- QQ 邮箱 `SMTP + IMAP` 已接通
- `Email Inbound Runtime` 为 `running`
- 新邮件已能稳定进入系统
- `send_email` 已能真实发出邮件

---

## 2. 观测入口

每轮测试都建议同时看下面几处：

- WebChat `系统检查 -> Email Inbound Runtime`
- WebChat `系统检查 -> Email Outbound Runtime`
- WebChat `记忆查看 -> 消息审计`
- WebChat `记忆查看 -> 消息审计 -> 邮件线程整理`
- 邮件线程会话顶部提示
- Gateway 日志里的：
  - `[email-inbound]`
  - `[email-outbound]`
  - `[email-followup]`
  - `IMAP inbound poll failed`

---

## 3. 前置建议

### 3.1 建议准备的测试附件

建议先准备 4 类小文件，避免测试时临时找素材：

- `test-note.txt`
- `test-image.png`
- `test-doc.pdf`
- `test-sheet.xlsx` 或任意小型二进制文件

建议每个文件都尽量小，先控制在 `1 MB` 以内，避免把“附件测试”变成“大文件稳定性测试”。

### 3.2 建议测试邮箱

建议至少准备两个邮箱：

- `4918822@qq.com`
- 另一个外部邮箱（如 Gmail / Outlook / 其他常用邮箱）

这样能区分：

- “只是在本邮箱里回环”
- 还是“真的经过外部收发链路”

---

## 4. 手测用例

## 4.1 收信：纯文本 + 单附件

### 步骤

1. 用另一个邮箱向 `4918822@qq.com` 发一封新邮件。
2. 邮件主题建议写：`附件测试 A1 - 单附件`
3. 正文使用纯文本，不要 HTML。
4. 附上 `test-note.txt`。
5. 等待一个轮询周期后查看系统。

### 预期结果

- 系统能正常收进来，不报 IMAP 解析错误
- `消息审计` 中能看到这条 `email_inbound`
- 审计里附件数量应为 `1`
- `邮件线程整理` 能看到对应线程
- 打开线程会话后，Agent 仍能正常给出处理建议

### 重点记录

- 是否成功建线程
- 附件数量是否正确
- 日志是否出现 `IMAP read timeout` 或 MIME 解析报错

---

## 4.2 收信：HTML + 单附件

### 步骤

1. 用另一个邮箱向 `4918822@qq.com` 发一封新邮件。
2. 主题建议：`附件测试 A2 - HTML 邮件`
3. 正文尽量使用富文本 / HTML 格式。
4. 附上 `test-doc.pdf`。

### 预期结果

- 系统仍能正常收信
- triage / 回复建议不应明显异常
- `消息审计` 中附件数量应为 `1`
- 线程会话中的邮件信息应仍然可读，不应只剩大量 HTML 噪音

### 重点记录

- HTML 正文摘要是否还能看懂
- 是否出现 MIME 边界解析异常

---

## 4.3 收信：内联图片

### 步骤

1. 用另一个邮箱发送一封带内联图片的邮件。
2. 主题建议：`附件测试 A3 - 内联图片`
3. 正文中插入一张图片，而不是只做普通附件上传。

### 预期结果

- 系统能正常收进来
- 不会因为内联资源导致整封邮件收信失败
- 审计中的附件 / 内联附件统计应合理
- 线程会话仍可正常建立

### 重点记录

- 是否能区分普通附件与内联附件
- 是否因为图片资源导致正文异常膨胀或乱码

---

## 4.4 收信：多附件

### 步骤

1. 用另一个邮箱发送一封带 `2-3` 个附件的邮件。
2. 主题建议：`附件测试 A4 - 多附件`
3. 可附：
   - `test-note.txt`
   - `test-image.png`
   - `test-doc.pdf`

### 预期结果

- 系统能正常收进来
- 审计中的附件数量与实际一致
- 不会因此异常重复建线程
- 不会明显增加 IMAP 超时或导致本轮收信卡死

### 重点记录

- 附件数量
- 线程是否唯一
- 日志是否出现长时间卡顿或超时

---

## 4.5 发信：新邮件 + 单附件

### 步骤

1. 在 WebChat 正常会话中让 Agent 发一封带附件的新邮件。
2. 建议提示词：

```text
帮我发一封测试邮件给 <你的另一个邮箱>，
主题：附件测试 B1 - 发信单附件
正文：这是一封带单附件的发信测试邮件。
并附上一个测试文件。
```

3. 如果当前能力还不能让 Agent 自行选择本地文件，可改为你手动在交互中补文件路径或改为后续代码链路测试。
4. 确认审批窗口。

### 预期结果

- 会弹出发信确认窗口
- 确认窗口能显示附件相关信息
- SMTP 真实发出后，对端邮箱能收到附件
- `消息审计` 中能看到 `email_outbound`

### 重点记录

- 确认窗是否显示附件数量
- 对端是否真的收到附件，而不是只收到正文

---

## 4.6 发信：在线程内回复并带附件

### 步骤

1. 先打开一条已有邮件线程会话。
2. 在该线程里让 Agent 回复，并附上一个附件。
3. 建议提示词：

```text
请回复这封邮件，
告诉对方我已收到，并附上测试附件。
```

### 预期结果

- 不再报“需要本地 WebChat 会话”之类的旧错误
- 仍会走邮件审批窗口
- 确认窗中应保留当前线程的 `threadId / replyToMessageId`
- 对端邮箱应收到一封带附件的线程回复

### 重点记录

- 是否继续沿用原邮件线程
- 是否正确保留 `threadId / replyToMessageId`
- reminder 是否会在回复后自动关闭

---

## 4.7 稳定性：附件场景下的重启恢复

### 步骤

1. 先完成一次“带附件收信”或“带附件发信”。
2. 重启 Gateway。
3. 再发一封新的带附件邮件到 `4918822@qq.com`。

### 预期结果

- 重启后系统仍能正常收新邮件
- 不会重新回扫大量旧邮件 backlog
- 附件场景不会导致线程绑定丢失

### 重点记录

- 重启后是否仍只关注新邮件
- 是否出现附件相关回归

---

## 5. 推荐执行顺序

如果不想一次测太多，建议按下面顺序：

1. `4.1` 收信：纯文本 + 单附件
2. `4.2` 收信：HTML + 单附件
3. `4.4` 收信：多附件
4. `4.6` 在线程内回复并带附件
5. `4.7` 重启恢复

原因：

- 先确认收信附件基础链路
- 再确认 MIME 边界
- 再确认线程回复附件
- 最后确认重启后没有回归

## 7. 当前已知边界

当前这份清单主要验证的是：

- 附件元信息
- MIME / 线程 / 回复 / 审计 / 提醒闭环

它还不等于已经完整验证：

- 超大附件稳定性
- 各种编码附件的跨邮箱兼容性
- 附件内容级理解或自动提取
- 恶意附件安全策略

这些应在后续独立作为“附件增强”或“安全治理”测试，不建议混在当前基础联调里。
