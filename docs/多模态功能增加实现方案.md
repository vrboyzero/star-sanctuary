# 图片识别 / 视频识别 Agent 工具落地说明

## 需求理解

你的目标不是单纯把 `c:\Users\admin\.star_sanctuary\skills\screen-capture\scripts\media-analyze-openai.py` 平移进仓库，而是把它升级成 **Star Sanctuary 原生 Agent 能力**，分成两部分：

1. `图片识别工具`
   - Agent 可以主动调用。
   - 用户既可以让 Agent 分析当前消息上传的图片附件，也可以指定某个目录下的图片文件名让 Agent 分析。
   - 工具使用独立的 `api_key / base_url / model` 配置，不和主聊天模型强绑定。

2. `视频识别工具`
   - Agent 可以主动调用。
   - 用户既可以让 Agent 分析当前消息上传的视频附件，也可以指定某个目录下的视频文件名让 Agent 分析。
   - 同样支持独立的 `api_key / base_url / model` 配置。

3. `自动化触发`
   - 当 WebChat 消息里带有图片或视频附件时，希望系统尽量自动完成识别，不需要用户额外提醒“请看附件”“附件在某目录”等。
   - 目标是减少 Agent 因附件路径定位失败、忘记主动看附件、或者不知道该调用什么工具而失误。

4. `保留显式工具调用`
   - 除了自动触发，还需要保留工具本身，方便用户说“去识别 `xxx.png` / `xxx.mp4`”时，Agent 明确调用。

## 已确认的项目现状

基于仓库当前代码，相关链路已经存在一部分，不应重复实现：

1. WebChat 已支持上传附件，并有大小限制配置。
   - 入口：`apps/web/public/app.js`
   - 附件处理：`apps/web/public/app/features/attachments.js`
   - 相关环境变量：`BELLDANDY_ATTACHMENT_MAX_FILE_BYTES`、`BELLDANDY_ATTACHMENT_MAX_TOTAL_BYTES`

2. `message.send` 已有附件预处理链路。
   - 主入口：`packages/belldandy-core/src/query-runtime-message-send.ts`
   - 附件理解：`packages/belldandy-core/src/attachment-understanding-runner.ts`

3. 当前系统对图片/视频附件的处理方式是：
   - 图片：
     - 如果当前模型声明了 `image_input`，会直接注入多模态内容。
     - 否则只把图片保存到 `stateDir/storage/attachments/`，并把路径写进 prompt。
   - 视频：
     - 如果当前模型声明了 `video_input`，会走视频多模态注入，并在 OpenAI/Moonshot 兼容链路里做上传转换。
     - 否则同样只写入路径提示。

4. 当前问题不在“附件无法上传”，而在“当主模型没有稳定视觉能力时，系统没有自动补一个独立视觉理解层”。
   - 这正是这次要补的部分。

5. 仓库里已经有多媒体工具接入模式。
   - 工具出口：`packages/belldandy-skills/src/builtin/multimedia/`
   - 注册入口：`packages/belldandy-core/src/bin/gateway.ts`
   - 现有独立配置范式可参考：
     - 图片生成：`BELLDANDY_IMAGE_*`
     - TTS：`BELLDANDY_TTS_*`

## 方案判断

### 为什么不能只做“普通工具”

如果只新增两个工具，而不接入 `message.send` 的附件预处理链路，那么用户上传图片/视频后，Agent 仍然要先意识到“我该调用哪个工具”，再自己定位附件路径。这和你要解决的问题不一致。

所以这次实现应拆成两层：

1. **自动附件理解层**
   - 当 `message.send` 收到图片/视频附件时，在进入 Agent 主推理前，自动调用独立视觉识别能力。
   - 识别结果以文本摘要的形式注入 prompt / `promptDeltas` / `attachmentStats` 附近的上下文。
   - 这样即使主模型本身没有视觉能力，Agent 也能直接“看到”附件内容摘要。

2. **显式 Agent 工具层**
   - 新增 `image_understand` / `video_understand` 两个工具。
   - 用于 Agent 主动分析本地文件、重新分析附件、或者按用户指定路径分析媒体文件。

这两个层次应复用同一套底层分析 runtime，避免一套逻辑走自动注入，一套逻辑走工具调用而分叉。

## 总体设计

### 目标行为

#### A. 用户上传图片附件

- 系统在 `message.send` 阶段保存附件后，检测到 `image/*`。
- 若启用了图片理解能力，则自动调用图片识别 runtime。
- 把识别结果以简洁文本注入 Agent 输入，例如：
  - 图片摘要
  - 关键标签
  - OCR/界面错误文本（若模型返回）
  - 原附件文件名与保存路径
- Agent 无需再自己找路径或主动“看图”。

#### B. 用户上传视频附件

- 系统在 `message.send` 阶段检测到 `video/*`。
- 若启用了视频理解能力，则自动调用视频识别 runtime。
- 将结果注入 Agent 输入。
- 若识别失败或未配置，则降级为现有路径提示，不阻断消息发送。

#### C. 用户显式要求分析文件

示例：

- “帮我识别 `screenshots/error.png` 里写了什么”
- “看看 `demo.mp4` 这个视频讲了什么”

此时 Agent 调用：

- `image_understand`
- `video_understand`

工具负责：

- 解析相对路径 / 绝对路径
- 校验路径必须位于 `workspaceRoot`、`extraWorkspaceRoots`、或 `stateDir/storage/attachments`
- 调用统一的视觉分析 runtime
- 返回结构化结果 + 简洁文本结果

## 当前实现结构

### 1. 底层 runtime

当前核心目录：

- `packages/belldandy-skills/src/builtin/multimedia/understand-shared.ts`
- `packages/belldandy-skills/src/builtin/multimedia/image-understand.ts`
- `packages/belldandy-skills/src/builtin/multimedia/video-understand.ts`

职责划分如下：

1. `understand-shared.ts`
   - 读取和规范化配置
   - OpenAI-compatible client 初始化
   - 超时控制
   - 统一返回结构
   - 路径/文件校验公共逻辑

2. `image-understand.ts`
   - 图片压缩与编码
   - 调用 `chat.completions` 或 `responses`
   - 输出结构化结果：
     - `summary`
     - `tags`
     - `ocrText`
     - `content`
     - `source`

3. `video-understand.ts`
   - 处理视频输入
   - 第一版优先复用当前项目对视频多模态的既有能力判断
   - 若目标服务支持视频文件直传/引用，则走原生视频输入
   - 若不支持，第一版可限制为“仅支持兼容视频输入的 OpenAI-compatible 服务”，不额外做复杂抽帧

### 2. Agent 工具层（已落地）

在 `packages/belldandy-skills/src/builtin/multimedia/` 下新增两个工具：

- `image_understand`
- `video_understand`

当前参数：

#### `image_understand`

- `file_path`: 必填，图片文件路径
- `prompt`: 选填，自定义识别提示词
- `focus_mode`: 选填，`overview | detail_query`
- `focus_target`: 选填，指定要重点看的对象 / 区域 / 文字 / UI 元素
- `include_key_regions`: 选填，是否返回图片重点区域
- `max_key_regions`: 选填，最多返回多少个重点区域

#### `video_understand`

- `file_path`: 必填，视频文件路径
- `prompt`: 选填，自定义识别提示词
- `focus_mode`: 选填，`overview | timeline | timestamp_query`
- `target_timestamp`: 选填，指定时间点，例如 `00:17` / `01:02:30`
- `include_timeline`: 选填，是否返回关键时间线
- `max_timeline_items`: 选填，最多返回多少个关键片段

当前返回结构：

- `success`
- `summary`
- `content`
- `tags`
- `keyRegions`
- `targetDetail`
- `timeline`
- `targetMoment`
- `source.path`
- `source.mime`
- `model`

补充说明：

- 图片工具现在也应支持“总览 + 局部关注”两种模式：
  - `focus_mode=overview`：回答整张图片主要内容，并返回 `keyRegions`
  - `focus_mode=detail_query`：重点回答 `focus_target` 对应的对象、区域、文字或细节，并返回 `targetDetail`
- 自动图片附件识别默认走 `overview + keyRegions`，方便 Agent 先知道“整张图是什么”和“哪几个区域最值得关注”。
- 自动视频附件识别默认走 `overview + timeline`，方便 Agent 先知道“整个视频讲了什么”和“几个关键时间点发生了什么”。
- 当用户追问“某个时间点在做什么”时，Agent 应优先主动调用 `video_understand`，并带上：
  - `focus_mode=timestamp_query`
  - `target_timestamp=<用户关心的时间点>`
- 这条规则不应只停留在工具文档里，还应进入 Agent 的运行时工具使用提示，使其在用户说“00:17 在干什么”“1 分 23 秒画面是什么”时更稳定主动调用 `video_understand`。
- `timeline` 用于回答“视频大概流程”；`targetMoment` 用于回答“这个具体时间点附近发生了什么”。
- `keyRegions` 用于回答“图片里哪些局部最重要”；`targetDetail` 用于回答“这个指定局部 / 元素具体是什么”。

当前工具 contract：

- family: `other`
- channels: `gateway`, `web`
- riskLevel: `medium`
- outputPersistencePolicy: `artifact`

当前做法：

2. 在 `handleImageAttachment` / `handleVideoAttachment` 之前或之内，引入“自动识别分支”：

   - 路径与文件名
   - 必要时保留“识别失败/未配置”的状态说明

4. 同时写入 `promptDeltas`

### 5. 识别缓存（已落地）

当前已复用附件 fingerprint 思路，为图片/视频理解提供缓存：

- 音频转写缓存：
  - `stateDir/storage/attachment-understanding-cache/audio-transcription/<fingerprint>.json`
- 图片缓存：
  - `stateDir/storage/attachment-understanding-cache/image-understanding/<fingerprint>.json`
- 视频缓存：
  - `stateDir/storage/attachment-understanding-cache/video-understanding/<fingerprint>.json`

缓存收益：

- 用户重复发送同一附件时避免重复消耗
- 会话内二次调用工具时可直接复用
- 缓存是落盘持久化的，不是只存在内存中；因此如果不主动清理，重启 Gateway 后旧缓存仍会继续命中

## 已落地实现说明

目标
为 Star Sanctuary 增加原生的图片识别工具与视频识别工具，并把图片/视频附件识别能力接入 `message.send` 自动预处理链路，减少 Agent 因路径定位和是否该“看附件”判断失误带来的问题。

当前约束
- 必须基于现有附件链路实现，不能和当前 `preparePromptWithAttachments` 机制打架。
- 需要支持独立 `.env` 配置，避免和主聊天模型配置混淆。
- 视频能力依赖目标 OpenAI-compatible 服务实际是否支持视频输入；第一版不应承诺所有兼容服务都可用。
- 自动识别属于额外模型调用，必须允许通过环境变量关闭。
- 需要保证失败可降级，不能因为视觉识别失败阻断正常发消息。

已完成实现
1. 已抽取共享视觉理解 runtime，封装配置读取、OpenAI-compatible 客户端、超时和统一返回结构。
2. 已实现 `image_understand` 工具，支持本地文件路径识别、结构化结果返回，以及“总览 + 局部关注”模式。
3. 已实现 `video_understand` 工具，支持总览、时间线和指定时间点查询。
4. 已将两个工具注册到 `multimedia` 组，并补充导出与 contract。
5. 已在 `attachment-understanding-runner.ts` 中接入自动图片/视频识别，将结果注入 `promptText` 与 `promptDeltas`。
6. 已为图片/视频自动识别增加基于 fingerprint 的缓存与失败降级逻辑。
7. 已补充配置通道白名单与 WebChat 设置项，使独立 `api_key / base_url / model` 可在设置弹窗“工具”页中维护，并带精简中英文说明。
8. 已补充单测，覆盖工具调用、自动注入、缓存命中、未配置、失败降级和路径越界拦截。

验证方式
- Unit:
  - `image_understand` 在配置缺失时正确报错
  - `image_understand` 能返回结构化识别结果
  - `video_understand` 在服务不支持或配置缺失时正确降级/报错
  - 路径校验能拦截工作区外非法路径
  - 相同 fingerprint 可命中缓存
- Integration:
  - `message.send` 上传图片附件时，Agent 输入中出现自动识别摘要
  - `message.send` 上传视频附件时，Agent 输入中出现自动识别摘要或明确降级说明
  - 自动识别失败时，消息仍能正常发送
- Manual:
  - WebChat 上传图片后，不额外提醒“看附件”，Agent 仍能直接描述图片
  - WebChat 上传视频后，不额外提醒“看附件”，Agent 能给出视频理解结果或清晰失败原因
  - 用户让 Agent 分析指定目录文件时，工具调用成功
  - 打开 WebChat 设置弹窗的“工具”页，可以看到新增配置项，且中英文说明简洁清晰
  - 保存后重新打开设置，敏感字段保持脱敏显示，非敏感字段能正确回填

## 关键取舍

### 取舍 1：自动识别放在工具层还是消息预处理层

结论：**放在消息预处理层，工具层作为补充。**

原因：

- 你的核心诉求是“用户上传后尽量自动”，不是“多一个工具名字”。
- 只有 `message.send` 预处理层能在 Agent 首轮推理前稳定插入附件识别结果。

### 取舍 2：视频第一版是否做抽帧兜底

结论：**第一版先不做复杂抽帧兜底。**

原因：

- 当前项目已经有视频多模态注入链路。
- 抽帧意味着引入 ffmpeg/采样策略/时间线摘要/OCR 合并，复杂度明显上升。
- 第一版先把“独立视频识别模型 + 自动注入 + 显式工具”打通更稳。

后续若视频兼容性不足，再拆分 `split_task` 做“抽帧兜底版”。

### 取舍 3：是否和主 OpenAI 配置自动回退

结论：**第一版不建议默认回退。**

原因：

- 当前仓库对 `BELLDANDY_IMAGE_OPENAI_API_KEY` 已经采用独立强约束。
- 视觉识别是独立成本面，单独配置更符合运维预期。

## 风险与未决点

1. `视频识别` 所使用的 OpenAI-compatible 服务，其视频输入协议未必统一。
   - 这部分需要在实现前结合你准备使用的具体服务做一次最小验证。

2. 如果启用了“主模型多模态注入 + 独立自动识别注入”，上下文里会同时出现：
   - 原始图片/视频内容
   - 视觉摘要文本
   需要控制提示词长度，避免重复信息过多。

3. 如果图片很多、视频很大，自动识别会增加响应耗时和费用。
   - 因此必须有开关和超时。

## 当前结论

这个需求是合理的，而且和仓库现状契合度很高。最稳妥的实现方式不是“只加两个工具”，而是：

**共享视觉识别 runtime + 两个显式工具 + `message.send` 自动附件识别接线 + 独立配置 + 缓存 + 失败降级。**

这样既解决“上传附件后 Agent 不会主动看”的问题，也保留“让 Agent 去识别某个指定文件”的工具能力。

## 摄像头截图自动图片识别接线说明（已落地）

### 当前代码现状

1. `camera_snap` 现在是独立工具调用链路，不经过 `message.send` 附件预处理。
   - 工具入口：
     - `packages/belldandy-skills/src/builtin/multimedia/camera.ts`

2. `camera_snap` 成功后会直接返回截图路径。
   - 当前返回字段主要是：
     - `provider`
     - `path`
     - `mirrorUrl`
     - `state`

3. 浏览器 loopback provider 会把截图直接保存到工作区：
   - `workspaceRoot/screenshots/*.png`
   - 代码位置：
     - `packages/belldandy-skills/src/builtin/multimedia/camera-runtime.ts`

4. native desktop provider 也会返回已落盘图片路径。
   - 代码位置：
     - `packages/belldandy-skills/src/builtin/multimedia/camera-native-desktop-provider.ts`

5. 因为这条链路是“工具执行结果”而不是“用户上传附件”，所以不能直接依赖 `attachment-understanding-runner.ts` 自动生效。

### 已实现目标

当 Agent 调用 `camera_snap` 截图成功后，不需要它再额外主动调用 `image_understand`，而是让 `camera_snap` 的返回结果里直接带上这张截图的自动识别结果，降低 Agent 在“先截图、再识别截图内容”两步流程中的遗漏和失败率。

### 实施原则

1. 不改 `message.send` 附件链路。
   - 因为摄像头截图不是用户上传附件，而是工具产物。

2. 优先在 `camera_snap` 工具层完成自动接线。
   - 这是最小改动路径。
   - Agent 在同一次工具结果里就能拿到截图路径和识别摘要。

3. 复用现有图片识别模型配置。
   - 摄像头截图本质仍是图片。
   - 不需要新增一套独立的 `api_key / base_url / model`。
   - 直接复用：
     - `BELLDANDY_IMAGE_UNDERSTAND_*`

4. 识别失败不能让 `camera_snap` 整体失败。
   - 截图成功应仍返回成功。
   - 图片识别失败只作为附加状态写回结果。

### 当前接线方式

#### 结论

“摄像头截图后自动图片识别”已经实现为 `camera_snap` 的成功后置步骤，而不是新增一个额外的 Agent 提示规则。

#### 当前做法

1. 在 `packages/belldandy-skills/src/builtin/multimedia/camera.ts` 中扩展 `camera_snap`。
   - 当前流程已经变为：
     - 调 `captureCameraSnapshot(...)`
     - 截图成功后，若图片识别已启用，则立即对 `result.path` 调用现有图片识别 runtime
     - 再把识别结果合并进 `camera_snap` 的输出 JSON

2. 复用现有图片识别 runtime。
   - 直接使用：
     - `readImageUnderstandConfig()`
     - `understandImageFile()`

## 屏幕截图工具 + 浏览器截图自动识别实施方案（2026-04-30）

### 当前状态

1. 已有 `browser_screenshot`
   - 位置：
     - `packages/belldandy-skills/src/builtin/browser/tools.ts`
   - 当前只能截图活动浏览器页面。
   - 当前只返回截图保存路径文本，尚未自动串联图片识别。

2. 已有 `camera_snap`
   - 位置：
     - `packages/belldandy-skills/src/builtin/multimedia/camera.ts`
   - 已经支持“截图成功后自动图片识别”。

3. 已有共享图片识别缓存层
   - 位置：
     - `packages/belldandy-skills/src/builtin/multimedia/understanding-cache.ts`
   - `camera_snap` 与附件图片理解链路已能共用该缓存。

4. 当前仍没有真正的系统级屏幕截图 Agent 工具
   - 还缺：
     - 整个桌面截图
     - 指定显示器截图
     - 指定窗口截图

### 方案取舍结论

#### 对比过的方案

通过 Context7 对比过几类常见方案：

1. `screenshot-desktop`
   - 优点：
     - 接入简单
     - 适合整屏 / 指定显示器截图
   - 缺点：
     - 不适合做“任意窗口截图”
     - 不能很好复用现有 `native_desktop helper` 体系

2. `node-screenshots` / `xcap`
   - 优点：
     - 支持显示器与窗口级截图
     - 能力比较完整
   - 缺点：
     - 引入新的原生依赖面
     - `xcap` / `node-screenshots` 背后是原生实现，安装、构建、分发与兼容性成本更高
     - 与当前仓库已经存在的 PowerShell + ffmpeg + helper 架构有重复

#### 结论

当前项目最稳的落地方向，不是再引一个新的原生截图库，而是：

1. 继续扩展现有 `native_desktop helper`
2. 让它同时承担：
   - 摄像头快照
   - 系统级屏幕截图
3. 再把 `browser_screenshot` 与 `screen_capture` 统一接到现有图片自动识别链路

这样做的原因：

1. 复用现有 helper 启动、诊断、超时、错误映射、Windows 兼容层
2. 复用现有 `screenshots/` 工件目录约定
3. 复用现有图片识别配置与缓存
4. 避免再增加新的原生构建与分发风险

### 总体目标

本轮目标分两块：

1. 新增真正的 `screen_capture` 工具
   - 支持：
     - 桌面截图
     - 指定显示器截图
     - 指定窗口截图

2. 把“工具产出的图片”统一接入自动图片识别
   - 覆盖：
     - `browser_screenshot`
     - `screen_capture`
     - 现有 `camera_snap`

最终效果：

1. Agent 截浏览器页后，工具结果里直接带图片识别摘要
2. Agent 截桌面 / 窗口后，工具结果里直接带图片识别摘要
3. Agent 不需要再二次摸索截图文件路径并手动调用 `image_understand`

### 工具设计

#### 1. 升级 `browser_screenshot`

当前 `browser_screenshot` 只返回：

- 截图文件保存路径

计划改为与 `camera_snap` 对齐，返回：

- `path`
- `imageUnderstandingStatus`
  - `completed | disabled | failed`
- `imageUnderstandingPreview`
- `imageUnderstanding`
- `imageUnderstandingError`

1. 浏览器截图成功后，对输出文件路径执行一次共享图片理解 helper
2. 优先命中共享缓存
3. 未命中再调用 `understandImageFile(...)`
4. 图片识别失败不能让 `browser_screenshot` 整体失败

#### 2. 新增 `screen_capture`

新增 Agent 工具：

- `screen_capture`

首版建议参数：

- `target`
  - `desktop | display | window | region`
- `displayId`
  - 当 `target=display` 时使用
- `windowRef`
  - 当 `target=window` 时优先使用
- `windowTitle`
  - 作为兼容 / 回退定位方式
- `region`
  - `{ x, y, width, height }`
- `delayMs`
- `name`
- `format`
  - 首版建议仅允许 `png | jpeg`

  - 桌面 / 区域截图时可选

- `path`
- `captureTarget`
- `display`
- `window`
- `imageUnderstandingStatus`
- `imageUnderstandingPreview`
- `imageUnderstanding`
- `imageUnderstandingError`

为了让“任意窗口截图”更稳定，建议同时补一个只读工具：

- `screen_list_targets`

用途：

1. 列出显示器
2. 列出可截图窗口
3. 返回稳定引用字段，供 `screen_capture.windowRef` 使用

这是为了避免 Agent 只能靠猜标题去截窗口。

若严格控制范围，也可以分两步：

1. 本轮先做 `screen_capture`
2. 下轮补 `screen_list_targets`

但从稳定性考虑，建议同一轮一起做。

1. Agent 调 `browser_screenshot` 后，不需要再额外调用 `image_understand`
2. Agent 调 `screen_capture` 后，不需要再额外调用 `image_understand`
3. 重复截图同一文件内容时，共享缓存可命中

#### Manual

1. 浏览器打开一个含明显文字的页面，执行 `browser_screenshot`
2. 截主屏桌面一次，确认直接返回图片识别摘要
3. 截指定窗口一次，确认直接返回图片识别摘要
4. 继续追问“图里写了什么 / 这个按钮在哪 / 当前窗口是什么内容”，Agent 不需要再自己猜路径

### 风险与控制

1. 窗口截图在不同 ffmpeg 版本上的能力可能不完全一致
   - 控制方式：
     - helper 启动时做能力探测
     - 不支持句柄抓取时走标题或区域回退

2. 某些窗口最小化、无标题、被系统保护时，可能无法稳定截图
   - 控制方式：
     - 工具返回清晰错误原因
     - 提示先把窗口恢复到可见状态

3. 自动图片识别会增加一次模型调用
   - 控制方式：
     - 提供独立布尔开关
     - 命中缓存时不重复识别

   - 当前状态：

       - `/files` 上传接口不可用

       - `ffmpeg` 抽关键帧
       - `image_understand` 逐帧理解
       - 本地拼装总览 / 时间线 / 指定时间点回答
     - 这样即便目标服务不支持原生视频输入，系统仍能给出“足够可用”的视频理解结果

     1. 新增共享 fallback runtime

          - 调用 `ffmpeg/ffprobe` 抽帧

        - `understandVideoFile(...)` 先尝试现有原生视频链路
