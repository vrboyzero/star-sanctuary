# 治理审阅信息状态增强实现

## 1. 需求理解

这次需求不是减少信息，也不是改业务逻辑，而是给 WebChat 中几个“治理 / 审阅 / 追溯”页面的右下详情区增加一层可切换的信息显示级别。

目标可以拆成三件事：

1. 先把当前右下详情区里实际展示的信息做一次盘点。
2. 把这些信息按“重要级 / 增强级”分层。
3. 增加一个由环境变量控制的显示模式：
   - `全开信息状态`：保持现在的完整展示。
   - `简洁信息状态`：只显示重要级信息。

这里的“简洁”不是重新设计一套新页面，也不是把原字段删掉，而是在现有详情面板渲染链路上增加一层显示裁剪规则。这样能保证：

- 默认数据来源、交互按钮、后端接口不变；
- 用户需要时仍可切回全量模式；
- 后续新增治理字段时，也能继续沿用同一套分级机制。

## 2. 影响范围

本次先聚焦你明确提到的这些右下详情区：

1. 记忆查看
   - 任务记录
   - 记忆条目
   - 共享审批
   - 消息审计
2. 经验能力
   - 经验候选页右侧详情
3. 长期任务
   - 长期任务右侧详情

对应前端主模块如下：

## 3. 当前实现现状

### 3.1 页面组织方式

当前这些页面基本都采用相同结构：

1. 顶部统计卡片
2. 左侧列表
3. 右侧详情面板

这次需求只针对第 3 部分，也就是右下详情区，不改左侧列表，不改顶部统计卡片。

### 3.2 右侧详情面板的共同特征

当前右侧详情区有几个共性：

1. 字段非常多，而且很多字段以 `memory-detail-card` 网格方式平铺。
2. 核心判断信息和追踪排查信息是混在一起的。
3. 有些字段是“现在就要判断用”的，有些字段是“出现问题再深挖用”的。
4. 目前没有统一的“显示等级”抽象，都是各模块直接拼 HTML。

所以后续实现上，最合适的不是写一堆 `if (compact)` 到处散落，而是先建立“详情显示级别”这一层统一判断。

## 4. 当前信息盘点与分级建议

下面的分级标准统一按这个原则：

- 重要级：用户做当前判断、继续操作、快速理解状态所必需的信息。
- 增强级：补充解释、审计追踪、来源细节、排障细节、深度上下文。

## 11. 技术设计

### 11.1 总体方案

建议引入统一的详情显示级别：

- `full`
- `compact`

语义定义：

- `full`：保持当前渲染结果，不删信息。
- `compact`：仅渲染重要级区块和字段。

不要按页面分别发明：

- `memoryCompact`
- `goalCompact`
- `experienceCompact`

因为这会让后续维护非常碎。建议统一抽象为“治理审阅详情显示级别”。

### 11.2 配置来源

建议新增环境变量：

`BELLDANDY_WEB_GOVERNANCE_DETAIL_MODE`

可选值：

- `full`
- `compact`

默认值建议：

- `full`

原因：

1. 保持现有行为不变，升级不惊扰现有用户。
2. 用户明确开启后才进入简洁模式。

### 11.3 为什么不能只靠现有 `apps/web/public/config.js`

当前 `index.html` 通过：

```html
<script src="/config.js?v=1.0"></script>
```

直接加载静态文件 `apps/web/public/config.js`。

这意味着：

1. 这个文件本身不会自动读取网关运行时环境变量。
2. 如果只在静态文件里写死开关，用户改的是前端资源，不是 `.env.local` 级配置。
3. 这不符合“通过环境变量切换”的需求。

所以后续实现建议改成：

1. 网关在启动时读取环境变量。
2. 通过新的 HTTP 路由或动态脚本响应，把运行时配置注入前端。
3. 前端统一从 `window.BELLDANDY_WEB_CONFIG` 读取这个值。

### 11.4 推荐接入方式

建议把 `/config.js` 从“纯静态文件”升级为“动态生成 + 可保留静态默认值”的运行时配置脚本。

推荐方案：

1. 保留 `window.BELLDANDY_WEB_CONFIG` 作为前端唯一读取入口。
2. 在网关 HTTP 路由中增加 `/config.js` 动态响应。
3. 动态响应里输出：
   - 现有链接配置
   - 新增 `governanceDetailMode`
4. 前端页面仍然通过 `<script src="/config.js">` 读取，不改页面接入方式。

这样好处是：

1. 前端改动最小。
2. 用户仍然通过环境变量配置。
3. 后续还有别的前端运行时开关，也能继续挂在这一个对象上。

### 11.5 前端读取抽象

建议新增一个统一读取函数，例如：

- `getGovernanceDetailMode()`
- `isCompactGovernanceDetailMode()`

行为约束：

1. 非法值一律回退到 `full`
2. 所有右侧详情页都走这一个入口
3. 不要在每个模块里直接写 `window.BELLDANDY_WEB_CONFIG?.xxx`

### 11.6 前端渲染策略

建议按“区块级裁剪 + 字段级裁剪”两层处理：

1. 区块级裁剪
   - 整个卡片在 compact 模式下不渲染
   - 例如工具调用明细、通知分发队列、metadata
2. 字段级裁剪
   - 同一个 grid 内只保留重要字段
   - 例如只保留开始时间 / 结束时间 / 耗时 / Token，隐藏其余追踪字段

优先顺序建议：

1. 先做区块级裁剪
2. 再做字段级裁剪

原因：

1. 区块级裁剪更稳定，diff 更小
2. 字段级裁剪更适合网格卡片过密的地方

### 11.7 推荐抽象形态

建议增加轻量工具方法，而不是让每个页面各自拼逻辑：

1. `shouldRenderGovernanceEnhancement()`
2. `filterDetailFieldsByImportance(fields, mode)`
3. `renderWhenFull(html)`

如果不想额外抽象太重，也至少要保证：

1. 判断逻辑集中
2. 值枚举集中
3. 页面内不要散落魔法字符串

## 12. 页面落地建议

## 13. 验收标准

实现完成后应满足：

1. 环境变量为 `full` 时，页面展示与当前基本一致。
2. 环境变量为 `compact` 时，右下详情区明显收敛，但不影响主要操作。
3. 左侧列表和顶部统计卡片行为不变。
4. 不新增接口数据依赖，不改变原有后端业务数据结构。
5. 页面切换、列表点击、详情切换不报错。

## 14. 风险与注意点

### 14.1 最大风险

最大风险不是实现开关本身，而是“重要级误判”，导致简洁模式下把用户当前操作必须的信息也隐藏掉。

所以后续实现时要优先确保这些内容绝不被隐藏：

1. 当前状态
2. 当前对象标题
3. 当前对象与来源任务 / 会话 / 长期任务的关系
4. 可执行动作按钮
5. 结果摘要 / 内容摘要

### 14.2 配置注入风险

如果仍使用静态 `public/config.js`，就无法真正做到环境变量驱动。

所以后续实现阶段需要明确：

1. 是否把 `/config.js` 改成动态路由
2. 或者新增一个专门的运行时配置接口，再由前端初始化读取

推荐优先前者，因为现有页面已经依赖这个入口。

### 14.3 兼容风险

长期任务页由多个异步面板拼接而成：

- governance
- tracking
- capability
- progress
- handoff

后续实现简洁模式时，要避免出现：

1. 主详情是 compact
2. 异步子面板仍按 full 渲染

因此显示级别判断应当在这些子面板模块中统一接入。

## 15. 建议的实现顺序

后续如果进入开发阶段，建议顺序如下：

1. 增加环境变量读取与前端运行时配置注入
2. 增加统一详情显示级别工具函数
3. 接入记忆查看
   - 任务记录
   - 记忆条目 / 共享审批
   - 消息审计
4. 接入经验候选页
5. 接入长期任务主详情与治理/追踪面板
6. 补针对性的前端测试

## 16. 最终建议

这次需求的最佳实现方式，不是“删卡片”，而是建立一套统一的“治理审阅详情显示级别”机制，然后把右下详情区按重要级 / 增强级分层。

建议最终采用：

1. 环境变量：`BELLDANDY_WEB_GOVERNANCE_DETAIL_MODE`
2. 模式值：`full` / `compact`
3. 前端运行时配置：继续挂到 `window.BELLDANDY_WEB_CONFIG`
4. 渲染策略：区块级裁剪优先，字段级裁剪补充

这样后续你如果还想把同样的“简洁状态”扩展到子任务、治理面板、观测面板，也可以继续沿用，不需要重做一轮设计。
