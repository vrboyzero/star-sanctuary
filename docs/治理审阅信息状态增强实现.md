# 治理审阅信息状态增强实现

## 1. 需求理解

这次需求不是减少信息，也不是改业务逻辑，而是给 WebChat 中几个“治理 / 审阅 / 追溯”页面的右下详情区增加一层可切换的信息显示级别。

目标可以拆成三件事：

1. 先把当前右下详情区里实际展示的信息做一次盘点。
2. 把这些信息按“重要级 / 增强级”分层。
3. 增加一个由环境变量控制的显示模式：
   - `全开信息状态`：保持现在的完整展示。
   - `简洁信息状态`：只显示重要级信息。

这里的“简洁”不是重新设计一套新页面，也不是把原字段删掉，而是在现有详情面板渲染链路上增加一层显示裁剪规则。这样能保证：

- 默认数据来源、交互按钮、后端接口不变；
- 用户需要时仍可切回全量模式；
- 后续新增治理字段时，也能继续沿用同一套分级机制。

## 2. 影响范围

本次先聚焦你明确提到的这些右下详情区：

1. 记忆查看
   - 任务记录
   - 记忆条目
   - 共享审批
   - 消息审计
2. 经验能力

3. 长期任务
   - 长期任务右侧详情

## 3. 当前实现现状

### 3.1 页面组织方式

当前这些页面基本都采用相同结构：

1. 顶部统计卡片
2. 左侧列表
3. 右侧详情面板

这次需求只针对第 3 部分，也就是右下详情区，不改左侧列表，不改顶部统计卡片。

### 3.2 右侧详情面板的共同特征

当前右侧详情区有几个共性：

1. 字段非常多，而且很多字段以 `memory-detail-card` 网格方式平铺。
2. 核心判断信息和追踪排查信息是混在一起的。
3. 有些字段是“现在就要判断用”的，有些字段是“出现问题再深挖用”的。
4. 目前没有统一的“显示等级”抽象，都是各模块直接拼 HTML。

所以后续实现上，最合适的不是写一堆 `if (compact)` 到处散落，而是先建立“详情显示级别”这一层统一判断。

## 4. 当前信息盘点与分级建议

下面的分级标准统一按这个原则：

- 重要级：用户做当前判断、继续操作、快速理解状态所必需的信息。
- 增强级：补充解释、审计追踪、来源细节、排障细节、深度上下文。

## 11. 技术设计

### 11.1 总体方案

建议引入统一的详情显示级别：

- `full`
- `compact`

语义定义：

- `full`：保持当前渲染结果，不删信息。
- `compact`：仅渲染重要级区块和字段。

不要按页面分别发明：

- `memoryCompact`
- `goalCompact`
- `experienceCompact`

因为这会让后续维护非常碎。建议统一抽象为“治理审阅详情显示级别”。

### 11.2 配置来源

建议新增环境变量：

`BELLDANDY_WEB_GOVERNANCE_DETAIL_MODE`

可选值：

- `full`
- `compact`

默认值建议：

- `compact`

原因：

1. 这次目标本身就是缓解治理审阅页面右下详情区的信息压迫感，默认进入简洁态更符合使用预期。
2. 完整信息仍然保留在 `full` 模式中，用户需要深挖时再切回即可。
3. 即使后续个别页面还需要微调重要级划分，默认 `compact` 也更容易尽早暴露哪些字段确实还需要保留。

### 11.3 为什么不能只靠现有 `apps/web/public/config.js`

当前 `index.html` 通过：

```html
<script src="/config.js?v=1.0"></script>
```

直接加载静态文件 `apps/web/public/config.js`。

这意味着：

1. 这个文件本身不会自动读取网关运行时环境变量。
2. 如果只在静态文件里写死开关，用户改的是前端资源，不是 `.env.local` 级配置。
3. 这不符合“通过环境变量切换”的需求。

所以后续实现建议改成：

1. 网关在启动时读取环境变量。
2. 通过新的 HTTP 路由或动态脚本响应，把运行时配置注入前端。
3. 前端统一从 `window.BELLDANDY_WEB_CONFIG` 读取这个值。

### 11.4 推荐接入方式

建议把 `/config.js` 从“纯静态文件”升级为“动态生成 + 可保留静态默认值”的运行时配置脚本。

推荐方案：

1. 保留 `window.BELLDANDY_WEB_CONFIG` 作为前端唯一读取入口。
2. 在网关 HTTP 路由中增加 `/config.js` 动态响应。
3. 动态响应里输出：
   - 现有链接配置
   - 新增 `governanceDetailMode`
4. 前端页面仍然通过 `<script src="/config.js">` 读取，不改页面接入方式。

这样好处是：

1. 前端改动最小。
2. 用户仍然通过环境变量配置。
3. 后续还有别的前端运行时开关，也能继续挂在这一个对象上。

另外，这个配置不应只停留在 `.env.local` 层。

建议同步接入 WebChat 的：

- 设置窗口
- 系统页

提供一个明确的可视化配置项，直接读写：

- `BELLDANDY_WEB_GOVERNANCE_DETAIL_MODE`

这样用户可以在界面中看到当前模式，也能在需要时切换并保存，而不是每次都手改环境变量文件。

### 11.5 前端读取抽象

建议新增一个统一读取函数，例如：

- `getGovernanceDetailMode()`
- `isCompactGovernanceDetailMode()`

行为约束：

1. 非法值一律回退到 `compact`
2. 所有右侧详情页都走这一个入口
3. 不要在每个模块里直接写 `window.BELLDANDY_WEB_CONFIG?.xxx`

### 11.6 前端渲染策略

建议按“区块级裁剪 + 字段级裁剪”两层处理：

1. 区块级裁剪
   - 整个卡片在 compact 模式下不渲染
   - 例如工具调用明细、通知分发队列、metadata
2. 字段级裁剪
   - 同一个 grid 内只保留重要字段
   - 例如只保留开始时间 / 结束时间 / 耗时 / Token，隐藏其余追踪字段

优先顺序建议：

1. 先做区块级裁剪
2. 再做字段级裁剪

原因：

1. 区块级裁剪更稳定，diff 更小
2. 字段级裁剪更适合网格卡片过密的地方

### 11.7 推荐抽象形态

建议增加轻量工具方法，而不是让每个页面各自拼逻辑：

1. `shouldRenderGovernanceEnhancement()`
2. `filterDetailFieldsByImportance(fields, mode)`
3. `renderWhenFull(html)`

如果不想额外抽象太重，也至少要保证：

1. 判断逻辑集中
2. 值枚举集中
3. 页面内不要散落魔法字符串

## 14. 风险与注意点

### 14.1 最大风险

最大风险不是实现开关本身，而是“重要级误判”，导致简洁模式下把用户当前操作必须的信息也隐藏掉。

所以后续实现时要优先确保这些内容绝不被隐藏：

1. 当前状态
2. 当前对象标题
3. 当前对象与来源任务 / 会话 / 长期任务的关系
4. 可执行动作按钮
5. 结果摘要 / 内容摘要

### 14.2 配置注入风险

如果仍使用静态 `public/config.js`，就无法真正做到环境变量驱动。

所以后续实现阶段需要明确：

1. 是否把 `/config.js` 改成动态路由
2. 或者新增一个专门的运行时配置接口，再由前端初始化读取

推荐优先前者，因为现有页面已经依赖这个入口。

### 14.3 兼容风险

长期任务页由多个异步面板拼接而成：

- governance
- tracking
- capability
- progress
- handoff

后续实现简洁模式时，要避免出现：

1. 主详情是 compact
2. 异步子面板仍按 full 渲染

因此显示级别判断应当在这些子面板模块中统一接入。

### 17.2 风险性评估

   - 当前很多详情区是长模板字符串，直接大面积重排容易放大 diff。

低风险判断依据：

1. 不涉及数据库迁移。
2. 不涉及接口契约变化。
3. 不涉及真实数据写入策略变化。
4. 不影响左侧列表与顶部统计。
5. 出问题时可通过环境变量立即切回 `full` 查看完整信息，或移除异常配置让系统回退到默认 `compact`。

   - 中

## 18. 低风险低改动实现方案

### 18.1 总体策略

建议采用以下原则：

1. 不做页面结构重构。
2. 不改后端业务接口。
3. 不新建复杂配置系统。
4. 复用现有 `window.BELLDANDY_WEB_CONFIG`。
5. 以前端详情渲染函数内条件裁剪为主。
6. 优先裁掉整块增强信息，少动现有核心字段拼装。

这是一条“保守但稳定”的实现路线。

### 18.2 推荐最小实现路径

第一步：接入环境变量到前端运行时配置

建议新增环境变量：

- `BELLDANDY_WEB_GOVERNANCE_DETAIL_MODE`

建议值：

- `full`
- `compact`

建议默认：

- `full`

最小实现方式：

1. 在网关 HTTP 层新增对 `/config.js` 的动态响应。
2. 继续输出 `window.BELLDANDY_WEB_CONFIG = {...}`。
3. 把现有链接配置保留。
4. 额外加入：
   - `governanceDetailMode`

为什么这是低风险方案：

1. 不需要改 `index.html` 的加载方式。
2. 不需要增加新的请求接口。
3. 不需要改前端初始化时序。
4. 与现有 `applyWebConfigLinks`、`residentAgentRosterEnabled` 的使用模式一致。

第二步：增加统一读取函数

建议新增一个非常轻的前端工具模块，例如：

- `app/features/governance-detail-mode.js`

只负责：

1. 读取 `window.BELLDANDY_WEB_CONFIG?.governanceDetailMode`
2. 归一化非法值
3. 暴露：
   - `getGovernanceDetailMode()`
   - `isCompactGovernanceDetailMode()`

为什么这是低风险方案：

1. 避免后续在多个文件里直接读全局对象。
2. 改动集中，未来想扩值也容易。
3. 不影响任何现有业务逻辑。

第三步：按页面做“区块级裁剪优先”

建议优先裁掉整块增强区块，而不是先在每个 grid 内删单卡片。

推荐首批区块级裁剪对象：

1. 任务记录
   - Work Recap
   - Resume Context
   - 来源解释
   - 复盘
   - Activity / Worklog
   - Method Usage 明细
   - Skill Usage 明细
   - 工具调用
   - Linked Memories
   - Artifacts
2. 记忆条目 / 共享审批
   - Metadata
   - 来源解释扩展区
   - 冲突说明
   - 来源审计
3. 消息审计
   - 风险清单
   - 检查清单
   - 建议回复草稿
   - 错误详情
   - 深度排障字段
4. 经验候选
   - Learning / Review Input 全量块
   - Source Snapshot
   - Source Memories
   - Source Artifacts
   - Tool Calls
   - 合成消化细节
5. 长期任务
   - Handoff 全量块
   - Governance 中模板 / 最近通知 / 分发队列 / Bridge 摘要 / Learning Input
   - Tracking 的全文列表和 history
   - Canvas / Capability / Progress 面板

为什么先这样做：

1. 改动最少。
2. 最容易保证简洁模式“明显变简洁”。
3. 不容易误伤核心字段。
4. HTML 模板 diff 更小。

第四步：再补少量字段级裁剪

仅在这些地方建议做字段级裁剪：

1. 记忆条目详情网格
2. 消息审计详情网格
3. 长期任务基础信息网格

原因：

1. 这些 grid 单字段过密，单靠裁区块不够。
2. 但其余页面很多内容本来就已经有明显区块边界，不值得继续细拆。
