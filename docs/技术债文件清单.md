## 本轮技术债偿还目标

- 在不伤及当前项目功能、能力、协议、配置语义与 WebChat 使用路径的前提下，对以下 4 个曾经的超大文件进行结构性拆解：
  - `packages/belldandy-core/src/server.ts`
  - `packages/belldandy-core/src/server.test.ts`
  - `apps/web/public/app.js`
  - `packages/belldandy-core/src/bin/gateway.ts`
- 本轮目标不是“重写”这 4 个文件，而是把它们从“主体逻辑承载文件”逐步收缩为：
  - 装配文件
  - 分发文件
  - 域级协调文件
- 拆解必须保持：
  - 现有 Gateway RPC 名称不变
  - 现有 HTTP / WebSocket 入口不变
  - 现有 WebChat 页面结构与主要操作路径不变
  - 现有环境变量、doctor 主结构、channel 启动顺序与运行时语义不变

## 总体可行性评估

### 结论

- 该技术债偿还工作整体`可行，且适合现在开始`。
- 原因不是这些大文件已经“完全无法维护”，而是它们已经明显跨过了“继续新增逻辑会自然恶化”的节点。
- 当前仓库已经存在较多可复用的承接模式，因此不需要引入新的大型框架，只需沿着现有模块化方向继续外移。

### 支撑可行性的现状

- `server.ts` 周边已经有独立 runtime / handler 模式可复用：
  - `query-runtime-message-send.ts`
  - `query-runtime-subtask.ts`
  - 多个 doctor/report builder
- `app.js` 周边已经有较多 feature 文件可承接：
  - `goals-*`
  - `memory-viewer.js`
  - `subtasks-overview.js`
  - `settings.js`
  - `tool-settings.js`
  - `workspace.js`
- `gateway.ts` 已经在最近几轮中出现“主体逻辑外移到独立 runtime 文件”的成功样本：
  - `background-recovery-runtime.ts`
  - `subtask-takeover-runtime.ts`
- `server.test.ts` 虽然巨大，但测试边界其实已经天然存在，可以按运行域拆分，而不是重新设计测试体系。

### 可行性判断

| 文件 | 可行性判断 | 说明 |
|---|---|---|
| `gateway.ts` | 高 | 以装配与启动块外移为主，属于低语义风险拆分 |
| `app.js` | 中高 | 已有 feature 文件较多，继续抽 orchestration 成本可控 |
| `server.ts` | 中 | 收益最大，但涉及 RPC 分发、doctor、HTTP/WS 入口，必须分批做 |
| `server.test.ts` | 高 | 拆测试文件本身风险不高，但要跟生产域边界同步推进 |

## 总体风险评估

### 主要风险

1. 初始化顺序漂移
   - 尤其在 `gateway.ts` 与 `server.ts`
   - 风险表现：某些 runtime 在依赖尚未就绪时提前启动

2. 隐式共享状态被拆断
   - 尤其在 `app.js`
   - 风险表现：UI 焦点、切屏、已选项、modal 状态出现局部回归

3. RPC / doctor 接线丢失
   - 尤其在 `server.ts`
   - 风险表现：方法仍存在，但 query runtime、doctor、broadcast 或鉴权逻辑漏接

4. 测试清理顺序失稳
   - 尤其在 `server.test.ts`
   - 风险表现：单测单跑正常，全量跑互相污染

### 风险控制原则

- 每一批只做`结构外移`，不顺手改业务语义。
- 每一批必须先做“接线文件变薄”，而不是“统一抽象”。
- 生产代码按运行域拆，测试代码按同样的运行域拆。
- 新文件只接受显式依赖参数，避免反向 import 原大文件形成循环依赖。
- 每一批都必须有定向验证，不接受“等全部拆完再测”。

## 总体工作量评估

### 结论

- 这是`中到大工作量`任务，不适合一次性做完。
- 更合理的方式是拆成`两轮到三轮`。

### 粗粒度预估

| 文件 | 工作量 | 说明 |
|---|---:|---|
| `gateway.ts` | 中 | 启动块和配置块可外移，改动面可控 |
| `app.js` | 中到大 | 共享 DOM / state / orchestration 多，前端接线较密 |
| `server.ts` | 大 | 巨型 `handleReq`、`system.doctor`、HTTP/WS 接线都需要拆 |
| `server.test.ts` | 中到大 | 测试域重组、helper 抽离、异步清理稳定化都要做 |

### 推荐分轮

- 第一轮：
  - `server.test.ts` testkit 抽离
  - `gateway.ts`
  - `app.js` 第一刀
  - `server.ts` 第一刀
- 第二轮：
  - `app.js` 第二刀
  - `server.ts` 第二刀
  - `server.test.ts` 域测试重组
- 第三轮：
  - 仅在前两轮稳定后再做剩余尾项

## 总体执行顺序建议

1. `server.test.ts` 先做 `T1: testkit 抽离`
2. `gateway.ts`
3. `app.js`
4. `server.ts`
5. `server.test.ts` 再做 `T2-T4`

### 顺序理由

- `server.test.ts` 的 `T1` 只抽共享 helper，语义风险低，但能先为后续分拆建立稳定 testkit。
- `gateway.ts` 仍然适合作为生产代码侧的第一刀，低风险、高确定性。
- `app.js` 已有承接 feature 文件，适合在入口瘦身模式建立后继续推进。
- `server.ts` 最重、最敏感，放在生产代码拆分的后位更稳。
- `server.test.ts` 的 `T2-T4` 仍应跟着 `server.ts` 的域边界一起拆，避免测试边界重复整理。
