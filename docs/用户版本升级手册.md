# Star Sanctuary 用户版本升级手册

> 适用对象：Star Sanctuary 使用者（安装器版 / GitHub Release 轻量正式附件 / Windows 运行包 / Docker 部署 / 源码部署）
> 更新日期：2026-04-27

---

## 1. 升级前须知

- 建议先查看目标版本 Release 说明：`https://github.com/vrboyzero/star-sanctuary/releases`
- 升级前建议备份状态目录，默认是 `~/.star_sanctuary`
- 升级后建议先验证 `/health` 接口中的 `version`
- GitHub Release 现在除了源码归档，还会提供正式轻量附件：
  - `star-sanctuary-dist-v<version>.zip`
  - `star-sanctuary-dist-v<version>.tar.gz`
  - `star-sanctuary-dist-v<version>.manifest.json`
  - `star-sanctuary-dist-v<version>.sha256`
- `Single-Exe` 当前发布口径为：`Windows x64 + Full`
- Star Sanctuary 当前只会自动检查新版本并提示，不会自动升级

---

## 2. 当前支持的升级方式（统一口径）

当前支持 5 类升级路径：

1. 安装器版用户：重新运行 `install.ps1 / install.sh`
2. GitHub Release 轻量正式附件用户：下载新的 `star-sanctuary-dist-v<version>.*` 并切换
3. Windows 运行包用户：下载新的 `Portable` 或 `Single-Exe`
4. Docker 用户：更新镜像 tag 后重建容器
5. 源码用户：切换到新 tag 或更新 `main` 后重新 `pnpm install / build`

说明：

- 这里的“升级”默认指版本升级
- 当前没有自动自升级；所有升级都需要用户显式执行
- 安装器版当前升级语义是：
  - 会备份现有 `current/`
  - 成功后切到新版本
  - 失败时回滚到旧版本
  - 若安装根已存在 `.env.local`，升级时默认跳过 `bdd setup`
  - 如需升级后强制重跑配置，可显式使用 `-ForceSetup / --force-setup`

---

## 3. 如何确认当前版本

### 3.1 通过健康接口确认

```bash
curl -s http://127.0.0.1:28889/health
```

预期示例：

```json
{
  "status": "ok",
  "timestamp": "2026-03-23T09:00:00.000Z",
  "version": "0.3.7"
}
```

### 3.2 查看启动日志

启动日志中会打印版本：

```text
Star Sanctuary Version: v0.3.7
```

### 3.3 查看发布包元数据

- 轻量正式附件：查看解压目录中的 `package.json`
- `Portable`：查看包目录中的 `version.json`
- `Single-Exe`：查看包目录中的 `single-exe.json`

## 4. 安装器版用户升级

### 4.1 Windows PowerShell

升级到最新版本：

```powershell
irm https://raw.githubusercontent.com/vrboyzero/star-sanctuary/main/install.ps1 | iex
```

升级到指定版本：

```powershell
& ([scriptblock]::Create((irm https://raw.githubusercontent.com/vrboyzero/star-sanctuary/main/install.ps1))) -Version vX.Y.Z
```

升级并跳过 setup：

```powershell
& ([scriptblock]::Create((irm https://raw.githubusercontent.com/vrboyzero/star-sanctuary/main/install.ps1))) -Version vX.Y.Z -NoSetup
```

升级并强制重跑 setup：

```powershell
& ([scriptblock]::Create((irm https://raw.githubusercontent.com/vrboyzero/star-sanctuary/main/install.ps1))) -Version vX.Y.Z -ForceSetup
```

### 4.2 Linux / macOS

升级到最新版本：

```bash
curl -fsSL https://raw.githubusercontent.com/vrboyzero/star-sanctuary/main/install.sh | bash
```

升级到指定版本：

```bash
curl -fsSL https://raw.githubusercontent.com/vrboyzero/star-sanctuary/main/install.sh | bash -s -- --version vX.Y.Z
```

升级并跳过 setup：

```bash
curl -fsSL https://raw.githubusercontent.com/vrboyzero/star-sanctuary/main/install.sh | bash -s -- --version vX.Y.Z --no-setup
```

升级并强制重跑 setup：

```bash
curl -fsSL https://raw.githubusercontent.com/vrboyzero/star-sanctuary/main/install.sh | bash -s -- --version vX.Y.Z --force-setup
```

### 4.3 说明

- 升级后会保留安装根 `.env` / `.env.local`
- 升级后默认继续复用已有配置，而不是强制重新进入 setup
- 若升级后你就是想刷新配置，再显式执行：
  - Windows：`<InstallDir>\bdd.cmd setup`
  - Linux/macOS：`<InstallDir>/bdd setup`

## 5. GitHub Release 下载版用户升级

### 5.1 轻量正式附件用户

适用对象：

- 会使用终端
- 自己部署
- 不需要内置 Node runtime

建议步骤：

1. 关闭当前正在运行的 Star Sanctuary。
2. 备份状态目录 `~/.star_sanctuary`（可选但推荐）。
3. 从 GitHub Release 下载新的正式轻量附件：
   - Windows：`star-sanctuary-dist-v<version>.zip`
   - Linux / macOS：`star-sanctuary-dist-v<version>.tar.gz`
4. 解压到新目录。
5. 若你在旧目录旁放过 `.env` / `.env.local`，按需复制到新目录。
6. 启动：
   - Windows：`start.bat`
   - Linux / macOS：`./start.sh`

说明：

- 轻量正式附件已包含构建好的 `dist`、Web 静态资源、模板、安装/启动脚本与 `.env.example`
- 它**不包含** Node runtime 与 `node_modules`
- 启动脚本会在缺依赖时自动执行 `corepack pnpm install`
- 当前安装器默认仍消费 release 源码归档；轻量正式附件主要服务手工部署、官网下载与后续包管理器准备

### 5.2 Portable 用户

建议步骤：

1. 关闭当前正在运行的 Star Sanctuary。
2. 备份旧包目录中的 `.env` / `.env.local`（如果有）。
3. 备份状态目录 `~/.star_sanctuary`（可选但推荐）。
4. 下载新的 `Portable Slim` 或 `Portable Full` 包并解压到新目录。
5. 将旧包中的 `.env` / `.env.local` 按需复制到新目录。
6. 启动新目录中的 `start.bat` 或 `start.ps1`。

说明：

- `Portable` 的用户数据默认不跟包目录绑定，仍写在 `~/.star_sanctuary`
- 若你删除了新包里的 `runtime/`，下次启动会按当前产物重新准备运行时
- 不要把旧包里的 `runtime/`、`launcher/`、`payload/` 只拷一部分到新包里混用

### 5.3 Single-Exe 用户

当前 `Single-Exe` 只发布 `Windows x64 Full`。

建议步骤：

1. 关闭当前正在运行的 Star Sanctuary。
2. 备份状态目录 `~/.star_sanctuary`（可选但推荐）。
3. 下载新的 `Single-Exe Full` 包。
4. 保留新包目录中的：
   - `star-sanctuary-single.exe`
   - `single-exe.json`
   - `README-single-exe.md`
   - `README-single-exe-zh.md`
   - `.env.example`
5. 双击新的 `star-sanctuary-single.exe` 启动。

说明：

- `Single-Exe` 的运行时缓存默认位于：
  - `%LOCALAPPDATA%\StarSanctuary\runtime\<version>-win32-x64`
- 升级到新版本后，会使用新的 runtime 版本目录
- `.env` / `.env.local`、sessions、logs、workspace、state 默认仍位于 `~/.star_sanctuary`
- 如果升级后首次启动失败，可删除当前版本对应的 runtime 缓存目录后重新运行

### 5.4 回滚建议

如果 Release 下载版升级后需要回滚：

1. 关闭当前 Star Sanctuary 进程。
2. 重新启动上一版的包。
3. 如有必要，删除本次升级新生成的 runtime 缓存目录后再启动旧版。

注意：

- 常规升级/回滚不会主动清空 `~/.star_sanctuary`
- 为避免误操作，生产或长期使用环境仍建议升级前先备份

## 6. Windows 运行包与轻量正式附件的选择建议

如果你只是想快速判断该走哪条升级线：

- 想继续用终端安装 / 升级：走安装器版
- 想手工下载一个更小的正式附件自己部署：走轻量正式附件
- 想下载后直接双击运行：走 `Portable` 或 `Single-Exe`

一句话区分：

- `star-sanctuary-dist-v<version>.*`：给命令安装、开发者部署、后续包管理器使用
- `Portable / Single-Exe`：给 Windows 普通用户直接运行

## 7. Docker 用户升级

### 7.1 推荐方式：固定版本号升级（生产环境推荐）

1. 修改 `.env`：

```bash
BELLDANDY_IMAGE=vrboyzero/star-sanctuary:X.Y.Z
```

2. 拉取镜像并重建服务：

```bash
docker pull vrboyzero/star-sanctuary:X.Y.Z
docker compose up -d --remove-orphans
```

3. 验证版本：

```bash
curl -s http://127.0.0.1:28889/health
```

### 7.2 可选方式：latest（仅测试环境）

```bash
docker pull vrboyzero/star-sanctuary:latest
docker compose up -d --remove-orphans
```

> `latest` 可能包含尚未验证的变化，不建议生产使用。

### 7.3 回滚（Docker）

将 `.env` 中镜像版本改回旧版本后重启：

```bash
BELLDANDY_IMAGE=vrboyzero/star-sanctuary:X.Y.Z
docker compose up -d --remove-orphans
```

## 8. 源码用户升级

### 8.1 升级到指定稳定版本（推荐）

```bash
git fetch --tags
git checkout vX.Y.Z
corepack pnpm install --frozen-lockfile
corepack pnpm build
```

启动：

- Windows：`.\start.bat`
- Linux/macOS：`./start.sh`

### 8.2 跟随主分支（开发试用）

```bash
git pull origin main
corepack pnpm install
corepack pnpm build
```

## 9. 版本检查提醒（自动提示）

Star Sanctuary 启动后会异步检查 GitHub 最新 Release。若有新版本，会输出日志提示，不会自动升级。

示例：

```text
[INFO] New version available: vX.Y.Z (current: vA.B.C)
[INFO] Upgrade: https://github.com/vrboyzero/star-sanctuary/releases/tag/vX.Y.Z
```

可选配置（`.env` / `.env.local`）：

```bash
BELLDANDY_UPDATE_CHECK=true
BELLDANDY_UPDATE_CHECK_TIMEOUT_MS=3000
BELLDANDY_UPDATE_CHECK_API_URL=https://api.github.com/repos/vrboyzero/star-sanctuary/releases/latest
```

关闭检查：

```bash
BELLDANDY_UPDATE_CHECK=false
```

## 10. 常见问题排查

### 10.1 升级后版本没变化

检查：

1. 是否拿到了正确的包或镜像版本
2. 是否重启了服务
3. `/health` 返回里是否含 `version`
4. 轻量正式附件的 `package.json`、`Portable` 的 `version.json` 或 `Single-Exe` 的 `single-exe.json` 是否与目标版本一致

### 10.2 升级后启动失败

建议顺序：

1. 查看最后一段日志或控制台输出
2. 对轻量正式附件确认本机 Node.js 22 与 `corepack pnpm install` 是否执行成功
3. 对 `Single-Exe` 删除当前版本 runtime 缓存目录后重试
4. 对 `Portable` 保留 `payload/`，删除 `runtime/` 后重试
5. 仍失败则回滚到上一个稳定版本

### 10.3 数据会不会丢失

常规升级/回滚不会主动清空 `~/.star_sanctuary`。

但为避免误操作，仍建议升级前备份。
