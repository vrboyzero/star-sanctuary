# Star Sanctuary 用户版本升级手册

> 适用对象：Star Sanctuary 使用者（Docker 部署 / 源码部署）  
> 更新日期：2026-03-05

---

## 1. 升级前须知

- 建议先查看目标版本 Release 说明：`https://github.com/vrboyzero/star-sanctuary/releases`
- 升级前建议备份状态目录（默认 `~/.star_sanctuary`）
- 升级后请先验证 `/health` 接口中的 `version`

---

## 2. 如何确认当前版本

### 2.1 通过健康接口确认

```bash
curl -s http://127.0.0.1:28889/health
```

预期示例：

```json
{
  "status": "ok",
  "timestamp": "2026-03-05T09:00:00.000Z",
  "version": "0.1.0"
}
```

### 2.2 查看启动日志

启动日志中会打印版本：
```
Star Sanctuary Version: v0.1.0
```

---

## 3. Docker 用户升级

## 3.1 推荐方式：固定版本号升级（生产环境推荐）

1. 修改 `.env`：

```bash
BELLDANDY_IMAGE=vrboyzero/star-sanctuary:0.1.1
```

2. 拉取镜像并重建服务：

```bash
docker pull vrboyzero/star-sanctuary:0.1.1
docker compose up -d --remove-orphans
# 旧环境可用：docker-compose up -d --remove-orphans
```

3. 验证版本：

```bash
curl -s http://127.0.0.1:28889/health
```

## 3.2 可选方式：latest（仅测试环境）

```bash
docker pull vrboyzero/star-sanctuary:latest
docker compose up -d --remove-orphans
```

> `latest` 可能包含尚未验证的变化，不建议生产使用。

## 3.3 回滚（Docker）

将 `.env` 中镜像版本改回旧版本后重启：

```bash
BELLDANDY_IMAGE=vrboyzero/star-sanctuary:0.1.0
docker compose up -d --remove-orphans
```

## 4. 源码用户升级

## 4.1 升级到指定稳定版本（推荐）

```bash
git fetch --tags
git checkout v0.1.1
corepack pnpm install --frozen-lockfile
corepack pnpm build
```

启动：
- Windows：`.\start.bat`
- Linux/macOS：`./start.sh`

## 4.2 跟随主分支（开发试用）

```bash
git pull origin main
corepack pnpm install
corepack pnpm build
```

## 5. 版本检查提醒（自动提示）

Star Sanctuary 启动后会异步检查 GitHub 最新 Release。若有新版本，会输出日志提示，不会自动升级。

示例：
```
[INFO] New version available: v0.2.0 (current: v0.1.0)
[INFO] Upgrade: https://github.com/vrboyzero/star-sanctuary/releases/tag/v0.2.0
```

可选配置（`.env`/`.env.local`）：

```bash
BELLDANDY_UPDATE_CHECK=true
BELLDANDY_UPDATE_CHECK_TIMEOUT_MS=3000
BELLDANDY_UPDATE_CHECK_API_URL=https://api.github.com/repos/vrboyzero/star-sanctuary/releases/latest
```

关闭检查：

```bash
BELLDANDY_UPDATE_CHECK=false
```

## 6. 常见问题排查

### 6.1 升级后版本没变化

检查：
1. 是否拉取到了正确镜像 tag（Docker）
2. 是否切到了正确 git tag（源码）
3. 是否重启了服务
4. `/health` 返回里是否含 `version`

### 6.2 升级后启动失败

建议顺序：
1. 查看容器/服务日志
2. 回滚到上一个稳定版本
3. 保留日志并反馈 issue

### 6.3 数据会不会丢失

常规升级/回滚不会清空 `~/.star_sanctuary`（挂载 volume 持久化）。  
但为避免误操作，生产环境仍建议升级前备份。
