# Star Sanctuary 用户版本升级手册

> 适用对象：Star Sanctuary 使用者（标准发布版 / Docker 部署 / 源码部署）
> 更新日期：2026-03-27

---

## 1. 升级前须知

- 建议先查看目标版本 Release 说明：`https://github.com/vrboyzero/star-sanctuary/releases`
- 升级前建议备份状态目录，默认是 `~/.star_sanctuary`
- 升级后建议先验证 `/health` 接口中的 `version`
- `Single-Exe` 当前发布口径为：`Windows x64 + Full`

---

## 2. 如何确认当前版本

### 2.1 通过健康接口确认

```bash
curl -s http://127.0.0.1:28889/health
```

预期示例：

```json
{
  "status": "ok",
  "timestamp": "2026-03-23T09:00:00.000Z",
  "version": "0.2.2"
}
```

### 2.2 查看启动日志

启动日志中会打印版本：

```text
Star Sanctuary Version: v0.2.2
```

### 2.3 查看标准包元数据

- `Portable`：查看包目录中的 `version.json`
- `Single-Exe`：查看包目录中的 `single-exe.json`

---

## 3. 标准发布版用户升级

### 3.1 Portable 用户

建议步骤：

1. 关闭当前正在运行的 Star Sanctuary。
2. 备份旧包目录中的 `.env` / `.env.local`（如果有）。
3. 备份状态目录 `~/.star_sanctuary`（可选但推荐）。
4. 下载新的 `Portable Slim` 或 `Portable Full` 包并解压到新目录。
5. 将旧包中的 `.env` / `.env.local` 按需复制到新目录。
6. 启动新目录中的 `start.bat` 或 `start.ps1`。

说明：

- `Portable` 的用户数据默认不跟包目录绑定，仍写在 `~/.star_sanctuary`
- 若你删除了新包里的 `runtime/`，下次启动会按当前产物重新准备运行时
- 不要把旧包里的 `runtime/`、`launcher/`、`payload/` 只拷一部分到新包里混用

### 3.2 Single-Exe 用户

当前 `Single-Exe` 只发布 `Windows x64 Full`。

建议步骤：

1. 关闭当前正在运行的 Star Sanctuary。
2. 备份状态目录 `~/.star_sanctuary`（可选但推荐）。
3. 下载新的 `Single-Exe Full` 包。
4. 保留新包目录中的：
   - `star-sanctuary-single.exe`
   - `single-exe.json`
   - `README-single-exe.md`
   - `README-single-exe-zh.md`
   - `.env.example`
5. 双击新的 `star-sanctuary-single.exe` 启动。

说明：

- `Single-Exe` 的运行时缓存默认位于：
  - `%LOCALAPPDATA%\StarSanctuary\runtime\<version>-win32-x64`
- 升级到新版本后，会使用新的 runtime 版本目录
- `.env` / `.env.local`、sessions、logs、workspace、state 默认仍位于 `~/.star_sanctuary`
- 如果升级后首次启动失败，可删除当前版本对应的 runtime 缓存目录后重新运行

### 3.3 回滚建议

如果标准包升级后需要回滚：

1. 关闭当前 Star Sanctuary 进程。
2. 重新启动上一版的包。
3. 如有必要，删除本次升级新生成的 runtime 缓存目录后再启动旧版。

注意：

- 常规升级/回滚不会主动清空 `~/.star_sanctuary`
- 为避免误操作，生产或长期使用环境仍建议升级前先备份

## 4. Docker 用户升级

### 4.1 推荐方式：固定版本号升级（生产环境推荐）

1. 修改 `.env`：

```bash
BELLDANDY_IMAGE=vrboyzero/star-sanctuary:0.2.2
```

2. 拉取镜像并重建服务：

```bash
docker pull vrboyzero/star-sanctuary:0.2.2
docker compose up -d --remove-orphans
```

3. 验证版本：

```bash
curl -s http://127.0.0.1:28889/health
```

### 4.2 可选方式：latest（仅测试环境）

```bash
docker pull vrboyzero/star-sanctuary:latest
docker compose up -d --remove-orphans
```

> `latest` 可能包含尚未验证的变化，不建议生产使用。

### 4.3 回滚（Docker）

将 `.env` 中镜像版本改回旧版本后重启：

```bash
BELLDANDY_IMAGE=vrboyzero/star-sanctuary:0.1.1
docker compose up -d --remove-orphans
```

## 5. 源码用户升级

### 5.1 升级到指定稳定版本（推荐）

```bash
git fetch --tags
git checkout v0.2.2
corepack pnpm install --frozen-lockfile
corepack pnpm build
```

启动：

- Windows：`.\start.bat`
- Linux/macOS：`./start.sh`

### 5.2 跟随主分支（开发试用）

```bash
git pull origin main
corepack pnpm install
corepack pnpm build
```

## 6. 版本检查提醒（自动提示）

Star Sanctuary 启动后会异步检查 GitHub 最新 Release。若有新版本，会输出日志提示，不会自动升级。

示例：

```text
[INFO] New version available: v0.2.2 (current: v0.2.1)
[INFO] Upgrade: https://github.com/vrboyzero/star-sanctuary/releases/tag/v0.2.2
```

可选配置（`.env` / `.env.local`）：

```bash
BELLDANDY_UPDATE_CHECK=true
BELLDANDY_UPDATE_CHECK_TIMEOUT_MS=3000
BELLDANDY_UPDATE_CHECK_API_URL=https://api.github.com/repos/vrboyzero/star-sanctuary/releases/latest
```

关闭检查：

```bash
BELLDANDY_UPDATE_CHECK=false
```

## 7. 常见问题排查

### 7.1 升级后版本没变化

检查：

1. 是否拿到了正确的包或镜像版本
2. 是否重启了服务
3. `/health` 返回里是否含 `version`
4. `Portable` 的 `version.json` 或 `Single-Exe` 的 `single-exe.json` 是否与目标版本一致

### 7.2 升级后启动失败

建议顺序：

1. 查看最后一段日志或控制台输出
2. 对 `Single-Exe` 删除当前版本 runtime 缓存目录后重试
3. 对 `Portable` 保留 `payload/`，删除 `runtime/` 后重试
4. 仍失败则回滚到上一个稳定版本

### 7.3 数据会不会丢失

常规升级/回滚不会主动清空 `~/.star_sanctuary`。

但为避免误操作，仍建议升级前备份。
