# 官网展示用的安装命令方案

PLAN

Goal
为 Star Sanctuary 设计一版适合官网展示的安装命令方案，缩短当前基于 GitHub Raw 的安装命令长度，提升可读性、可记忆性与产品化观感；命令主体优先体现 **项目名 StarSanctuary**，同时不破坏现有 `install.ps1` / `install.sh` 安装脚本体系。

Constraints
- 当前 `README.md` 先不改，等实现完成后再统一调整官网与 README 文案。
- 优先保持现有安装脚本 `install.ps1` / `install.sh` 可继续复用，不做无必要重写。
- 官网展示命令应稳定，不应把 GitHub 仓库路径、分支名、原始脚本地址直接暴露为长期用户入口。
- 方案应兼顾 Windows PowerShell 与 Linux/macOS Shell。
- 需要保留后续切换脚本源、切换版本策略、增加灰度能力的余地。

Steps
1. 明确为什么当前命令长，以及短命令要解决的真实问题。
2. 设计官网展示命令、项目名安装入口、真实脚本源三层结构。
3. 比较若干候选方案，确定推荐方案与备用方案。
4. 补充包管理器维护成本、开发工作量与兼容现有安装流程的判断。
5. 给出官网展示文案口径、实现要点、上线顺序与回滚方式。

Validation
- 官网看到的命令长度明显缩短。
- 未来 GitHub 仓库路径、分支、脚本落点变化时，官网命令保持不变。
- Windows 与 Linux/macOS 都能继续沿用当前安装脚本逻辑。

## 1. 背景判断

当前安装命令：

```powershell
irm https://raw.githubusercontent.com/vrboyzero/star-sanctuary/main/install.ps1 | iex
```

```bash
curl -fsSL https://raw.githubusercontent.com/vrboyzero/star-sanctuary/main/install.sh | bash
```

它们显得长，主要不是因为 `irm` / `curl` 本身，而是因为：

1. `raw.githubusercontent.com` 域名较长
2. URL 中包含 `owner / repo / branch / file`
3. 当前用户入口直接暴露了 GitHub Raw 脚本源

所以问题的本质不是“GitHub 不能用”，而是：

- **GitHub Raw 很适合作为脚本源**
- **GitHub Raw 不适合作为官网长期展示入口**

### 1.1 当前安装器实际安装的是什么

当前安装器默认**不是安装 `main` 分支最新源码**，而是：

- 默认查询 GitHub Releases 的 `latest`
- 解析出该 release 对应的 `tag`
- 下载这个 `tag` 对应的**源码归档**
- 在用户本机执行安装与构建

也就是说，当前实际模式是：

```text
最新 Release 版本
  -> 对应 tag 的源码归档
    -> 本地安装 / 本地构建
```

它不是下面两种模式：

- 不是“直接安装 `main` 分支最新源码”
- 不是“直接安装 release 中的现成预编译二进制运行时”

从当前脚本行为来看：

- `install.ps1` 默认 `Version = "latest"`
- `install.sh` 默认 `VERSION="latest"`
- 两者都会请求：

```text
https://api.github.com/repos/<owner>/<repo>/releases/latest
```

然后再下载该 release 对应的源码包：

- PowerShell 侧走 `zipball_url`
- Shell 侧走 `tarball_url`

如果手动指定版本号，例如 `v0.x.y`，当前逻辑也是下载这个 `tag` 的源码归档，而不是去取 `main`。

这个判断很重要，因为它意味着：

- 你现在的安装口径本质上是 **“按 release 安装”**
- 后续即使把官网命令缩短，也**不必改变现有安装语义**
- 新短命令完全可以继续复用当前“latest release -> 对应源码归档 -> 本地构建”这套流程

## 2. 设计目标与命名原则

官网展示用安装命令应满足以下目标：

- 更短
- 更像正式产品，而不是开发者临时脚本地址
- 命令主体优先体现 **项目名**，而不是官网品牌站
- 更容易记忆
- 后续仓库迁移、分支调整、脚本升级时，用户命令不变
- 可以在服务端做版本路由、灰度、跳转与审计

这里要特别区分两件事：

1. **命令里展示的品牌主体**
2. **命令背后的分发机制**

如果继续走“远程拉脚本执行”这条路线，那么命令里就一定会有 URL。  
因此它不可能直接凭空变成：

```text
StarSanctuary install
```

除非用户机器上已经先装好了一个叫 `StarSanctuary` 的本地命令，或者已经接入了 `brew / winget / npm` 这类包管理器。

所以当前阶段真正能做的是：

- 让 **URL 主体体现项目名**
- 不必先改成包管理器模式

## 3. 推荐方案

### 3.1 总体结构

建议把安装链路拆成三层：

1. **官网展示命令**
2. **项目名安装入口域名**
3. **真实脚本源地址**

关系如下：

```text
官网展示命令
  -> install.starsanctuary.<your-domain>
    -> 由你控制的项目名入口
      -> 再转发或回源到 GitHub / Release / 其他对象存储
```

### 3.2 推荐展示命令

Windows：

```powershell
irm https://install.starsanctuary.<your-domain>/win | iex
```

Linux / macOS：

```bash
curl -fsSL https://install.starsanctuary.<your-domain>/unix | bash
```

这是当前最推荐的官网展示口径。

原因：

- 足够短
- 容易理解
- 主体是 `StarSanctuary`，不是官网品牌站
- 平台区分清晰
- 后端实现简单
- 不必先改安装器主流程

### 3.3 可接受的另一种写法

如果你不想单独开 `install.` 子域名，也可以直接用项目名主域：

```powershell
irm https://starsanctuary.<your-domain>/install.ps1 | iex
```

```bash
curl -fsSL https://starsanctuary.<your-domain>/install.sh | bash
```

这同样满足“主体体现项目名”的要求。  
只是相比 `/win`、`/unix` 方案，文件名写法会略长一点。

## 4. 备用网址设计

为了兼顾可读性与实现灵活度，建议预留以下别名：

### 4.1 文件式别名

```powershell
irm https://install.starsanctuary.<your-domain>/install.ps1 | iex
```

```bash
curl -fsSL https://install.starsanctuary.<your-domain>/install.sh | bash
```

优点：

- 和当前仓库脚本名一致
- 对技术用户更直观

缺点：

- 比 `/win`、`/unix` 稍长

### 4.2 版本式别名

例如：

```text
https://install.starsanctuary.<your-domain>/stable/win
https://install.starsanctuary.<your-domain>/stable/unix
https://install.starsanctuary.<your-domain>/nightly/win
https://install.starsanctuary.<your-domain>/nightly/unix
```

优点：

- 以后可以区分稳定版 / 预览版 / nightly

缺点：

- 官网首屏命令会变长

建议：

- 官网主命令先只展示 `/win` 与 `/unix`
- 高级安装页或文档页再介绍 `stable / nightly`

## 5. 不推荐继续直接展示 GitHub Raw 的原因

不推荐官网长期直接展示：

```powershell
irm https://raw.githubusercontent.com/vrboyzero/star-sanctuary/main/install.ps1 | iex
```

```bash
curl -fsSL https://raw.githubusercontent.com/vrboyzero/star-sanctuary/main/install.sh | bash
```

原因：

### 5.1 视觉上不像正式产品入口

- 更像源码仓库地址
- 官网观感偏“开发者脚本”，不够产品化

### 5.2 入口被仓库结构绑死

只要以下任一项变化，就需要改官网文案：

- GitHub 用户名
- 仓库名
- 默认分支
- 脚本路径

### 5.3 缺少中间控制层

如果后续需要：

- 切换到 Release 附件
- 切到 CDN
- 临时回滚到旧版本脚本
- 灰度某一版安装器

直接展示 GitHub Raw 都不方便。

## 6. 推荐实现口径

### 6.1 官网展示口径

建议官网展示：

```powershell
irm https://install.starsanctuary.<your-domain>/win | iex
```

```bash
curl -fsSL https://install.starsanctuary.<your-domain>/unix | bash
```

说明文案建议：

- Windows PowerShell
- macOS / Linux

不要在官网首屏直接展示 GitHub Raw 地址。

### 6.2 实际脚本源口径

短域名后面可以先简单回源到当前仓库脚本：

```text
/win   -> GitHub Raw install.ps1
/unix  -> GitHub Raw install.sh
```

这样首版实现成本最低。

等后续需要时，再把短域名后面的真实源切到：

- GitHub Release 产物
- 对象存储
- 你自己的安装分发服务

用户侧命令不变。

## 7. 候选方案对比

### 方案 A：继续直接展示 GitHub Raw

示例：

```powershell
irm https://raw.githubusercontent.com/vrboyzero/star-sanctuary/main/install.ps1 | iex
```

```bash
curl -fsSL https://raw.githubusercontent.com/vrboyzero/star-sanctuary/main/install.sh | bash
```

评价：

- `record_only`
- 可用，但不适合作为正式官网主入口

### 方案 B：短域名 + 回源 GitHub Raw

示例：

```powershell
irm https://install.starsanctuary.<your-domain>/win | iex
```

```bash
curl -fsSL https://install.starsanctuary.<your-domain>/unix | bash
```

评价：

- `fix_now`
- 这是当前阶段最优方案

### 方案 C：包管理器命令

示例：

```text
winget install StarSanctuary
brew install star-sanctuary
npm i -g star-sanctuary
```

评价：

- `defer`
- 命令最短，但维护成本更高
- 适合后续成熟阶段补充，不是当前最小成本方案

#### 方案 C 的真实含义

这里要特别注意：

- `winget install StarSanctuary`
- `brew install star-sanctuary`

这类命令的价值，不只是“更短”。

它们背后意味着：

1. 项目已经进入某个包管理器生态
2. 项目已经提供了包管理器能识别的稳定发布物
3. 项目已经维护了版本清单、校验信息、升级路径、卸载路径

也就是说，**方案 C 不是改一句官网命令，而是新增一套正式分发体系。**

#### 方案 C 需要准备的东西

无论是 `winget` 还是 `brew`，至少都要准备：

1. **稳定版本号**
   - 例如 `v0.3.5`
   - 不能长期只靠 `main` 分支最新提交

2. **稳定下载地址**
   - 指向明确 release 产物
   - URL 结构不能频繁变化

3. **稳定发布物**
   - 安装包
   - zip / tar.gz
   - 或者可直接分发的预编译运行时

4. **校验信息**
   - `sha256` 等 hash
   - 用于包管理器校验下载内容

5. **明确安装后入口**
   - 最终用户装完后，命令名是什么
   - 安装到哪里
   - 如何升级、如何卸载

#### 方案 C 与当前安装器的关系

当前安装器的语义是：

```text
latest release
  -> 对应源码归档
    -> 本地执行 install/build/setup
```

如果做包管理器，有两种方向：

##### 方向 1：尽量复用现有安装器

思路：

- `winget` / `brew` 下载一个很薄的入口
- 再由这个入口继续调用现有 `install.ps1` / `install.sh`

优点：

- 改动小
- 现有安装逻辑能复用

缺点：

- 对包管理器生态来说不够“原生”
- 升级 / 卸载 / 文件追踪会更别扭
- 长期维护体验通常一般

##### 方向 2：为包管理器做原生分发适配

思路：

- 为 `winget` 提供 Windows 安装包
- 为 `brew` 提供 formula 可消费的源码包或 bottle
- 安装、升级、卸载尽量走包管理器原生规则

优点：

- 用户体验更标准
- 升级和卸载语义更清晰
- 更适合长期维护

缺点：

- 开发与发布工作明显更多

结论：

- 如果只是“想要更短的官网命令”，没必要立刻上方案 C
- 如果是“想进入系统级包管理器生态”，那就要按正式分发项目来做

#### 方案 C 的推荐先后顺序

如果以后真的要做，建议顺序是：

1. 先做 `winget`
2. 再做 `brew`
3. 最后再评估 `npm`

原因：

- 你的当前用户重心更偏 Windows
- 现有安装器已经有 PowerShell 侧实现
- `winget` 的路径与现有 Windows 发布方式更接近
- `brew` 需要考虑 macOS / Linux 差异，适配成本更高

#### 当前判断（2026-04-27）

当前不再考虑“短期折中版”：

- **不把 `brew install star-sanctuary` 做成 `install.sh` 的别名**
- **方案 C 直接按长期更正统的分发方案推进**
- **先实现 `winget`，`brew` 暂缓**

原因很直接：

- `winget` 天然更接近当前 Windows 用户面和现有 `portable / single-exe` 分发线
- `brew` 如果只是包一层 `install.sh`，命令虽然变短，但安装语义并不真正符合 Homebrew 生态
- 一旦开始做方案 C，更值得把第一步投入到后续可长期维护、可继续扩展的链路上

## 8. 为什么 `brew install star-sanctuary` 维护成本更高

短命令本身不复杂，复杂的是它背后的发布和维护链路。

主要成本在以下几类：

### 8.1 要维护包描述与发布元数据

例如：

- Homebrew formula / tap
- winget manifest
- npm package metadata

这些都不是一次性工作。只要发新版本，就需要跟着更新。

### 8.2 要维护稳定的发布产物

包管理器依赖这些东西长期稳定：

- 下载地址
- 压缩包结构
- 安装后入口名称
- checksum / hash

如果产物结构经常变化，包管理器安装链路会很容易碎。

### 8.3 要维护多平台兼容性

例如：

- Windows
- macOS Intel
- macOS Apple Silicon
- Linux 不同发行版

不同平台的安装路径、权限模型、依赖处理方式都不完全一样。

### 8.4 要维护升级、回滚和排障

包管理器模式下，不只是“能安装”：

- 还要能升级
- 还要能回滚
- 还要能解释失败原因

否则命令虽然短，但用户体验未必更好。

### 8.5 要维护 CI / 自动发布流程

要真正长期用好包管理器，通常还要补：

- 发版产物生成
- 自动更新 manifest / formula
- 自动算 hash
- 自动校验下载地址

不自动化的话，每次发版都会有额外维护成本。

### 8.6 现有安装器不等于现成可接 `brew`

当前仓库的安装逻辑更接近：

- 下载源码
- 本地 `corepack pnpm install`
- 本地 `corepack pnpm build`
- 再运行 setup

这套逻辑对独立安装器是合理的，但对 `brew` 来说不够自然。

因为 `brew` 更希望消费的是：

- 版本化源码包
- 或 bottle（二进制包）
- 并由 formula 明确声明安装过程

所以如果未来真接 `brew`，大概率需要对当前发布物再做一层适配，而不是简单把 `brew install` 当成 `install.sh` 的别名。

## 8A. 要实现 `winget install StarSanctuary`，具体要做什么

### 8A.1 目标结果

用户能够在 Windows 上执行：

```powershell
winget install StarSanctuary
```

或：

```powershell
winget install <正式 PackageIdentifier>
```

然后由 Windows Package Manager 完成下载、安装、升级、卸载管理。

### 8A.2 需要准备的发布物

至少要有一个 Windows 可分发安装物，例如：

- `.msi`
- `.exe`
- `.msix`
- 或者社区可接受的便携包（portable）

就 Star Sanctuary 当前形态看，更现实的是：

1. 基于现有 Single-Exe / Portable 产物，整理出稳定 Windows 分发包
2. 为它提供稳定版本下载地址
3. 为该安装物生成 hash

### 8A.3 需要用到的工具

至少会涉及：

- `wingetcreate`
  - 用来生成 package manifest
- `winget validate`
  - 用来本地校验 manifest
- GitHub 仓库协作
  - 提交到 `microsoft/winget-pkgs`

### 8A.4 需要维护的内容

通常需要维护：

1. `PackageIdentifier`
2. `PackageVersion`
3. `InstallerUrl`
4. `InstallerSha256`
5. `InstallerType`
6. `Publisher`
7. `ShortDescription`
8. 升级行为与静默安装参数

也就是说，每次发新版本时，至少要重复做：

1. 生成新的 Windows 安装包
2. 上传到稳定下载地址
3. 更新 manifest 中的版本号与 hash
4. 本地校验
5. 向 `winget-pkgs` 提交 PR

### 8A.5 推荐实施步骤

推荐落地顺序：

1. 先确定 Windows 正式安装包形态
   - 优先选一个长期稳定的类型
   - 不建议在 `.exe` / `.msi` / 便携模式之间频繁切换

2. 固化 release 发布目录结构
   - 每个版本下载地址有固定模式
   - 文件名有固定模式

3. 手工做第一版 manifest
   - 用 `wingetcreate new`
   - 或手工维护 YAML

4. 本地校验
   - `winget validate`

5. 提交到 `winget-pkgs`

6. 等第一版稳定后，再考虑自动化更新 manifest

### 8A.6 对 Star Sanctuary 的额外开发工作

除了 manifest 本身，你还大概率要补：

- 静默安装参数整理
- 卸载命令验证
- 升级覆盖安装验证
- 首次启动体验验证
- 安装路径稳定性验证

如果当前 Windows 产物还没有“标准安装包语义”，那么这部分会是主要工作量。

### 8A.7 当前首轮实施边界

本轮开始正式推进 `winget`，但边界明确如下：

1. **不走主安装器通道**
   - 不让 `winget` 再去回调 `install.ps1`
   - 而是直接消费一份正式 Windows 发布资产

2. **首轮先以 Windows portable 正式附件为基础**
   - 先不碰 `single-exe`
   - 先不做 `.msi / .msix`
   - 先把 `portable` 整理为稳定 zip 附件，作为 `winget` 的安装输入

3. **先做仓库内可重复生成的产物与 manifest**
   - 新增 `build:winget`
   - 新增 `verify:winget`
   - 本地输出：
     - Windows portable `zip`
     - `sha256`
     - `winget` YAML manifests

4. **先不直接提交外部仓库**
   - `microsoft/winget-pkgs` 提交仍属于后续动作
   - 这一轮先把本仓库侧的产物准备链打通

5. **GitHub Release 的 Windows 附件上传链路后补**
   - 当前 tag release job 跑在 `ubuntu-latest`
   - 现有 `build-portable.mjs` 只支持 Windows
   - 所以下一步要补的是 Windows 侧 release asset job，而不是把 manifest 生出来就算完成

### 8A.8 当前实施进度

当前仓库内已经完成第一批 `winget` 基础支撑：

- 已新增 `build:winget`
- 已新增 `verify:winget`
- 已把 Windows tag release asset job 接入 GitHub Actions workflow
- 已可从现有 Windows portable 目录生成：

## 8B. 要实现 `brew install star-sanctuary`，具体要做什么

### 8B.1 目标结果

用户能够在 macOS / Linux 上执行：

```bash
brew install star-sanctuary
```

然后由 Homebrew 安装、管理、升级和卸载。

### 8B.2 需要准备的发布物

通常有两条路径：

#### 路径 A：源码包 + formula

提供：

- 版本化源码 tarball / zip
- formula 中的 `url`
- formula 中的 `sha256`
- formula 中的 `install` 逻辑

#### 路径 B：预编译 bottle

提供：

- 各平台 bottle
- 对应 hash
- formula 中的 bottle 配置

对于 Star Sanctuary 这种需要 Node / pnpm / workspace 构建的项目，长期更稳的通常是：

- 要么做更标准的源码安装路径
- 要么干脆准备更稳定的预编译分发物

### 8B.3 需要用到的工具

通常会涉及：

- `brew tap-new`
  - 新建自己的 tap
- `brew create <URL>`
  - 从源码包 URL 创建 formula 骨架
- `brew audit`
  - 检查 formula
- `brew test-bot`
  - 做更完整的构建 / 测试检查

### 8B.4 需要维护的内容

至少要维护：

1. formula 文件
2. `url`
3. `sha256`
4. `version`
5. `depends_on`
6. `install` 方法
7. `test do`

如果后续要上 bottle，还要继续维护：

- bottle 构建产物
- bottle hash
- 多平台兼容

### 8B.5 推荐实施步骤

推荐顺序：

1. 先确定 Homebrew 安装语义
   - 是装源码构建版
   - 还是装预编译版

2. 先做自己的 tap
   - 不建议一上来就试图进入 `homebrew/core`

3. 用 `brew create <URL>` 生成初版 formula

4. 手工补 `install` 与 `test do`

5. 本地执行：
   - `brew install --build-from-source`
   - `brew audit`

6. 稳定后再考虑：
   - bottle
   - CI 自动化
   - 是否进入更公共的分发入口

### 8B.6 对 Star Sanctuary 的额外开发工作

这是最容易低估的地方。

因为你当前安装器依赖：

- Node.js
- corepack
- pnpm workspace
- 本地构建

如果直接把当前 `install.sh` 逻辑塞进 Homebrew，通常会不够优雅。

更现实的额外工作可能包括：

- 梳理最小运行时依赖
- 调整安装后入口位置
- 让 formula 安装结果更符合 Homebrew 目录语义
- 确保升级 / 卸载不会留下太多项目自管目录残留

也就是说，`brew install star-sanctuary` 更像是在补一套“macOS / Linux 原生分发适配层”。

### 8B.7 当前判断

这一点现在已经明确：

```text
brew 暂不实现；
也不接受 “brew -> install.sh” 这种短期折中接法。
```

原因是：

- 这会让命令看起来像包管理器原生接入，但底层实际上还是独立安装脚本
- 后续升级、卸载、目录语义、依赖声明都还是会别扭
- 这条线一旦开始，最好就是按 Homebrew 自己的分发语义来做

因此当前顺序已经固定为：

```text
先把 winget 正式接入打通；
brew 后面单独按正统方案实现。
```

## 8D. 正式轻量资产方案

上面讨论 `winget` / `brew` / 官网下载时，需要特别区分两类发布物：

1. **小白运行包**
2. **正式轻量资产**

这两类东西不应该混为一谈。

### 8D.1 小白运行包不代表正式轻量资产体积

当前仓库里的：

- `portable`
- `single-exe`

本质上都是为 Windows 小白用户准备的“可直接运行包”。

它们的目标是：

- 用户不安装 Node.js
- 用户不安装 pnpm
- 双击即可运行
- 运行期还能做自恢复

因此它们会主动把运行环境一起带上，包括：

- Node runtime
- runtime 目录
- payload
- runtime manifest / 自恢复所需元数据

所以这类包天然会比较大。  
它们适合官网专门下载入口，不适合拿来代表“正式轻量 Release 资产”的大小。

### 8D.2 正式轻量资产应该是什么

如果参考 `openclaw` 的思路，真正适合挂到 GitHub Releases、安装器、后续包管理器消费的，不一定是“带运行环境的包”，而更应该是：

- 已构建好的业务产物
- 静态 Web 资源
- 必要模板
- 安装脚本
- 元数据与校验文件

但**不包含运行环境本体**，例如：

- 不带 Node runtime
- 不带 `node_modules`
- 不带 `portable` runtime 目录
- 不带 `single-exe` 的壳层与内嵌 payload

也就是说，这类正式轻量资产更接近：

```text
源码安装链路的“发布态输入”
```

而不是：

```text
给小白用户直接双击运行的完整环境包
```

### 8D.3 体积预估

按当前仓库状态做本地估算后，可以得到一个比较明确的区间：

1. 当前仓库直接 `npm pack --dry-run` 的包：
   - 压缩后约 `3.06 MB`
   - 解包后约 `13.47 MB`
2. 如果按“正式轻量资产”口径，只取：
   - `packages/*/dist`
   - `apps/web/public`
   - 模板
   - bundled skills
   - 安装脚本
   - `.env.example`
   - `pnpm-lock.yaml`
   - 并排除 `.map`、测试产物

则当前测得：

- 核心发布内容约 `8.58 MB`
- 再加模板 / 技能 / 安装脚本等约 `0.32 MB`
- 合计解包内容约 `8.9 MB`

因此可以把“正式轻量资产”的现实区间先判断为：

- **压缩包体积**：大概率约 `3 MB ~ 8 MB`
- **解包体积**：大概率约 `9 MB ~ 15 MB`

这个量级和当前 GitHub 自动生成的源码包是同一数量级，和 `portable` / `single-exe` 的几百 MB 级别不是一回事。

### 8D.4 正式轻量资产建议包含什么

建议首版正式轻量资产至少包含：

- `packages/*/dist`
- `apps/web/public`
- `packages/belldandy-agent/src/templates`
- `packages/belldandy-skills/src/bundled-skills`
- `.env.example`
- `install.ps1`
- `install.sh`
- `start.bat`
- `start.sh`
- `pnpm-lock.yaml`
- `pnpm-workspace.yaml`
- 根 `package.json`
- 必要的各 workspace `package.json`

建议默认不包含：

- `node_modules`
- `runtime/`
- `payload/`
- `artifacts/portable/**`
- `artifacts/single-exe/**`
- 各类测试文件
- `.map`
- 临时目录与实验目录

### 8D.5 如果要发布这类正式轻量资产，还差哪些工作

相比 `portable` / `single-exe`，这条线的工作量小很多，主要是：

1. 新增一个轻量打包脚本
   - 例如：`scripts/build-release-light-assets.mjs`
   - 负责收集发布文件、裁剪无关文件、输出统一目录
2. 固化命名规则
   - 例如：
     - `star-sanctuary-dist-v0.3.5.zip`
     - `star-sanctuary-dist-v0.3.5.tar.gz`
3. 生成校验文件
   - 例如：
     - `SHA256SUMS.txt`
     - `release-manifest.json`
4. 把 Release workflow 接上资产上传
   - 当前 GitHub Release 只创建 release，不上传自定义 assets
5. 增加最小发布验证
   - 校验压缩包结构
   - 校验 hash
   - 校验关键入口文件是否齐全

### 8D.6 它和当前安装流程的关系

这类正式轻量资产**不必马上改变当前安装流程**。

可以分两步走：

#### 第一步：先只作为 GitHub Release 正式附件

作用：

- 官网下载
- 后续给包管理器准备稳定输入
- 让 Release 页面不再只有源码归档

此时当前安装器仍然继续使用：

```text
latest release -> tag 源码归档 -> 本地安装 / 本地构建
```

#### 第二步：等轻量资产稳定后，再让安装器切过去

未来如果要进一步提速安装流程，可以再改成：

```text
latest release -> 正式轻量资产 -> 本地补依赖 / 安装
```

这样做的好处是：

- 第一阶段改动小
- 不会一下子改动安装语义
- 出问题时也容易回退

### 8D.7 推荐判断

如果当前目标是：

- 让 GitHub Releases 更像正式产品发布页
- 给官网下载准备更合理的正式附件
- 为以后 `winget` / `brew` 做准备

那么最值得先做的，不是把 `portable` 挂上去，而是先补一条：

```text
正式轻量资产发布线
```

也就是：

- 保留 `portable / single-exe` 作为“小白包”
- 另做一套不带运行环境的轻量 Release assets

这是当前最合理、改动最小、也最符合后续扩展方向的做法。

```text

```

- 轻量正式附件
- 给命令安装、开发者部署、后续包管理器使用
- 不等于小白双击运行包

建议官网下载区可以拆成两个分组：

1. **命令安装与开发者分发**
   - `star-sanctuary-dist-v<version>.zip`
   - `star-sanctuary-dist-v<version>.tar.gz`
   - `manifest.json`
   - `sha256`
2. **小白运行包**
   - `portable`
   - `single-exe`

官网文案口径建议如下：

##### 轻量正式附件

- 用于命令安装、开发者部署、后续包管理器接入
- 不包含 Node runtime
- 不包含 `node_modules`
- 体积更小，下载更快
- 不面向“小白双击即用”

##### 小白运行包

- 面向 Windows 小白用户
- 内置运行环境
- 下载后可直接运行
- 体积明显更大

#### 8D.8.13 与 `portable / single-exe` 的下载分工文案

为了避免用户混淆，官网最好把三类下载入口的定位写清楚：

##### A. 命令安装 / 开发者部署

适用人群：

- 会使用终端
- 准备按文档安装
- 需要后续脚本化部署
- 需要更适合包管理器的正式附件

推荐入口：

- 安装命令
- `star-sanctuary-dist-v<version>.zip`
- `star-sanctuary-dist-v<version>.tar.gz`

##### B. Windows 小白可运行版

适用人群：

- 不想安装 Node.js / pnpm
- 希望下载后尽快运行
- 更关注“能用”而不是“体积小”

推荐入口：

- `portable`
- `single-exe`

##### C. Release 页面附件定位

GitHub Release 页面后续建议形成如下语义：

- **轻量正式附件**：作为主附件，服务命令安装、开发者下载、方案 C 前置准备
- **小白运行包**：作为面向终端用户的专用下载物，继续走官网专门入口

这样分工以后，官网就不会出现“一个页面里全是下载按钮，但用户不知道该点哪个”的问题。

#### 8D.8.14 官网可直接使用的展示文案草案

可以先准备一版简洁口径：

```text
轻量正式附件
适合命令安装、开发者部署和后续包管理器接入使用。
```

## 9. 改命令会涉及哪些开发工作

### 9.1 如果只改“官网展示命令入口”

这是最轻量的方案。

开发工作通常只有：

1. 准备一个以 `StarSanctuary` 为主体的域名或子域名
2. 配置 `/win`、`/unix` 或 `/install.ps1`、`/install.sh` 路由
3. 让这些入口回源到现有 `install.ps1` / `install.sh`
4. 更新官网安装页文案
5. 最后再更新 README

这类工作量不大，本质是“增加一层稳定入口”。

### 9.2 如果要直接接入包管理器

开发工作就明显更大，通常会涉及：

1. 梳理正式 release 产物结构
2. 保证安装包 URL 稳定
3. 生成并维护 checksum
4. 编写 formula / manifest / package
5. 配 CI 自动发布
6. 做升级验证
7. 做多平台测试

这已经不是简单“换一条命令”，而是在建设一整套正式分发体系。

## 10. 改命令能不能不改现在的安装流程

**可以，而且这是当前最推荐的做法。**

推荐结构：

```text
新短命令
  -> 新项目名安装入口
    -> 继续指向现在的 install.ps1 / install.sh
      -> 继续安装 latest release 对应源码归档
```

也就是说：

- 用户看到的命令变短
- 安装逻辑仍然沿用当前脚本
- 后续如果要换真实脚本源，可以在入口层偷偷切换
- 用户命令不需要重新学习

所以当前阶段完全没必要为了改短命令，就重写安装器主流程。

## 11. 路由设计建议

建议为项目安装入口单独做一个以项目名为主体的子域名：

```text
install.starsanctuary.<your-domain>
```

推荐路由：

```text
GET /win
GET /unix
GET /install.ps1
GET /install.sh
GET /stable/win
GET /stable/unix
GET /nightly/win
GET /nightly/unix
```

最低实现只需要先做：

```text
/win
/unix
```



这样 GitHub Raw 仍然保留，但从“主入口”降级为“备用入口”。

## 13. 升级命令口径建议

如果以后安装器支持统一升级入口，官网可以延续同样思路：

Windows：

```powershell
irm https://install.starsanctuary.<your-domain>/win | iex
```

Linux / macOS：

```bash
curl -fsSL https://install.starsanctuary.<your-domain>/unix | bash
```

也就是说：

- 安装与升级共用同一入口
- 安装器内部自行判断“首次安装”还是“升级”

这会比再单独造一套 upgrade 命令更简单。

## 15. 回滚策略

如果短域名入口临时出问题：

1. 官网临时切回 GitHub Raw 备用命令
2. 短域名恢复后再切回

因此建议官网文档始终保留一份“备用安装入口”，但不放在首页主视觉区域。

## 16. 结论

结论很明确：

- 当前命令长，**很大一部分确实是 GitHub Raw 地址导致的**
- 但真正应该优化的不是仓库脚本本身，而是**用户入口层**
- 如果命令主体想体现品牌，优先应该体现 **StarSanctuary 项目名**，而不是官网品牌站
- `brew install star-sanctuary` 这类命令之所以短，是因为它把复杂度转移到了发布体系与持续维护上
- 现阶段最合适的方案不是立即做包管理器，而是：

```text
项目名安装入口 + 现有安装脚本回源
```

推荐最终官网展示命令：

```powershell
irm https://install.starsanctuary.<your-domain>/win | iex
```

```bash
curl -fsSL https://install.starsanctuary.<your-domain>/unix | bash
```

这套方案的优点是：

- 最小改动
- 观感更好
- 后续可扩展
- 不绑死 GitHub 仓库路径
- 不需要先改当前安装流程
