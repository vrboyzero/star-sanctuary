# Star Sanctuary 超长期任务系统实现方案

## 1. 目标

本文档定义 Star Sanctuary 面向“中大型项目 / 超长流程 / 多天至多周任务”的目标驱动执行系统。目标不是替代现有 `methods`、`memory`、`sub-agent`、`canvas`，而是把这些已实现能力组织成一套统一的长期任务操作系统。

这套系统需要解决四个核心问题：

1. 用户给出一个大目标后，系统能先把目标补全，而不是直接进入零散执行。
2. 系统能把目标沉淀为一整套规范化文档与可追踪状态，而不是只留在会话上下文里。
3. 主 Agent 在执行每个阶段时，必须围绕当前目标动态规划所需的方法、模组、子代理、工具、Skill、MCP，而不是被静态工具集限制。
4. 系统需要在超长周期下保持可恢复、可审计、可复盘、可沉淀。

---

## 2. 现状评估

Star Sanctuary 已经具备超长期任务系统的关键底座，但这些能力目前还是分散的：

- `methods/` 已支持 `method_list`、`method_search`、`method_read`、`method_create`，适合沉淀 SOP。
- `Task Memory` 已支持任务开始/结束捕获、工具调用记录、artifact 记录、memory links、任务摘要。
- `experience_candidates` 已支持从历史任务自动/手动生成 method/skill 候选并发布。
- `SubAgentOrchestrator` 已支持子 Agent 并发、排队、超时、深度限制与生命周期钩子。
- `canvas_*` 已支持画布式任务/知识可视化，并可绑定 method、memory、session、file。
- `Context Compaction`、`Auto Recall`、`Memory Viewer` 已解决长上下文和历史回溯问题。
- `Cron`、`Heartbeat`、`MCP`、`Skills`、`AgentRegistry` 已提供自动化执行和能力扩展基础。

当前缺口主要有六个：

1. 缺少“目标级容器”。现在有任务级记录，但没有长期目标级对象。
2. 缺少统一文档流水线。目标方案、设计方案、架构方案、拆分计划、Task List、`NORTHSTAR.md` 还没有统一生成与维护规范。
3. 缺少机器可执行的 DAG 任务图。现有 Task Memory 更像执行回执，不是长期计划引擎。
4. 缺少目标级追踪视图。当前任务、里程碑、阶段阻塞、审批点还未统一可视化。
5. 缺少“能力规划层”。主 Agent 还没有围绕节点目标显式规划 methods / skills / MCP / 子 Agent / 工具缺口。
6. 缺少长期任务状态机。现在已有 task capture，但没有“从目标初始化到阶段完成再到结项沉淀”的总流程。

结论：本项目不需要再新造一套平行系统，而应在现有 `memory + methodology + sub-agent + canvas + hooks` 基础上增建“目标层 + DAG 层 + 文档层 + 治理层”。

## 3. 设计原则

### 3.1 目标优先

所有规划、执行、调度、复盘，都必须围绕“当前目标是否更接近完成”判断，而不是围绕“当前有哪些工具”判断。

### 3.2 外部状态优先

超长期任务的关键状态不能只存在于模型上下文，必须外部化为文档、任务图、进度日志、画布、记忆记录。

### 3.3 复用优先

优先复用现有能力：

- SOP 复用 `methods/`
- 历史经验复用 `Task Memory` 与 `experience_candidates`
- 并行执行复用 `SubAgentOrchestrator`
- 可视化复用 `canvas`
- 长上下文稳定性复用 `Context Compaction`

### 3.4 双层表达

长期任务同时维护：

- 人类可读层：Markdown 文档、任务拆解清单、进度日志、`NORTHSTAR.md`
- 机器可执行层：`tasks.json`、goal state、checkpoint state、run metadata

### 3.5 动态能力规划

每个任务节点执行前，主 Agent 都要先判断：

- 现有方法是否可复用
- 现有 Skill / MCP / 工具是否足够
- 是否需要子 Agent
- 是否需要临时生成新方法、Skill 草稿、MCP 配置或执行模组

### 3.6 风险分级与 HiTL

涉及部署、数据库写入、敏感配置、破坏性文件操作、主版本依赖升级、外部发布等节点，必须进入审批断点。

## 4. 总体架构

建议将超长期任务系统拆成五层：

### 4.1 Goal Layer（目标层）

负责把用户需求转化为结构化目标对象，包含：

- 目标描述
- 边界与非目标
- 约束
- 成功标准
- 阶段划分
- 风险矩阵
- 审批点

### 4.2 Blueprint Layer（蓝图文档层）

负责输出和维护任务全周期文档：

- 目标方案
- 完整设计方案
- 项目架构与实现方案
- 实现方案拆分计划书
- 可追踪 Task List
- `NORTHSTAR.md`

### 4.3 DAG Execution Layer（执行图层）

负责把长期任务拆成机器可调度的 DAG 节点，并管理：

- 节点依赖
- 节点状态
- 节点负责人
- 能力规划
- 验收标准
- 回滚方案
- 里程碑推进

### 4.4 Memory & Experience Layer（记忆沉淀层）

复用现有 MemoryManager 体系，扩展为目标级闭环：

- Task Memory 记录每次节点执行结果
- methods 记录稳定流程
- experience candidates 记录可复用 method/skill 草稿
- memory chunks 保存项目知识、决策与复盘

### 4.5 Governance & Visualization Layer（治理与可视化层）

负责：

- 目标面板
- 任务图/阶段图
- 里程碑与阻塞展示
- 审批点管理
- 进度日志
- 画布联动

## 5. 标准文档体系

考虑到未来会同时存在多个不同的超长期任务，且这些任务不应绑定在当前项目仓库目录下，建议把“正式文档”和“运行态文件”统一收敛到状态目录 `~/.star_sanctuary/` 下，由状态目录承载多任务并存。

同时建议采用“默认路径 + 每个 goal 可单独指定路径”的混合模式：

- 默认情况下，长期任务运行态根目录放在 `~/.star_sanctuary/goals/<goal-id>/`
- 用户可为某个长期任务单独指定 `goalRoot`，例如放到 `E:\`、`D:\` 或某个项目自己的工作目录下
- 该路径既要写入 `NORTHSTAR.md` 供人类查看，也必须写入机器可读注册表供系统恢复

建议为每个长期目标建立一个独立工作目录：

```text
~/.star_sanctuary/docs/long-tasks/<goal-slug>/
├── 00-goal.md
├── 01-target-solution.md
├── 02-design.md
├── 03-architecture-and-implementation.md
├── 04-split-plan.md
├── 05-task-list.md
├── NORTHSTAR.md
├── tasks.json
├── progress.md
└── handoff.md
```

各文件职责如下：

- `00-goal.md`
  - 用户原始目标
  - 需求完善结果
  - 边界 / 非目标
  - 成功标准
  - 风险与约束

- `01-target-solution.md`
  - 最终目标方案
  - 方案选择理由
  - 备选方案与放弃原因

- `02-design.md`
  - 设计方案
  - 接口契约
  - 数据流
  - 验收口径

- `03-architecture-and-implementation.md`
  - 架构分层
  - 模块职责
  - 实现路径
  - 阶段目标

- `04-split-plan.md`
  - 类似“拆分计划书”
  - 每阶段实施步骤
  - 每阶段回归方案
  - 风险与回滚

- `05-task-list.md`
  - 人类可读任务清单
  - 与 `tasks.json` 一一映射

- `NORTHSTAR.md`
  - 本长期任务的局部宪法
  - 当前目标、约束、非目标、质量门槛、执行纪律
  - 当前 `goalRoot`、文档路径、运行态路径、artifact 路径等路径声明

- `tasks.json`
  - 机器可执行 DAG 任务图

- `progress.md`
  - 时序进度日志
  - 里程碑、阻塞、决策、下一步

- `handoff.md`
  - 中断恢复与交接摘要

建议同时新增长期任务注册表与目标级运行时目录：

```text
~/.star_sanctuary/goals/
├── index.json
└── <goal-id>/
```

其中：

- `index.json` 作为机器可读注册表，维护 `goalId -> goalRoot / northstarPath / status / lastActive` 等映射
- `<goal-id>/` 是默认运行态目录；若用户为某个 goal 指定了自定义路径，则注册表指向该路径

建议的默认运行时目录结构：

```text
~/.star_sanctuary/goals/<goal-id>/
├── state.json
├── checkpoints.json
├── runtime.json
├── board-ref.json
├── runs/
└── artifacts/
```

说明：

- `~/.star_sanctuary/docs/long-tasks/` 存放正式文档，但它属于状态目录而不是仓库目录。
- `~/.star_sanctuary/goals/` 是长期任务默认根目录，同时持有全局注册表 `index.json`。
- 默认情况下运行态、检查点、执行缓存和 artifact 映射都在 `~/.star_sanctuary/goals/<goal-id>/` 下。
- 若用户为某个长期任务指定了自定义 `goalRoot`，则该 goal 的实际运行态可迁移到外部路径，但注册表仍保留在 `~/.star_sanctuary/goals/index.json` 中。
- 一个 goal 一套独立目录，便于多个长期任务同时存在、暂停、恢复、归档。
- 仓库内只保留“系统实现方案”这类平台级文档，不默认承载具体长期任务实例文档。
- 若某个长期任务最终需要沉淀为项目交付物，可在结项时选择性导出到仓库，而不是默认直接写入仓库。
- `NORTHSTAR.md` 需要同步展示路径信息，但不应作为路径配置的唯一真源。

## 6. 核心数据模型

### 6.1 Goal

建议新增长期目标实体 `LongTermGoal`：

```json
{
  "id": "goal_app_modularization",
  "slug": "app-modularization",
  "title": "app.js 模块化重构",
  "status": "planning",
  "objective": "将 app.js 拆分为多模块并保持行为一致",
  "nonGoals": [
    "不引入新框架",
    "不改动既有协议"
  ],
  "constraints": [
    "保持 WebChat 可运行",
    "优先渐进式重构"
  ],
  "successCriteria": [
    "模块拆分完成",
    "关键回归通过"
  ],
  "currentPhase": "design",
  "goalRoot": "E:\\project\\sample\\.star_sanctuary-goals\\goal_app_modularization",
  "pathSource": "user-configured",
  "registryPath": "~/.star_sanctuary/goals/index.json",
  "docRoot": "~/.star_sanctuary/docs/long-tasks/app-modularization",
  "northstarPath": "~/.star_sanctuary/docs/long-tasks/app-modularization/NORTHSTAR.md",
  "tasksPath": "~/.star_sanctuary/docs/long-tasks/app-modularization/tasks.json",
  "progressPath": "~/.star_sanctuary/docs/long-tasks/app-modularization/progress.md",
  "runtimeRoot": "E:\\project\\sample\\.star_sanctuary-goals\\goal_app_modularization",
  "boardId": "goal_app_modularization_main",
  "createdAt": "2026-03-20T00:00:00.000Z",
  "updatedAt": "2026-03-20T00:00:00.000Z"
}
```

字段说明补充：

- `goalRoot`
  - 当前长期任务的实际根目录，允许是默认路径，也允许是用户指定的绝对路径
- `pathSource`
  - 标识该路径来自默认规则还是用户手动指定，例如 `default` / `user-configured`
- `registryPath`
  - 全局注册表位置，系统恢复任务时先查注册表，再打开对应 `NORTHSTAR.md`

建议新增目标级 DAG 节点实体：

```json
    "tools": ["file_read", "apply_patch"],

  "artifacts": [
    "apps/web/public/memory.js"
  ]

```

### 6.3 Run Record

建议新增节点执行记录 `GoalTaskRun`，与现有 `TaskRecord` 关联：

- `runId`
- `goalId`
- `nodeId`
- `conversationId`
- `agentId`
- `startedAt`
- `finishedAt`
- `result`
- `summary`
- `toolCalls`
- `memoryLinks`
- `methodUsages`
- `skillUsages`
- `artifacts`
- `approvalState`

落地策略：

1. 第一阶段可以先把 `goalId/nodeId/checkpointId` 放入现有 `tasks.metadata`。
2. 第二阶段再补独立表，避免一开始改动过大。

## 7. 目标驱动执行模板

### 7.1 Phase A：目标初始化

用户提出长期目标后，系统先进入“完善需求”模式，输出 `00-goal.md`。

必须补全的内容：

- 目标定义
- 业务背景
- 边界
- 非目标
- 约束
- 风险
- 成功标准
- 审批点

这一阶段不直接开始编码。

### 7.2 Phase B：蓝图与设计冻结

系统顺序生成：

1. `01-target-solution.md`
2. `02-design.md`
3. `03-architecture-and-implementation.md`
4. `04-split-plan.md`
5. `05-task-list.md`
6. `NORTHSTAR.md`
7. `tasks.json`

此阶段结束后，长期目标才算正式立项。

### 7.3 Phase C：任务图初始化

系统根据 `04-split-plan.md` 生成 `tasks.json` 和 Canvas 画布：

- Markdown 清单面向人审阅
- JSON DAG 面向 Agent 执行
- Canvas 面向结构理解与阶段协同

### 7.4 Phase D：节点执行循环

每个可执行节点都遵循固定循环：

1. 读取 `NORTHSTAR.md`
2. 读取当前节点与依赖节点结果
3. 检索相关 `methods`
4. 检索相关 memory / recent tasks / experience candidates
5. 生成本轮 `capabilityPlan`
6. 判断由主 Agent 执行还是拆给子 Agent
7. 执行
8. 回归验证
9. 更新 `tasks.json`、`progress.md`、Canvas、Task Memory
10. 判断是否需要沉淀 method / skill 草稿
11. 上下文压缩与 handoff 更新

### 7.5 Phase E：里程碑与结项

每个阶段结束时，系统要输出：

- 阶段完成摘要
- 已达成目标
- 剩余风险
- 未完成事项
- 新增/更新 methods
- 关键记忆沉淀

结项时输出最终复盘，并将高价值经验转成 methods / skills。

## 8. 长期任务专用通道与隔离

这是本方案必须补强的一点：超长期任务虽然复用现有工具与能力，但它不能直接混入普通对话任务通道，否则会带来状态污染、记忆串线、工具误用和体验混乱。

### 8.1 独立入口

建议新增“超长期任务专用入口”，与普通任务分开：

- 普通任务入口：继续沿用现有 chat/task 流程，适合即时问答、一次性修改、短任务。
- 长期任务入口：显式创建或恢复 `goal session`，进入目标级执行模式。

建议新增两类入口能力：

- `goal_init` / `goal_resume`
- Web 端 Goal Dashboard 中的“开始长期任务 / 恢复长期任务”

原则：

- 不是所有任务都进入长期任务系统。
- 只有明确立项的长期目标才进入 goal channel。
- 用户中途中断后，恢复时必须回到原 goal channel，而不是普通聊天通道继续。

### 8.2 独立会话命名空间

建议为长期任务引入独立会话命名空间：

- 普通会话：保持现有 `conversationId/sessionKey`
- 长期任务会话：新增 `goal:{goalId}` 命名空间
- 节点执行会话：新增 `goal:{goalId}:node:{nodeId}:run:{runId}`

这样做的作用：

- 目标级上下文与普通聊天隔离
- 同一长期任务的多个节点可独立运行、独立恢复
- 多个长期任务可并行存在，互不污染
- Task Memory、Recent Tasks、经验候选都可以按 `goalId` 精确过滤

### 8.3 工具复用但策略隔离

现有方法、技能、子代理、MCP、builtin tools 仍然复用，但在长期任务通道里需要增加一层“goal-aware 策略包装”。

具体要求：

- `methods`
  - 普通任务照常使用
  - 长期任务中优先读取与当前 `goalId`、当前阶段、当前节点相关的方法
  - 长期任务产生的新方法默认先记到 goal 相关上下文，再决定是否晋升为通用 method

- `skills`
  - 普通任务维持当前发现与调用逻辑
  - 长期任务需要记录“本节点实际使用了哪个 skill”，并纳入 goal run

- `sub-agent`
  - 普通任务仍可临时委托
  - 长期任务中的子 Agent 必须继承 `goalId/nodeId/runId`
  - 子 Agent 输出要挂回当前节点，而不是散落成普通独立任务

- `memory`
  - 普通任务继续使用全局记忆
  - 长期任务默认增加 `goalId` 过滤与关联，优先召回当前目标相关记忆、阶段决策、历史节点结果

- `canvas`
  - 普通画布继续独立使用
  - 长期任务自动绑定一个主板，作为目标级可视化工作区

结论：复用的是能力本身，隔离的是入口、上下文、状态归属、记录维度和调度策略。

### 8.4 暂停与恢复机制

长期任务的暂停是常态，不是异常。

因此必须支持：

- 用户手动暂停 goal
- 节点执行后自动暂停，等待下一次恢复
- 因 checkpoint 进入 `pending_approval`
- 因阻塞进入 `blocked`
- 因资源不足进入 `idle`

恢复时，系统应从以下资料恢复上下文，而不是依赖模型记忆：

- `NORTHSTAR.md`
- `tasks.json`
- `progress.md`
- 最新 checkpoint
- 最新 handoff
- 当前节点最近一次 run 记录
- goal board 摘要

### 8.5 与普通任务的硬性边界

需要明确以下边界：

1. 普通任务不得默认写入长期任务文档目录。
2. 普通任务不得默认占用长期任务的 `goal session`。
3. 长期任务专用工具不应污染普通任务提示词与调用路径。
4. 长期任务的 checkpoint、progress、task graph 更新，不应出现在普通任务执行路径里。
5. 长期任务需要显式 opt-in，而不是隐式触发。

因此系统实现上建议采用：

- 独立 goal RPC / tool 前缀
- 独立 goal sessionKey 命名空间
- 独立 goal prompt 注入协议
- 独立 goal runtime 目录
- 独立 goal dashboard 入口

## 9. 能力规划层设计

这是本系统最关键的新层。

### 9.1 为什么必须新增能力规划层

用户已经明确提出：执行过程中不要被已有的方法、模组、子代理、工具、Skill、MCP 限制，而要以当前目标为核心做资源规划。

因此每个节点都要显式生成 `capabilityPlan`，而不是隐式依赖模型临场发挥。

### 9.2 capabilityPlan 应包含的内容

- 当前节点目标
- 已知可复用 methods
- 已知可复用 skills
- 已知可用 builtin tools / MCP / plugins
- 是否需要子 Agent
- 是否需要新建临时方法或模组
- 是否存在能力缺口
- 缺口的获取策略

建议策略顺序：

1. 先搜现有 `methods`
2. 再搜 `skills`
3. 再看现有 builtin tools / MCP / plugins
4. 再决定是否调用子 Agent
5. 仍不足时，生成临时执行模板或新能力草稿
6. 若涉及高风险扩展，再进入 HiTL

### 9.3 动态扩展原则

若发现现有能力不足，系统允许：

- 新增 method 草稿
- 新增 skill 草稿
- 新增 MCP 配置草案
- 新增临时子 Agent 角色
- 新增专门的节点级执行模版

但要求：

- 先解决当前节点目标
- 再判断是否值得沉淀为长期资产
- 新资产必须可审计、可回滚、可复用

## 12. HiTL 断点策略

长期任务系统必须预置断点，而不是临场判断。

建议至少在以下场景自动创建 checkpoint：

- 目标冻结
- 设计冻结
- 架构冻结
- 执行阶段进入高风险节点
- 部署前
- 数据迁移前
- 大范围文件删除前
- 主版本依赖升级前
- 面向外部系统写入前
- 结项前

checkpoint 内容应包含：

- 将执行什么
- 影响范围
- 回滚方案
- 风险等级
- 验证方式
- 是否阻塞后续 DAG

## 13. 记忆与沉淀闭环

长期任务系统不应只“完成任务”，还要把过程变成长期资产。

### 13.1 任务层

每个节点执行后写入 Task Memory：

- objective
- toolCalls
- artifactPaths
- memoryLinks
- summary
- reflection

### 13.2 方法层

以下情况优先沉淀 method：

- 某类节点执行两次以上
- 跨多个 goal 复用
- 有清晰步骤和常见失败经验

### 13.3 Skill / MCP 层

以下情况优先沉淀 skill / MCP：

- 明显需要可复用的操作模板
- 明显需要新的外部能力接入
- 可被多个目标多次使用

### 13.4 Goal 结项层

结项时应自动生成：

- goal retrospective
- reusable methods list
- reusable skills candidates list
- unresolved risks list

## 15. MVP 范围建议

第一版不要一口气做完整自治平台，建议 MVP 只覆盖以下能力：

1. 目标初始化与标准文档生成
2. `tasks.json` + `05-task-list.md` 双轨任务图

4. goal channel / pause / resume
5. 基础 checkpoint
6. 节点级 `capabilityPlan`
7. Canvas 自动创建长期任务主板

这 7 项完成后，Star Sanctuary 就已经从“会做任务的 Agent”升级为“会经营长期目标的 Agent”。

## 16. 关键决策

### 16.1 不新造第二套记忆系统

长期任务系统必须建立在现有 MemoryManager 之上，而不是再做一套独立记忆库。

### 16.2 不让长期任务只存在于对话

长期目标必须有文档、任务图、进度日志、画布、checkpoint。

### 16.3 文档与运行态统一进入状态目录

长期任务文档与运行态都进入 `~/.star_sanctuary/`，其中正式实例文档进 `~/.star_sanctuary/docs/long-tasks/`，运行态进 `~/.star_sanctuary/goals/`。仓库仅保留平台级方案文档，不默认承载具体长期任务实例。

### 16.4 长期任务与普通任务分通道

长期任务必须使用独立入口、独立 session namespace、独立 prompt 注入和独立运行态目录；允许复用工具与能力，但不允许与普通任务共用同一执行通道。

### 16.5 路径配置以注册表为准，`NORTHSTAR.md` 同步展示

每个长期任务都可以使用默认路径，也可以使用用户指定路径；真实路径配置应以 `~/.star_sanctuary/goals/index.json` 等机器可读注册表为准，`NORTHSTAR.md` 负责同步展示，不能作为唯一真源。

### 16.6 先目标级编排，再追求完全自治

当前项目已经具备很多自治能力，真正的短板不是“不会调用工具”，而是“没有目标级操作系统”。

---

## 17. 最终方案结论

Star Sanctuary 的超长期任务系统应定义为：

> 一个建立在 `methods + task memory + sub-agent orchestration + canvas + compaction + checkpoint` 之上的、以长期目标为中心的外部化执行操作系统。

它的核心不是再发明一个新 Agent，而是让现有 Agent 具备以下新能力：

- 能立项
- 能写蓝图
- 能拆 DAG
- 能按节点做能力规划
- 能在超长周期中恢复执行
- 能在关键点等待人审
- 能在结束后沉淀方法与技能

这就是 Star Sanctuary 下一阶段最值得建设的长期任务底座。
