# Star Sanctuary 小白版能力边界说明

本文用于说明当前小白版发布体系中：

- `Portable` 与 `Single-Exe` 的边界
- `Slim` 与 `Full` 的边界
- 包内能力与外置扩展生态如何共同覆盖项目正式能力

本文口径以当前已完成的 `Windows x64` 发布形态为准。

## 1. 先说结论

当前小白版的设计目标，不是让一个安装包文件本身直接塞满项目所有能力。

更准确的目标是：

> 用一个可运行的小白版作为稳定底座，再通过配置、外置扩展目录、工坊下载内容和用户数据目录，把能力继续补齐到接近源码版的正式可用范围。

因此需要分开理解：

- `Portable / Single-Exe`：是交付形态差异
- `Slim / Full`：是运行时能力档位差异
- `skills / plugins / methods / 工坊 apps`：是运行后继续扩展能力的生态层

## 2. Portable 和 Single-Exe 的关系

`Portable` 和 `Single-Exe` 的主要差异，不是业务能力本身，而是交付和启动方式不同。

### Portable

特点：

- 解压目录后运行
- 运行时文件直接保存在包目录中
- 更容易看到 `runtime/`、`launcher/`、`payload/` 这些结构

### Single-Exe

特点：

- 双击单个 `exe` 启动
- 首次运行时会把运行时解包到本机缓存目录
- 对用户来说更接近“下载一个程序直接运行”

### 当前判断

在同一能力档位下：

- `Portable Slim` 和 `Single-Exe Slim` 的目标能力边界应视为一致
- `Portable Full` 和 `Single-Exe Full` 的目标能力边界也应视为一致

也就是说：

> `Portable / Single-Exe` 决定的是“怎么交付”，不是“能不能覆盖项目正式能力”。

## 3. Slim 和 Full 的关系

`Slim / Full` 才是当前真正的能力边界差异。

### 3.1 两者默认都包含的能力

当前 `Slim` 和 `Full` 都默认保留：

- Web 界面与本地启动链路
- 浏览器控制
- 网页抓取 / 网页提取
- SQLite memory
- 向量检索基础能力
- bundled skills
- workspace 模板
- `.env` / `.env.local` 配置读取
- 外置 `skills/`、`plugins/`、`methods/` 加载能力

当前始终包含的运行时依赖有：

- `better-sqlite3`
- `sqlite-vec`
- `puppeteer-core`
- `jsdom`
- `@mozilla/readability`
- `turndown`

这意味着：

> 即使是 `Slim`，也不是“阉割演示版”，而是保留了浏览器控制、网页抓取、基础记忆和扩展生态入口的正式用户版。

### 3.2 Full 额外包含的能力

`Full` 在 `Slim` 基础上额外包含：

- `fastembed`
- `node-pty`

这对应两类能力增强：

- 本地 embedding 相关能力
- 原生 PTY 终端能力

### 3.3 Slim 的边界

`Slim` 当前不是全量原生依赖版。

它的边界主要在这里：

- 不默认打包 `fastembed`
- 不默认打包 `node-pty`

所以 `Slim` 的定位是：

- 保留大多数正式用户的主路径能力
- 继续保留浏览器链路和网页能力
- 避免把更重的本地原生依赖全部塞进默认包

### 3.4 Full 的定位

`Full` 当前更适合作为“接近完整能力覆盖”的用户版底座。

更准确地说：

> 如果目标是让普通用户在不接触源码仓库的前提下，尽可能接近项目正式能力上限，应优先选择 `Full`。

## 4. 扩展生态如何补齐能力

小白版并不是一个封死的黑盒。

当前路线本身就保留了外置扩展生态，主要包括：

- `skills/`
- `plugins/`
- `methods/`
- 工坊 `apps` / 下载内容
- 用户自己的 state dir 数据

这部分很关键，因为项目的很多正式能力，本来就不是依靠“主程序文件里一次性打满”来实现的，而是依靠：

- 主运行时
- 外置配置
- 外置扩展
- 外置数据

一起组成。

### 4.1 这意味着什么

意味着用户实际使用时，能力覆盖不是只看“下载包里已经放了什么”，还要看：

- 是否已经配置模型与 API Key
- 是否已经放入需要的 `skills/` / `plugins/`
- 是否已经通过工坊下载对应应用或资源
- 是否已经积累了用户自己的 workspace / memory / state 数据

所以更准确的说法应是：

> 小白版依靠“包内运行时 + 外置扩展生态”共同实现高能力覆盖，而不是靠单一安装包文件本体直接囊括一切。

## 5. 能力覆盖应该怎么理解

建议分成三层理解。

### 5.1 第一层：包内即开即用能力

这部分是用户下载后就应具备的基础能力：

- 启动应用
- 打开本地 Web 界面
- 使用浏览器控制
- 使用网页抓取 / 网页提取
- 使用 SQLite memory
- 读取 `.env` 配置
- 使用 bundled skills

`Slim` 和 `Full` 都覆盖这一层。

### 5.2 第二层：增强型本地能力

这部分依赖更重的原生运行时：

- 本地 embedding
- 原生 PTY 终端能力

这层当前应由 `Full` 覆盖为主。

### 5.3 第三层：生态扩展能力

这部分依赖运行后的持续扩展，不是一个安装包本体一次性决定的：

- 用户安装或放入新的 `skills`
- 用户安装或放入新的 `plugins`
- 用户沉淀自己的 `methods`
- 用户下载工坊 `apps` 或其他内容
- 用户积累自己的 state / workspace / logs / memory 数据

这层能力不是 `Slim / Full` 二选一能完全决定的，而是：

> 小白包提供底座，生态层继续把能力扩展到更完整的实际使用面。

## 6. 现在能不能覆盖项目“全能力”

如果这里的“全能力”指的是：

- 普通用户运行产品时会接触到的大多数正式能力
- 浏览器、网页、记忆、配置、扩展生态这些主路径能力
- 通过后续下载和扩展继续补齐更多能力

那么结论是：

**可以，路线成立。**

如果这里的“全能力”指的是：

- 仓库开发态能力
- `pnpm install / build / test`
- 源码调试
- 开发脚本
- 用 TypeScript 源文件直接开发和热更新整个仓库

那么结论是：

**不可以，这不是小白版的目标。**

因此更准确的判断是：

- `Slim`：覆盖核心正式能力，但不是最完整底座
- `Full`：更接近正式能力全覆盖的用户版底座
- 小白版整体：通过“运行时底座 + 外置扩展生态”实现高兼容和高能力覆盖

## 7. 当前推荐口径

对外说明时，建议统一使用下面这组口径。

### 7.1 对 Portable / Single-Exe 的口径

- 两者主要区别是交付方式，不是业务能力路线
- 用户按自己偏好的安装体验选择即可

### 7.2 对 Slim 的口径

- 面向大多数普通用户的默认版
- 已覆盖浏览器控制、网页抓取、基础记忆和扩展生态入口
- 不是阉割版
- 但不默认附带更重的本地原生依赖

### 7.3 对 Full 的口径

- 面向需要更完整本地能力的增强版
- 适合作为“接近完整能力覆盖”的用户版底座

### 7.4 对扩展生态的口径

- 项目正式能力并不只由安装包本体决定
- 小白版支持通过 `skills / plugins / methods / 工坊下载内容` 继续扩展
- 最终能力覆盖应理解为“包内能力 + 运行后扩展能力”的总和

## 8. 一句话版

一句话收口就是：

> `Portable / Single-Exe` 决定怎么交付，`Slim / Full` 决定能力档位，而项目正式能力的完整覆盖依赖“运行时底座 + 外置扩展生态”共同完成；其中 `Full` 更接近完整能力覆盖的用户版底座。
