# 方法与技能经验总结流程与生成操作说明

## 1. 先给结论

当前 `记忆查看 -> 经验消费总览` 里展示的内容，**不是 method / skill 的“生成器”本身**，而是：

- 已被实际消费过的 method / skill 的**使用统计总览**
- 从 usage 反查到来源 candidate、最近使用 task、已发布产物的**回溯入口**
- skill 的 freshness / stale / patch 风险信号的**观察面板**

也就是说，这个模块本身偏“看板 / 审计 / 追溯”，**不是操作台**。

当前仓库里，method 和 skill 的经验总结相关实现确实已经存在，而且链路是完整的：

1. task 完成后，系统可自动或按需把 task 提升为 `method` / `skill` 经验候选
2. 候选会保存到 experience candidate 层，状态初始为 `draft`
3. 审核通过后：
   - method candidate 发布到 `stateDir/methods/*.md`
   - skill candidate 发布到 `stateDir/skills/<dir>/SKILL.md`
4. 发布后的 method / skill 被再次使用时，会回写 usage，形成现在 UI 里的“经验消费总览”

但当前 UI 的“记忆查看 / 经验消费总览”里，**没有直接暴露“生成 / 接受 / 拒绝 / patch 正式 skill”按钮**。你现在看不到这类按钮，是因为前端这块确实只做了查看、跳转、撤销 usage，没有做生成/审核发布动作。

真正统计逻辑在：

### 3.2 skill usage

skill 被真实采用时，也会记录 usage。

`skills_search` 只负责搜索，不代表已经采用 skill。

- `packages/belldandy-skills/src/builtin/skills-tool.ts:198-200`

这说明“经验消费总览”统计的是**真实采用**，不是“搜过 / 看过”。

## 4. method / skill 经验总结是怎么生成的

### 4.1 task -> experience candidate

核心生成器在：

- `packages/belldandy-memory/src/experience-promoter.ts`

统一入口：

- `ExperiencePromoter.promoteTask(taskId, type)`
  - `packages/belldandy-memory/src/experience-promoter.ts:14-50`

它会从 task 中抽取：

- task 基本信息
- summary / reflection / outcome / objective
- toolCalls
- artifactPaths
- memoryLinks

然后生成：

- `method` 类型候选：`buildMethodDraft(...)`
  - `packages/belldandy-memory/src/experience-promoter.ts:101-161`
- `skill` 类型候选：`buildSkillDraft(...)`
  - `packages/belldandy-memory/src/experience-promoter.ts:182-223`

生成结果会写入 `experience_candidates`，状态是：

- `draft`

相关类型定义在：

- `packages/belldandy-memory/src/experience-types.ts:11-59`

### 4.2 自动生成

task 完成后，默认会尝试自动生成 method / skill candidate。

入口：

- `MemoryManager.completeTaskCapture(...)`
  - `packages/belldandy-memory/src/manager.ts:887-915`

默认开关：

- `BELLDANDY_EXPERIENCE_AUTO_PROMOTION_ENABLED`
- `BELLDANDY_EXPERIENCE_AUTO_METHOD_ENABLED`
- `BELLDANDY_EXPERIENCE_AUTO_SKILL_ENABLED`
- `packages/belldandy-core/src/bin/gateway.ts:2250-2252`

自动生成的前提：

- task 状态是 `success` 或 `partial`
- 且至少有 summary / reflection / toolCalls / artifactPaths / objective 中的一部分信号
- `packages/belldandy-memory/src/manager.ts:2036-2046`

### 4.3 post-run learning review 也会参与

Gateway 在 task 完成后，还会额外跑一次 `runPostTaskLearningReview(...)`：

- `packages/belldandy-core/src/bin/gateway.ts:2620-2633`
- `packages/belldandy-core/src/learning-review-runner.ts:81-143`

它会根据 task signal 判断是否值得生成：

- method candidate
- skill candidate

但注意：这个流程本质上还是**生成 candidate**，不是直接改正式 method / skill。

## 6. 这些结果能不能用来生成 method / skill

答案是：**可以，但不是“经验消费总览”直接生成，而是通过 candidate -> accept -> publish 这条链。**

### 6.1 method 的生成 / 发布

当 `method` candidate 被接受时：

- `acceptExperienceCandidate(...)`
  - `packages/belldandy-memory/src/manager.ts:1203-1218`
- 会调用 `publishMethodCandidate(...)`
  - `packages/belldandy-memory/src/manager.ts:2049-2055`

最终写入：

- `stateDir/methods/*.md`
  - 具体路径解析：`packages/belldandy-memory/src/manager.ts:2058-2079`

### 6.2 skill 的生成 / 发布

当 `skill` candidate 被接受时：

- 先调用 `publishSkillCandidate(...)`
  - `packages/belldandy-skills/src/skill-publisher.ts:10-27`
- 然后再把 candidate 标记为 accepted
  - `packages/belldandy-core/src/server-methods/memory-experience.ts:505-533`
  - `packages/belldandy-skills/src/builtin/memory.ts:1200-1269`

最终写入：

- `stateDir/skills/<safe-dir>/SKILL.md`
  - 路径规则：`packages/belldandy-skills/src/skill-publisher.ts:29-50`

并且会刷新 `SkillRegistry`：

- `packages/belldandy-skills/src/skill-publisher.ts:22-24`

发布后，该 skill 会被：

- `skills_search`
- `skill_get`

发现和读取。

## 7. 这些结果能不能用来“改进” method / skill

答案是：**可以作为改进输入，但当前实现更偏“生成新的 candidate 让人审”，不是直接在线 patch 正式 method / skill”。**

需要分开看：

### 7.1 method

当前 method 这条线主要是：

- 从 task 生成 method candidate
- 审核后发布为新的 method 文档

我没有在当前代码里看到“针对已发布 method 做自动 patch / merge / inline edit”的专门链路。

也就是说，method 的“改进”目前更像：

- 再生成一个新的 method candidate
- 人工审阅后决定是否替换、覆盖或另发一份 method

### 7.2 skill

skill 这条线比 method 多了一层“是否需要改进”的诊断信号，即 `skill freshness`。

关键实现：

- `packages/belldandy-core/src/skill-freshness.ts`
- `packages/belldandy-core/src/server-methods/memory-experience.ts:646-684`

skill freshness 会给出：

- `healthy`
- `warn_stale`
- `needs_patch`
- `needs_new_skill`

其中：

- `needs_patch` 表示：已有 accepted skill，但出现了待审 skill candidate，推断它可能是现有 skill 的补丁入口
- `needs_new_skill` 表示：出现了待审 candidate，但找不到可对应的现有 accepted skill，说明更像要新增 skill

对应代码：

- `packages/belldandy-core/src/skill-freshness.ts:325-359`
- `packages/belldandy-core/src/skill-freshness.ts:424-451`

但是这里仍然只是：

- 风险信号
- 审阅建议
- candidate 入口

不是直接 patch 正式 skill 文件的自动操作。

### 7.3 当前“改进”的真实含义

现在系统里“改进 skill”的实际落地方式是：

1. 使用信号 / freshness 信号发现 skill 可能过期或缺口
2. 生成新的 skill candidate
3. 人工审阅 candidate
4. 接受后重新发布成 skill

所以它更像“**candidate 驱动的更新**”，不是“**在线编辑正式 skill**”。

## 10. 最后总结

### 10.1 现在已经有的

- method / skill 使用回写与统计
- task -> method / skill candidate 生成
- 自动生成与 post-run learning review 生成
- candidate 查看
- candidate accept / reject
- accept 后发布到正式 methods / skills
- skill freshness 对“是否需要补丁 / 是否需要新增 skill”的诊断

### 10.2 当前“经验消费总览”的定位

它现在是：

- 使用统计看板
- 溯源入口
- freshness 风险提示入口

不是：

- 生成器
- 审核台
- 发布台
- 正式 skill / method 的在线编辑器

### 10.3 对你问题的直接回答

- `记忆查看 -> 经验消费总览` 背后，确实已经有 method 和 skill 的经验总结相关实现。
- 这些实现的主流程是：`task -> candidate(draft) -> accept/reject -> publish -> reuse -> usage stats`
- 最终结果可以用来生成 method 和 skill；skill 也可以借 freshness + pending candidate 信号来驱动“改进”
- 但当前“改进”不是直接 patch 正式 method / skill，而是通过新的 candidate 进入人工审阅和再发布
- 你在 UI 上看不到生成/改进按钮，是因为前端当前没有接这部分操作，只接了查看、跳转和 revoke usage

## 11. 如果后续要补 UI，最小可行入口建议

如果后续要补这块 UI，最小可行闭环建议是：

1. 在 task detail 增加：
   - `生成 method candidate`
   - `生成 skill candidate`
2. 在 candidate panel 增加：
   - `接受并发布`
   - `拒绝`
3. 在 skill freshness 卡片增加：
   - `打开 patch candidate`

## 12. 新增问题分析与实现方案

本节对应新增诉求：

1. method / skill 生成结果要更规范
2. 要有环境变量开关控制“生成/发布是否需要用户确认”
3. 要有系统级去重，且调用设置里补 method 列表
4. 暂不强化“改进”机制本身，沿用当前流程
5. 落地上一节给出的最小可行 UI 方案
6. 给出更稳妥的工程建议

### 12.1 问题分析

#### 问题一：当前 method / skill 自动生成内容不规范

现状：

- method draft 生成逻辑在 `packages/belldandy-memory/src/experience-promoter.ts:101-161`
- skill draft 生成逻辑在 `packages/belldandy-memory/src/experience-promoter.ts:182-223`

当前问题：

- method draft 结构过轻，离 `C:\Users\admin\.star_sanctuary\methods\方法论编写-撰写-指南.md` 要求的章节体系有明显差距
- skill draft 更像“任务摘要转写”，没有充分对齐 `C:\Users\admin\.star_sanctuary\skills\skill-creator\SKILL.md` 中强调的：
  - frontmatter 规范
  - description 的 WHAT / WHEN
  - 渐进式披露
  - 不把 skill 写成教程

结论：

- 需要改的是**生成模板 / 生成规则**
- 不需要改的是 candidate -> publish 主链路

#### 问题二：当前缺少可配置的确认开关

现状：

- memory 工具已经有 `needsPermission` / `riskLevel` 契约
- `experience_candidate_accept` 已经是 `riskLevel: "high"`
- 相关代码在 `packages/belldandy-skills/src/builtin/memory.ts`

当前问题：

- `task_promote_method`
- `task_promote_skill_draft`
- `experience_candidate_accept`

这些动作当前没有独立的环境变量控制是否需要用户确认。

结论：

- 不需要新造审批体系
- 应复用现有 tool control / confirm 机制
- 只需要把“是否确认”做成运行时可配置策略

#### 问题三：当前只有“同 task 去重”，没有系统级去重

现状：

- 当前 candidate 去重只做到 `taskId + type`
- 代码在 `packages/belldandy-memory/src/experience-promoter.ts:14-18`

当前问题：

- 同一类经验可能已经存在 method / skill 正式资产
- 也可能已经存在相似 candidate
- 现在不会在系统级别阻止重复沉淀

结论：

- 需要增加 system-level dedup
- 但不建议一上来就做 embedding 相似度
- 第一阶段用“规则去重 + 相似提示”更稳

#### 问题四：调用设置缺 method 列表，但底层能力其实已经有

现状：

- 系统已有 `method_list`
  - `packages/belldandy-skills/src/builtin/methodology/list.ts`
- 调用设置窗口数据来自 `tools.list`
  - `packages/belldandy-core/src/query-runtime-tools.ts`
- 目前只返回：
  - `builtin`
  - `mcp`
  - `plugins`
  - `skills`

当前问题：

- `index.html` 和 `tool-settings.js` 只渲染了 `技能`，没有 `方法`
- method 不在调用设置面板里可见

结论：

- 这是前端 / query runtime 视图层缺口
- 不是底层没有 method inventory

#### 问题五：改进流程先不扩张

这个判断是合理的。

当前 skill 已经有：

- `warn_stale`
- `needs_patch`
- `needs_new_skill`

相关代码：

- `packages/belldandy-core/src/skill-freshness.ts`

因此现阶段不应再引入“直接改正式 skill / method”这类更激进机制，而应继续坚持：

- 发现问题
- 生成 candidate
- 审阅
- 发布

这条治理链。

### 12.2 总体实现方案

#### 方案 A：规范生成模板

目标：

- method draft 对齐“方法论编写指南”
- skill draft 对齐 `skill-creator` 的 authoring 规范

method 生成规则调整：

- 文件名优先生成中文安全名
- 内容固定包含：
  - 标题
  - 一句话核心价值
  - `0. 元信息`
  - `1. 触发条件`
  - `2. 适用场景`
  - `3. 执行步骤`
  - `4. 工具选择`
  - `5. 失败经验`
  - `6. 成功案例`
  - `7. 相关资源`
  - `8. 更新记录`

skill 生成规则调整：

- frontmatter 必须稳定输出：
  - `name`
  - `description`
  - `version`
  - `tags`
  - `priority`
- `description` 必须同时回答：
  - 它做什么
  - 什么时候用
- body 优先输出：
  - 快速开始
  - 决策树
  - references / scripts 何时加载
  - 反模式

建议：

- 不要把参考文件内容原样拼进模板
- 应抽成内部 rubric，便于维护和测试

#### 方案 B：增加确认开关

建议新增环境变量：

- `BELLDANDY_METHOD_GENERATION_CONFIRM_REQUIRED`
- `BELLDANDY_SKILL_GENERATION_CONFIRM_REQUIRED`
- `BELLDANDY_METHOD_PUBLISH_CONFIRM_REQUIRED`
- `BELLDANDY_SKILL_PUBLISH_CONFIRM_REQUIRED`

建议默认值：

- generation：`false`
- publish：`true`

原因：

- 生成 candidate 可回滚，风险低
- 发布正式资产影响更大，更适合默认确认

实现方式：

- 继续复用现有工具确认流
- 不新增独立 HITL 通道

#### 方案 C：增加系统级去重

建议三层去重：

1. 同 task 去重
2. 正式资产精确去重
3. 相似资产提示

method 去重输入：

- 文件名
- 标题
- summary
- slug

skill 去重输入：

- skill name
- published dir name
- title
- slug
- description

建议行为：

- 精确重复：默认不生成新 candidate
- 相似重复：给出相似项提示，但允许继续生成

建议不要做的事：

- 第一阶段不上 embedding / 向量召回
- 第一阶段不做“强制相似阻断”

#### 方案 D：调用设置补 method 列表

建议：

- 在 `tools.list` payload 中增加 `methods`
- 在调用设置里增加 `方法` tab

method tab 的定位：

- 列表展示
- 打开文件
- 查看摘要 / 标题 / 状态

不建议：

- 给 method 做启用/禁用 toggle

原因：

- skill 是可注入能力，适合 on/off
- method 是知识资产，更适合作为索引视图

#### 方案 E：最小可行 UI 落地

按上一节建议，落地以下入口：

Task Detail：

- `生成 method candidate`
- `生成 skill candidate`

Candidate Panel：

- `接受并发布`
- `拒绝`

Skill Freshness：

- `打开 patch candidate`
- `标记 stale / 取消 stale`

不做：

- 在 usage overview 直接审批
- 直接编辑正式 method / skill

### 12.3 更稳妥的工程建议

#### 建议一：method tab 不要做成“技能 tab 的复制品”

method 在系统里不是 toggle-able capability，而是 SOP 资产。

因此更合适的呈现方式是：

- 查看列表
- 打开文件
- 按标题 / 摘要 / 文件名搜索

而不是：

- 开关启用

#### 建议二：去重结果要区分“阻断”和“提示”

推荐：

- exact duplicate -> 阻断
- similar duplicate -> 提示

否则会导致：

- 有价值的 v2 / patch candidate 也被粗暴拦下

#### 建议三：现阶段不要直接 patch 正式文件

因为这会破坏当前系统已经成型的治理边界：

- candidate layer
- review / accept
- publish

短期内最稳的仍是 candidate 驱动更新。

#### 建议四：后续可单独补一个 dedup preview RPC

例如：

- `experience.candidate.check_duplicate`

这样 UI 在点击“生成”前可以先预览：

- 是否已存在
- 有哪些相似 method / skill

但这不是第一阶段必须项。

## 20. 2026-04-20 命名规则与发布前校验收紧

本轮目标：

- 把 method 文件名、skill 目录名从“安全化”收紧到“明确规则化”
- 把发布前校验落到真正的 publish 链路，而不是只靠 draft 生成阶段自觉

### 20.1 现在的命名规则

method 文件名：

- 新生成 / 新创建 / 新发布的 method 统一使用三段式：
  - `[目标对象]-[动作类型]-[细分].md`
- 中文名和英文名都必须遵守三段式；不再接受“任意中文安全文件名”
- candidate 生成 slug、candidate accept 时的正式发布文件名、`method_create` 的文件名校验，已统一到这条规则
- 自动生成时的实际策略：
  - 中文任务优先生成中文三段式
  - 英文任务优先生成英文三段式
  - 若标题 / slug 已经是合法三段式，则直接沿用
  - 若不是合法三段式，则基于 title / objective / summary / taskId 做启发式规范化
- 发布重名时，不再通过额外增加新的 `-segment` 打破三段式，而是把修订后缀压入第三段，例如：
  - `候选层-实现-闭环_task_exp_1.md`

skill 文件夹名：

- 目录名使用 canonical skill machine name
- 规则为：
  - 小写
  - ASCII
  - kebab-case
  - 不含空格
  - 不含冗余 `skill-` 前缀
- 目录名与 frontmatter `name` 强制对齐
- 若草稿里的 `name` 不规范，发布时会改写成 canonical name 后再写入 `SKILL.md`

### 20.2 现在的发布前校验

method publish 前校验：

- 必须存在一级标题
- 必须包含以下固定章节：
  - `0. 元信息`
  - `1. 触发条件`
  - `2. 适用场景`
  - `3. 执行步骤`
  - `4. 工具选择`
  - `5. 失败经验`
  - `6. 成功案例`
  - `7. 相关资源`
  - `8. 更新记录`

skill publish 前校验：

- frontmatter 必须至少有：
  - `name`
  - `description`
- 必须存在一级标题
- 必须包含以下固定章节：
  - `快速开始`
  - `决策路由`
  - `输入`
  - `输出`
  - `参考指引`
  - `NEVER`

### 20.5 当前决策

本轮决定：

- 暂不继续做“历史资产扫描 + 命名迁移建议清单”

原因：

- 该项不影响当前主流程可用性
- 当前新生成 / 新发布链路已经使用新规则
- 旧资产治理属于存量一致性问题，可以在后续 webchat 优化或资产治理阶段统一处理

对当前功能流程的影响：

- 不影响：
  - candidate 生成
  - 生成前去重预检
  - candidate accept / reject
  - method 发布
  - skill 发布
- 仍然存在的差异：
  - 旧 method 文件名可能不符合当前“三段式强规则”
  - 旧 skill 目录名 / frontmatter `name` 可能不符合当前 canonical machine name 规则
  - 因此历史资产列表里可能会同时出现“旧规则命名”和“新规则命名”

结论：

- `defer`
- 可以继续按当前实现推进和使用，不阻塞主链路
