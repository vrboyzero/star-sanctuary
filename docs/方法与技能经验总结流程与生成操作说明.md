# 方法与技能经验总结流程与生成操作说明

## 1. 先给结论

当前 `记忆查看 -> 经验消费总览` 里展示的内容，**不是 method / skill 的“生成器”本身**，而是：

- 已被实际消费过的 method / skill 的**使用统计总览**
- 从 usage 反查到来源 candidate、最近使用 task、已发布产物的**回溯入口**
- skill 的 freshness / stale / patch 风险信号的**观察面板**

也就是说，这个模块本身偏“看板 / 审计 / 追溯”，**不是操作台**。

当前仓库里，method 和 skill 的经验总结相关实现确实已经存在，而且链路是完整的：

1. task 完成后，系统可自动或按需把 task 提升为 `method` / `skill` 经验候选
2. 候选会保存到 experience candidate 层，状态初始为 `draft`
3. 审核通过后：
   - method candidate 发布到 `stateDir/methods/*.md`
   - skill candidate 发布到 `stateDir/skills/<dir>/SKILL.md`
4. 发布后的 method / skill 被再次使用时，会回写 usage，形成现在 UI 里的“经验消费总览”

但当前 UI 的“记忆查看 / 经验消费总览”里，**没有直接暴露“生成 / 接受 / 拒绝 / patch 正式 skill”按钮**。你现在看不到这类按钮，是因为前端这块确实只做了查看、跳转、撤销 usage，没有做生成/审核发布动作。

### 3.2 skill usage

`skills_search` 只负责搜索，不代表已经采用 skill。

## 4. method / skill 经验总结是怎么生成的

### 4.1 task -> experience candidate

核心生成器在：

- `packages/belldandy-memory/src/experience-promoter.ts`

统一入口：

- `ExperiencePromoter.promoteTask(taskId, type)`
  - `packages/belldandy-memory/src/experience-promoter.ts:14-50`

它会从 task 中抽取：

- task 基本信息
- summary / reflection / outcome / objective
- toolCalls
- artifactPaths
- memoryLinks

然后生成：

- `method` 类型候选：`buildMethodDraft(...)`
  - `packages/belldandy-memory/src/experience-promoter.ts:101-161`
- `skill` 类型候选：`buildSkillDraft(...)`
  - `packages/belldandy-memory/src/experience-promoter.ts:182-223`

生成结果会写入 `experience_candidates`，状态是：

- `draft`

相关类型定义在：

- `packages/belldandy-memory/src/experience-types.ts:11-59`

### 4.2 自动生成

task 完成后，默认会尝试自动生成 method / skill candidate。

入口：

- `MemoryManager.completeTaskCapture(...)`
  - `packages/belldandy-memory/src/manager.ts:887-915`

默认开关：

- `BELLDANDY_EXPERIENCE_AUTO_PROMOTION_ENABLED`
- `BELLDANDY_EXPERIENCE_AUTO_METHOD_ENABLED`
- `BELLDANDY_EXPERIENCE_AUTO_SKILL_ENABLED`
- `packages/belldandy-core/src/bin/gateway.ts:2250-2252`

自动生成的前提：

- task 状态是 `success` 或 `partial`
- 且至少有 summary / reflection / toolCalls / artifactPaths / objective 中的一部分信号
- `packages/belldandy-memory/src/manager.ts:2036-2046`

### 4.3 post-run learning review 也会参与

Gateway 在 task 完成后，还会额外跑一次 `runPostTaskLearningReview(...)`：

- `packages/belldandy-core/src/bin/gateway.ts:2620-2633`
- `packages/belldandy-core/src/learning-review-runner.ts:81-143`

它会根据 task signal 判断是否值得生成：

- method candidate
- skill candidate

但注意：这个流程本质上还是**生成 candidate**，不是直接改正式 method / skill。

## 6. 这些结果能不能用来生成 method / skill

答案是：**可以，但不是“经验消费总览”直接生成，而是通过 candidate -> accept -> publish 这条链。**

### 6.1 method 的生成 / 发布

当 `method` candidate 被接受时：

- `acceptExperienceCandidate(...)`
  - `packages/belldandy-memory/src/manager.ts:1203-1218`
- 会调用 `publishMethodCandidate(...)`
  - `packages/belldandy-memory/src/manager.ts:2049-2055`

最终写入：

- `stateDir/methods/*.md`
  - 具体路径解析：`packages/belldandy-memory/src/manager.ts:2058-2079`

### 6.2 skill 的生成 / 发布

当 `skill` candidate 被接受时：

- 先调用 `publishSkillCandidate(...)`
  - `packages/belldandy-skills/src/skill-publisher.ts:10-27`
- 然后再把 candidate 标记为 accepted
  - `packages/belldandy-core/src/server-methods/memory-experience.ts:505-533`
  - `packages/belldandy-skills/src/builtin/memory.ts:1200-1269`

最终写入：

- `stateDir/skills/<safe-dir>/SKILL.md`
  - 路径规则：`packages/belldandy-skills/src/skill-publisher.ts:29-50`

并且会刷新 `SkillRegistry`：

- `packages/belldandy-skills/src/skill-publisher.ts:22-24`

发布后，该 skill 会被：

- `skills_search`
- `skill_get`

发现和读取。

## 7. 这些结果能不能用来“改进” method / skill

答案是：**可以作为改进输入，但当前实现更偏“生成新的 candidate 让人审”，不是直接在线 patch 正式 method / skill”。**

需要分开看：

### 7.1 method

当前 method 这条线主要是：

- 从 task 生成 method candidate
- 审核后发布为新的 method 文档

我没有在当前代码里看到“针对已发布 method 做自动 patch / merge / inline edit”的专门链路。

也就是说，method 的“改进”目前更像：

- 再生成一个新的 method candidate
- 人工审阅后决定是否替换、覆盖或另发一份 method

### 7.2 skill

skill 这条线比 method 多了一层“是否需要改进”的诊断信号，即 `skill freshness`。

关键实现：

- `packages/belldandy-core/src/skill-freshness.ts`
- `packages/belldandy-core/src/server-methods/memory-experience.ts:646-684`

skill freshness 会给出：

- `healthy`
- `warn_stale`
- `needs_patch`
- `needs_new_skill`

其中：

- `needs_patch` 表示：已有 accepted skill，但出现了待审 skill candidate，推断它可能是现有 skill 的补丁入口
- `needs_new_skill` 表示：出现了待审 candidate，但找不到可对应的现有 accepted skill，说明更像要新增 skill

对应代码：

- `packages/belldandy-core/src/skill-freshness.ts:325-359`
- `packages/belldandy-core/src/skill-freshness.ts:424-451`

但是这里仍然只是：

- 风险信号
- 审阅建议
- candidate 入口

不是直接 patch 正式 skill 文件的自动操作。

### 7.3 当前“改进”的真实含义

现在系统里“改进 skill”的实际落地方式是：

1. 使用信号 / freshness 信号发现 skill 可能过期或缺口
2. 生成新的 skill candidate
3. 人工审阅 candidate
4. 接受后重新发布成 skill

所以它更像“**candidate 驱动的更新**”，不是“**在线编辑正式 skill**”。

## 9. 为什么在当前 UI 里仍然很难找到生成 / 发布入口

这一节需要基于 **2026-04-21 当前仓库真实状态** 纠偏。

前文一些描述已经落后：当前 WebChat 前端 **并不是完全没有** method / skill 的生成、候选查看和 accept / reject 动作；问题在于这些入口 **分散、混放、缺少聚焦视图**，所以用户仍然会觉得“内容太多太乱，很难找到真正可发布的 method / skill”。

### 9.1 “经验消费总览”区块仍然只做查看和跳转

代码：

- `apps/web/public/app/features/memory-detail-render.js:298-460`

这里展示的是按真实 usage 聚合出来的热门 method / skill 双列看板。

当前动作仍然只有：

- Candidate
- Recent Task
- Open Artifact

所以它依然是：

- 使用统计看板
- 溯源入口
- freshness 风险提示入口

而不是：

- 生成器
- 审核台
- 发布台
- 正式 skill / method 的在线编辑器

### 9.2 candidate 详情面板其实已经有审核动作

candidate 详情面板代码：

- `apps/web/public/app/features/memory-viewer.js:3697-3775`

当前已经支持：

- 查看 candidate 内容
- 打开来源 task / memory / artifact / published path
- `接受并发布`
- `拒绝`
- skill freshness 的 stale 标记相关动作

也就是说，accept / reject / publish 这条链路，前端今天已经接出来了。

### 9.3 前端当前也已经有独立的“经验能力工作台”

代码：

- `apps/web/public/app/features/experience-workbench.js`
- `apps/web/public/index.html:399-485`

当前工作台已经会主动拉：

- `experience.candidate.list`
- `experience.candidate.get`
- `experience.candidate.accept`
- `experience.candidate.reject`
- `experience.candidate.generate`

其中：

- `经验候选` 页签：是 candidate 列表 + 右侧详情
- `经验消费总览` 页签：是 usage 聚合总览

所以“独立候选列表”在当前代码里其实已经存在。

### 9.4 真正的问题：入口分散且没有“只看可操作 draft”的聚焦视图

现在用户之所以仍然觉得难用，核心不是“完全没有按钮”，而是：

1. `经验候选` 把 draft / accepted / rejected 混在一起，信息密度高。
2. 真正最常用的“接受并发布”按钮只出现在右侧详情，不在列表主视图。
3. `经验消费总览` 的双列布局很好扫视，但它统计的是 usage，不是“待处理候选”。
4. “从 task 生成 candidate”“在候选详情里审核发布”“在 usage 看板里追溯”三件事分布在不同位置，用户心智负担高。

所以你的观感是对的：**当前 UI 有局部能力，但没有一个专门面向“可生成 / 可发布候选”的集中操作台。**

## 10. 最后总结

### 10.1 现在已经有的

- method / skill 使用回写与统计
- task -> method / skill candidate 生成
- 自动生成与 post-run learning review 生成
- candidate 查看
- candidate accept / reject
- accept 后发布到正式 methods / skills
- skill freshness 对“是否需要补丁 / 是否需要新增 skill”的诊断

### 10.2 当前“经验消费总览”的定位

它现在是：

- 使用统计看板
- 溯源入口
- freshness 风险提示入口

不是：

- 生成器
- 审核台
- 发布台
- 正式 skill / method 的在线编辑器

### 10.3 对你问题的直接回答

- `记忆查看 -> 经验消费总览` 背后，确实已经有 method 和 skill 的经验总结相关实现。
- 这些实现的主流程是：`task -> candidate(draft) -> accept/reject -> publish -> reuse -> usage stats`
- 最终结果可以用来生成 method 和 skill；skill 也可以借 freshness + pending candidate 信号来驱动“改进”
- 但当前“改进”不是直接 patch 正式 method / skill，而是通过新的 candidate 进入人工审阅和再发布
- 当前前端其实已经接出了部分操作，但入口分散，且缺少一个只聚焦 draft 候选的“能力获取”操作页签

## 11. 本次确认的 UI 方向：新增“能力获取”页签

基于当前仓库状态，本次更合适的最小可行方案不是继续把动作散落在 task detail / candidate detail / usage overview 里，而是：

1. 保留现有 `经验候选` 页签：
   - 继续作为完整候选列表
   - 保留 draft / accepted / rejected 的全量浏览能力
2. 保留现有 `经验消费总览` 页签：
   - 继续作为 usage 统计与追溯入口
   - 不把复杂审批逻辑强塞进这个看板
3. 在 `经验能力` 工作台新增一个 `能力获取` 页签：
   - 专门聚焦 **draft 候选**
   - 左列展示 method draft
   - 右列展示 skill draft
   - 直接提供 `查看详情 / 打开任务 / 接受并发布 / 拒绝` 这类高频操作

这样能最大化复用现有后端与当前页面视觉语言，同时显著降低“在海量候选里找可发布项”的成本。

## 12. 2026-04-21 文档状态更新与“能力获取”V1 方案

### 12.1 本次文档纠偏

截至 2026-04-21，下面这些说法已经不再准确：

- “前端没有 candidate list”
- “candidate 详情没有 accept / reject”
- “前端只支持查看和跳转”

当前真实状态是：

- 已有 `经验能力工作台`
- 已有 `能力获取`、`经验候选`、`经验消费总览` 三个页签
- `能力获取` 已放到第一个页签，并作为打开 `经验能力` 时的默认视图
- 已有 candidate generate / get / list / accept / reject / reject_bulk / stats 相关前端调用
- 已有 task detail 中的 generate method / generate skill 入口

此前缺少、现已在 V1 中补齐：

- 一个专门面向 **draft method / draft skill** 的集中操作页签
- 一个像 usage overview 那样“扫一眼就能做操作”的双列候选布局
- 大批量 draft 的单次批量拒绝入口

### 12.2 V1 目标

在 `经验能力` UI 中新增 `能力获取` 页签，用于专门展示所有 **可进行生成、发布等操作的 draft method / draft skill 候选**。

这个页签的目标不是替代：

- `经验候选` 的全量检索能力
- `经验消费总览` 的 usage 看板能力

而是补一个 **draft 候选操作台**。

### 12.3 V1 范围

V1 采用收敛策略：

- **默认只聚焦 `draft` 候选**
- `accepted / rejected` 继续留在原有 `经验候选` 页签中查看

原因：

- 先解决“太多太乱、找不到可发布项”的核心问题
- 避免把新页签做成又一个全量列表
- 让 `能力获取` 和 `经验候选` 的职责清晰分离

### 12.4 V1 交互设计

布局参考当前 `经验消费总览` 的蓝框双列 lane 结构：

- 左列：`Method`
- 右列：`Skill`

但每一行的内容改为“draft candidate 卡片”，不再显示 usage bar 的统计语义，而是显示：

- 标题
- 状态（固定为 draft）
- 来源 taskId
- 最近更新时间
- skill freshness（仅 skill）

按钮改为操作按钮：

- `查看详情`
- `打开任务`
- `接受并发布`
- `拒绝`

必要时可保留：

- `打开已发布产物`

但 V1 中 draft 候选通常还未发布，所以这个按钮不作为主按钮。

### 12.5 V1 实现原则

优先复用现有能力；仅在大批量 draft 操作确有必要时补最小后端支持：

1. 候选列表继续使用 `experience.candidate.list`
2. 顶部统计使用 `experience.candidate.stats`

### 12.1 问题分析

#### 问题一：当前 method / skill 自动生成内容不规范

#### 问题二：当前缺少可配置的确认开关

结论：

- 不需要新造审批体系
- 应复用现有 tool control / confirm 机制
- 只需要把“是否确认”做成运行时可配置策略

#### 问题三：当前只有“同 task 去重”，没有系统级去重

现状：

- 当前 candidate 去重只做到 `taskId + type`
- 代码在 `packages/belldandy-memory/src/experience-promoter.ts:14-18`

当前问题：

- 同一类经验可能已经存在 method / skill 正式资产
- 也可能已经存在相似 candidate
- 现在不会在系统级别阻止重复沉淀

结论：

- 需要增加 system-level dedup
- 但不建议一上来就做 embedding 相似度
- 第一阶段用“规则去重 + 相似提示”更稳

#### 问题四：调用设置缺 method 列表，但底层能力其实已经有

当前问题：

- `index.html` 和 `tool-settings.js` 只渲染了 `技能`，没有 `方法`
- method 不在调用设置面板里可见

结论：

- 这是前端 / query runtime 视图层缺口
- 不是底层没有 method inventory

#### 问题五：改进流程先不扩张

这个判断是合理的。

当前 skill 已经有：

相关代码：

- 发布

### 12.2 总体实现方案

#### 方案 A：规范生成模板

目标：

- method draft 对齐“方法论编写指南”
- skill draft 对齐 `skill-creator` 的 authoring 规范

method 生成规则调整：

- 内容固定包含：

  - `6. 成功案例`
  - `7. 相关资源`
  - `8. 更新记录`

skill 生成规则调整：

- frontmatter 必须稳定输出：
  - `name`
  - `description`

#### 方案 B：增加确认开关

建议新增环境变量：

- `BELLDANDY_METHOD_GENERATION_CONFIRM_REQUIRED`
- `BELLDANDY_SKILL_GENERATION_CONFIRM_REQUIRED`
- `BELLDANDY_METHOD_PUBLISH_CONFIRM_REQUIRED`
- `BELLDANDY_SKILL_PUBLISH_CONFIRM_REQUIRED`

- generation：`false`
- publish：`true`

原因：

- 生成 candidate 可回滚，风险低
- 发布正式资产影响更大，更适合默认确认

实现方式：

- 继续复用现有工具确认流
- 不新增独立 HITL 通道

#### 方案 C：增加系统级去重

建议三层去重：

1. 同 task 去重
2. 正式资产精确去重
3. 相似资产提示

method 去重输入：

- 文件名
- 标题
- summary
- slug

skill 去重输入：

- skill name
- published dir name
- title
- slug
- description

建议行为：

- 精确重复：默认不生成新 candidate
- 相似重复：给出相似项提示，但允许继续生成

建议不要做的事：

- 第一阶段不上 embedding / 向量召回
- 第一阶段不做“强制相似阻断”

#### 方案 D：调用设置补 method 列表

建议：

- 在 `tools.list` payload 中增加 `methods`
- 在调用设置里增加 `方法` tab

method tab 的定位：

- 列表展示
- 打开文件
- 查看摘要 / 标题 / 状态

不建议：

- 给 method 做启用/禁用 toggle

原因：

- skill 是可注入能力，适合 on/off
- method 是知识资产，更适合作为索引视图

#### 方案 E：最小可行 UI 落地

按上一节建议，落地以下入口：

Task Detail：

- `生成 method candidate`
- `生成 skill candidate`

Candidate Panel：

- `接受并发布`
- `拒绝`

Skill Freshness：

- `打开 patch candidate`
- `标记 stale / 取消 stale`

不做：

- 在 usage overview 直接审批
- 直接编辑正式 method / skill

### 12.3 更稳妥的工程建议

#### 建议一：method tab 不要做成“技能 tab 的复制品”

method 在系统里不是 toggle-able capability，而是 SOP 资产。

因此更合适的呈现方式是：

- 查看列表
- 打开文件
- 按标题 / 摘要 / 文件名搜索

而不是：

- 开关启用

#### 建议二：去重结果要区分“阻断”和“提示”

推荐：

- exact duplicate -> 阻断
- similar duplicate -> 提示

否则会导致：

- 有价值的 v2 / patch candidate 也被粗暴拦下

#### 建议三：现阶段不要直接 patch 正式文件

因为这会破坏当前系统已经成型的治理边界：

- candidate layer
- review / accept
- publish

短期内最稳的仍是 candidate 驱动更新。

#### 建议四：后续可单独补一个 dedup preview RPC

例如：

- `experience.candidate.check_duplicate`

这样 UI 在点击“生成”前可以先预览：

- 是否已存在
- 有哪些相似 method / skill

但这不是第一阶段必须项。

## 20. 2026-04-20 命名规则与发布前校验收紧

本轮目标：

- 把发布前校验落到真正的 publish 链路，而不是只靠 draft 生成阶段自觉

### 20.1 现在的命名规则

method 文件名：

- 新生成 / 新创建 / 新发布的 method 统一使用三段式：
  - `[目标对象]-[动作类型]-[细分].md`
- 中文名和英文名都必须遵守三段式；不再接受“任意中文安全文件名”
- candidate 生成 slug、candidate accept 时的正式发布文件名、`method_create` 的文件名校验，已统一到这条规则
- 自动生成时的实际策略：
  - 中文任务优先生成中文三段式
  - 英文任务优先生成英文三段式
  - 若标题 / slug 已经是合法三段式，则直接沿用
  - 若不是合法三段式，则基于 title / objective / summary / taskId 做启发式规范化
- 发布重名时，不再通过额外增加新的 `-segment` 打破三段式，而是把修订后缀压入第三段，例如：
  - `候选层-实现-闭环_task_exp_1.md`

skill 文件夹名：

- 目录名使用 canonical skill machine name
- 规则为：
  - 小写
  - ASCII
  - kebab-case
  - 不含空格
  - 不含冗余 `skill-` 前缀
- 目录名与 frontmatter `name` 强制对齐
- 若草稿里的 `name` 不规范，发布时会改写成 canonical name 后再写入 `SKILL.md`

### 20.2 现在的发布前校验

method publish 前校验：

- 必须存在一级标题
- 必须包含以下固定章节：
  - `0. 元信息`
  - `1. 触发条件`
  - `2. 适用场景`
  - `3. 执行步骤`
  - `4. 工具选择`
  - `5. 失败经验`
  - `6. 成功案例`
  - `7. 相关资源`
  - `8. 更新记录`

skill publish 前校验：

- frontmatter 必须至少有：
  - `name`
  - `description`
- 必须存在一级标题
- 必须包含以下固定章节：
  - `快速开始`
  - `决策路由`
  - `输入`
  - `输出`
  - `参考指引`
  - `NEVER`

### 21.6 与现有页面的关系

与“记忆查看”的关系：

- 记忆查看中保留当前最小入口，不立即移除
- 新页签上线后，candidate 的集中查看与审核主入口迁移到“经验能力”
- 记忆查看内的相关按钮可逐步补一个“在经验能力中打开”跳转

与“长期任务”的关系：

- 长期任务仍负责目标推进与建议恢复目标
- 若 goal 侧已经产生 method / skill 建议，可在后续把其跳转统一落到“经验能力”

与“调用设置 > 方法”tab 的关系：

- `方法` tab 继续面向已发布 methods 资产
- `经验能力` 面向未决 candidate 与最近生成结果
- 两者不要混做同一个面板，避免“草稿候选”和“正式资产”混杂

### 21.8 实施细节建议

状态模型建议：

- 不直接复用 `memoryViewerState`
- 新建 `experienceWorkbenchState`
- 至少包含：
  - `items`
  - `selectedId`
  - `selectedCandidate`
  - `filters`
  - `stats`
  - `requestToken`
  - `activeAgentId`
  - `pendingActionKey`

这样做的原因：

- 避免“记忆查看”筛选状态与“经验能力”互相污染
- 避免后续因为 tab 共享状态导致 UI 回跳或 detail 错位

渲染复用建议：

- candidate 详情的 HTML 片段可优先从现有 `memory-detail-render.js` 抽取或复用
- 但列表渲染与工作台外层布局建议放在独立 feature 中

交互建议：

- 左栏点击 `经验能力` 后，不自动改动“记忆查看”的 tab 状态
- 详情区若无选中项，默认展示“请先选择一个经验候选”
- 若 generate 结果命中已有 candidate，应沿用当前 dedup 确认逻辑，不做第二套实现

### 21.9 风险与兼容性

主要风险：

- candidate 相关逻辑当前散落在 memory feature 中，首轮拆分时容易出现重复绑定或状态不一致
- 如果直接深度复用 `memoryViewerState.selectedCandidate`，后续会让两个页面相互串状态
- 生成入口若设计过重，容易把“工作台”重新做成第二个“记忆查看”

兼容性判断：

- 本方案不改后端数据结构
- 不改现有 RPC 契约
- 不改发布规则
- 对已有用户流程基本兼容

推荐决策：

- `fix_now`

原因：

- 这是 UI 组织层面的主路径优化
- 底层能力已经存在，适合以最小风险做独立工作台收口

### 21.10 本轮建议的落地顺序

1. 先完成左侧按钮与 section 骨架
