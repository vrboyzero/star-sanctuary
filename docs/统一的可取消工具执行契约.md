# 统一的可取消工具执行契约

## 背景

当前 `tool-agent` 已经可以对部分 Hook 阶段做超时收敛，但真实工具执行仍主要通过：

- `packages/belldandy-agent/src/tool-agent.ts`
- `packages/belldandy-skills/src/executor.ts`

直接调用 `toolExecutor.execute(...)` 完成。

这意味着：

- 上层可以判断“本次 run 不再继续等”
- 但不能保证底层工具真的停止执行

如果直接在上层对 `toolExecutor.execute()` 使用 `Promise.race()` 一类的“假超时”做硬拦，会产生严重的行为漂移：

- 界面上已经显示超时 / 取消
- 但底层文件写入、外部请求、命令执行、子进程等副作用仍可能继续发生

因此，这一块不能靠局部补丁解决，需要单独设计统一的“可取消工具执行契约”。

## 为什么需要

统一可取消契约的目标不是“让超时提示更好看”，而是让：

- 用户看到的停止 / 超时状态
- Agent 运行状态
- 工具真实执行状态
- 最终落地副作用

尽可能保持一致。

如果没有统一契约，系统只能做到：

- 逻辑层放弃等待
- 但执行层继续运行

这会造成状态不一致、资源浪费和副作用不可控。

## 从用户角度的价值

用户侧最直接能感知到的收益有以下几类。

### 1. “停止”更可信

当用户点击停止、切换任务、重新发送、或系统触发超时时：

- 不是只有前端显示停了
- 而是底层工具也会收到取消信号并尽快结束

这能显著降低“明明停了却还在后台继续跑”的错觉和不信任感。

### 2. 结果状态更一致

没有取消契约时，最容易出现：

- 用户看到“超时 / 失败”
- 几秒后文件却被改了、命令却执行完了、外部系统却收到请求了

这会让用户难以判断系统到底有没有真的停下来。

统一契约的意义，就是尽量让：

- “显示为已停止”
- “实际也停止”

保持一致。

### 3. 减少后台资源浪费

如果长命令、HTTP 请求、子进程、流式读取在上层超时后仍继续运行，会持续占用：

- CPU
- 内存
- 文件句柄
- 网络连接
- 子进程资源

用户虽然看不到这些细节，但会直接感受到：

- 系统越来越慢
- 后续任务排队
- 停止后机器还在忙

### 4. 对高风险工具更安全

对于以下工具类型，取消语义尤其重要：

- 文件写入
- 命令执行
- 外部 HTTP/API 调用
- 数据库写入
- 长时间流式处理

如果这些工具不能被统一取消，用户对“停止”的控制权其实是假的。

### 5. 为后续能力打基础

未来很多体验能力都依赖统一取消语义，否则只能做成“表面可用”：

- 停止当前任务
- 子 Agent 超时后真实回收
- 切模型重试
- 切换策略后中断旧执行
- 长任务失败后的安全回滚与善后

## 为什么这轮没有直接做

这轮服务端审查的前提是：

- 不改坏现有功能正确性
- 不改变既有使用方式
- 不引入行为漂移

而统一可取消契约本质上是一个基础协议改造，不是单点修复。

如果在没有统一设计的情况下，直接给 `toolExecutor.execute()` 外层加“超时返回”，会出现：

- 上层状态先结束
- 下层副作用继续执行

这反而会比“老老实实等待”更危险。

所以本轮只先做了低风险收敛：

- Hook 超时边界
- 同会话 run 串行化
- 子 Agent timeout 终态幂等

但没有对真实工具执行语义做激进改造。

## 需要解决的核心问题

如果未来要补这块，至少要先明确以下问题。

### 1. 取消信号如何传递

建议统一采用 `AbortSignal` 或等价 cancel token，并从：

- `tool-agent`
- `toolExecutor`
- `ToolContext`
- 具体工具实现

全链路传递下去。

### 2. 不同工具类型如何响应取消

不同类型工具对取消的支持方式不同：

- HTTP 请求类：通常可通过 `AbortController` 中止
- 子进程类：需要显式 kill，并处理残留输出
- 文件流类：需要中止读写并关闭句柄
- 第三方 SDK：要确认是否原生支持取消

不能假设“所有工具都能被同一种方式停止”。

### 3. 取消后的语义如何定义

必须明确取消后的语义边界，例如：

- 是“保证无副作用”
- 还是“尽力停止，但可能已有部分副作用落地”
- 取消后如何记录 tool result / audit / transcript

这部分必须写清楚，否则使用者会误解“取消”等于“什么都没发生”。

### 4. 如何保持向后兼容

仓内已有很多工具实现，不能一次性全部重写。

更现实的做法应是：

- 先新增契约
- 允许旧工具暂时走“不可取消但可标识”的兼容路径
- 再逐类迁移关键工具

### 5. 取消和超时的关系

需要区分：

- 用户主动取消
- 系统超时中止
- 上层 run 已结束但底层工具未收尾

这三类场景在日志、审计、前端提示和状态回写上的语义不应完全混在一起。

## 建议的最小落地顺序

建议按以下顺序推进，而不是一次性全仓改造。

### 第一阶段：先补契约，不强推全量实现

- 在工具执行上下文中引入统一取消信号
- 在 `toolExecutor.execute()` 链路上把取消能力传下去
- 增加“该工具是否支持取消”的显式标识

### 第二阶段：优先覆盖高风险/长耗时工具

优先级建议：

- 外部 HTTP 请求类工具
- 命令执行 / 子进程类工具
- 长时间流式处理类工具
- 任何可能持续数十秒以上的工具

### 第三阶段：补可见性与审计

需要同步明确：

- 取消后的 tool result 结构
- transcript 是否记录“已取消”
- audit log 如何标识取消来源
- 前端如何区分“失败”和“取消”

## 当前建议结论

结论很明确：

- `toolExecutor.execute()` 值得设计统一的可取消工具执行契约
- 这项工作对用户真实体验有明显价值
- 但它应当单独立项，而不是混入本轮性能收敛中顺手硬补

当前更稳妥的策略是：

- 先把已确认的 Hook / 队列 / timeout / 状态漂移问题收敛
- 再把“真实工具执行可取消”作为下一阶段基础设施改造推进

## 未来落地时建议优先验证的用户场景

- 用户手动点击停止后，长工具是否真的停止
- 子 Agent timeout 后，底层工具是否还能继续产生副作用
- 同一会话快速重试时，旧工具是否会被正确中止
- 取消后前端展示、audit、transcript 是否保持一致
- 取消中的文件写入 / 命令执行 / 外部 API 是否存在半完成状态

## 设计草案 v0

本节目标不是一次性定最终方案，而是给出一个足够稳定、可拆任务的最小接口方向。

设计原则：

- 默认保持现有工具调用方式可兼容
- 先把取消能力“传下去”，再逐个工具补“真正响应取消”
- 不把“超时”和“取消”混成同一种错误
- 不伪造“无副作用”承诺

## v0 设计目标

v0 只解决以下 4 件事：

1. Agent / ToolExecutor / ToolContext 全链路可传递取消信号  
2. 工具实现可以显式声明“支持取消”或“暂不支持取消”  
3. 上层可以区分：
   - 用户主动取消
   - 系统超时取消
   - 普通执行失败
4. transcript / audit / 前端事件具备最小一致性

v0 不强求：

- 所有工具立刻支持真正中止
- 取消后自动回滚所有副作用
- 一次性统一所有第三方 SDK 的中断行为

## 建议接口变更

### 1. ToolContext 增加取消信号

建议在工具上下文中增加：

```ts
type ToolCancellationReason = "user_cancelled" | "timeout" | "session_closed" | "superseded";

interface ToolContext {
  conversationId: string;
  workspaceRoot: string;
  signal?: AbortSignal;
  cancellationReason?: ToolCancellationReason;
}
```

作用：

- 所有工具都能通过统一入口感知取消
- 不要求每个工具自定义一套取消参数

### 2. ToolExecutor.execute 增加可选执行控制参数

建议把执行入口扩成：

```ts
interface ToolExecutionOptions {
  signal?: AbortSignal;
  cancellationReason?: ToolCancellationReason;
  timeoutMs?: number;
}

execute(
  request: ToolCallRequest,
  conversationId: string,
  agentId?: string,
  userUuid?: string,
  senderInfo?: any,
  roomContext?: any,
  execution?: ToolExecutionOptions,
): Promise<ToolCallResult>
```

作用：

- 取消信号由上层统一传入
- 工具执行器可以把同一信号下发到 `ToolContext`
- 后续若要做 per-tool timeout，也有承载位置

### 3. Tool 定义增加取消能力声明

建议新增能力标识：

```ts
interface ToolExecutionCapabilities {
  cancellable?: boolean;
  cancellationMode?: "cooperative" | "best_effort" | "none";
}

interface Tool {
  definition: ToolDefinition;
  capabilities?: ToolExecutionCapabilities;
  execute(args: JsonObject, context: ToolContext): Promise<ToolCallResult>;
}
```

建议语义：

- `cooperative`
  - 工具能真实响应 `AbortSignal`
  - 收到取消后会尽快停止
- `best_effort`
  - 工具会尽力停止，但不能保证已发出的外部副作用可撤回
- `none`
  - 当前工具不支持取消，只能等待完成

这样前端和日志可以更诚实地表达：

- “已请求取消”
- “工具支持取消并已中止”
- “工具不支持取消，等待收尾”

### 4. ToolCallResult 增加取消态

建议扩展结果语义，而不是把取消塞进普通错误：

```ts
interface ToolCallResult {
  id: string;
  name: string;
  success: boolean;
  cancelled?: boolean;
  cancellationReason?: ToolCancellationReason;
  output: string;
  error?: string;
  durationMs: number;
}
```

建议解释：

- `success: false + cancelled: true`
  - 表示这是一次被取消的执行，不是普通业务失败
- `success: false + cancelled: false/undefined`
  - 才表示真正失败

这样后续：

- transcript
- audit log
- UI 提示
- 自动重试逻辑

都能基于取消态做分流。

## Agent 侧建议接口

### 1. AgentRunInput 增加 signal

建议：

```ts
interface AgentRunInput {
  conversationId?: string;
  text?: string;
  signal?: AbortSignal;
}
```

作用：

- 子 Agent timeout
- 用户点击停止
- 会话关闭
- 新 run 覆盖旧 run

都可以通过统一信号传到工具层。

### 2. ToolEnabledAgent 内部为每次 tool call 复用同一信号

建议运行模型和调用工具时共享同一 run 级 signal：

- 模型请求收到取消
- 工具调用收到取消
- Hook 可以读取取消原因

这样“整次 run 已结束，但某个工具还在后台继续跑”的概率会显著降低。

## ToolExecutor v0 行为建议

### 1. 统一创建取消上下文

`ToolExecutor.execute()` 在执行工具时：

- 把 `signal` 注入 `ToolContext`
- 若 `signal.aborted === true`，可在真正执行前直接返回取消结果

建议行为：

```ts
if (execution?.signal?.aborted) {
  return {
    id: request.id,
    name: request.name,
    success: false,
    cancelled: true,
    cancellationReason: execution.cancellationReason ?? "user_cancelled",
    output: "",
    error: "Tool execution cancelled before start",
    durationMs: Date.now() - start,
  };
}
```

### 2. 不强行 Promise.race 假中止工具主体

v0 阶段建议：

- `ToolExecutor` 可以监听取消
- 但不要对不支持取消的工具做“表面超时返回”

否则会出现：

- 外层已返回 cancelled
- 内层还在继续执行副作用

这会破坏当前系统最重要的行为一致性。

### 3. 对支持取消的工具允许显式中止

对于 `capabilities.cancellationMode === "cooperative"` 的工具：

- 工具内部应主动监听 `context.signal`
- 在合适边界主动抛出 `AbortError` 或返回取消结果

建议工具实现侧遵循：

```ts
if (context.signal?.aborted) {
  throw new DOMException("Aborted", "AbortError");
}
```

或等价错误类型。

`ToolExecutor` 再统一把这类错误映射为取消结果。

## transcript / audit / 前端事件建议

### 1. transcript

建议：

- 取消的工具调用仍保留 transcript
- 但明确标记为 cancelled
- 不把取消当成普通失败文本塞进自然语言正文

示意：

```json
{
  "role": "tool",
  "tool_call_id": "call-1",
  "content": "[cancelled] reason=timeout"
}
```

### 2. audit

建议 audit 至少包含：

- `cancelled: true/false`
- `cancellationReason`
- `tool supports cancellation`
- `cancellationMode`

这样后续才能统计：

- 哪些工具最常被取消
- 哪些工具超时最多
- 哪些工具“不支持取消”导致用户体验差

### 3. 前端事件

建议前端不要把取消全部显示成“失败”。

最小区分：

- 失败：工具执行失败
- 已取消：用户或系统中止
- 正在取消：已发送取消请求，但工具仍在收尾

这对用户理解系统状态很重要。

## 兼容迁移策略

建议按三层迁移，而不是一次性改全仓。

### 第一层：契约落地

- 给 `ToolContext` / `ToolExecutor.execute()` / `AgentRunInput` 加 `signal`
- 给 `ToolCallResult` 加取消态字段
- 给 `Tool` 加 `capabilities.cancellationMode`

这一步可以先不要求所有工具真的响应。

### 第二层：高风险工具优先迁移

优先工具类型：

- HTTP 请求类
- 命令执行类
- 长时文件流类
- 外部系统写入类

### 第三层：补前端和审计

- UI 显示取消态
- transcript 写入取消态
- audit 增加取消元数据

## v0 之后的升级方向

如果 v0 稳定，后面可以考虑：

- 为工具定义区分“可中止”与“可回滚”
- 为高风险工具提供 `onCancel` / `cleanup` 钩子
- 对长工具显示“正在取消”进度
- 对不可取消工具在 UI 上提前提示

## 建议拆分任务

后续如果真正立项，建议拆成以下子任务：

1. 扩展 `ToolContext / ToolCallResult / Tool` 类型定义  
2. 扩展 `ToolExecutor.execute()` 入口与上下文透传  
3. 扩展 `AgentRunInput` 与 `ToolEnabledAgent` run 级 cancel 透传  
4. 先迁移 HTTP 类工具  
5. 再迁移命令执行 / 子进程类工具  
6. 补 transcript / audit / 前端取消态展示  
7. 补定向测试与故障注入测试

## 当前建议结论（v0）

推荐方向如下：

- 要做统一可取消契约
- 先做“信号透传 + 状态建模”
- 暂不在 v0 阶段强推“所有工具都可真实中止”
- 任何不支持取消的工具，都必须在契约层显式暴露，而不是默认假装可取消

这样既能给未来补完留出稳定基础，也不会为了追求“看起来能取消”而引入新的行为漂移。
