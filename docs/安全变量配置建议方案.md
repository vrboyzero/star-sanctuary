# 安全变量配置建议方案

更新时间：2026-03-18

## 先说结论

如果你现在要一套默认推荐，而不是逐项微调：

- 安全配置：优先采用“平衡推荐档”
- 如果你只是本机自用、几乎不做外部集成：优先采用“最保守档”
- 如果你要接社区、Webhook、浏览器自动化、MCP、跨目录落盘：只能采用“受控开放档”，不要只开开关不补约束

这次建议只覆盖和下面几类边界直接相关的变量：

- 文件读写边界
- 外部网络入口与主动出站
- 工具执行权限
- 浏览器自动化访问范围
- WebSocket / Community API / Webhook 鉴权与暴露面

不纳入本次主分析的：

- 记忆质量、召回效果、Token 成本优化
- 纯模型能力变量（如模型名称优劣）
- TTS / STT 的体验类变量

## 关键实现结论

### 1. `BELLDANDY_HOST=0.0.0.0` 配合 `BELLDANDY_AUTH_MODE=none` 会被直接拦截

这条不是“建议”，而是代码硬保护。

结论：

- 公网或局域网监听时，必须配 token 或 password
- `none` 只适合本机监听

### 2. `BELLDANDY_COMMUNITY_API_ENABLED=true` 不能和 `AUTH_MODE=none` 共存

Gateway 启动时就会拒绝这种组合。

结论：

- 只要开启 `/api/message`
- 就必须启用认证
- 推荐使用独立的 `BELLDANDY_COMMUNITY_API_TOKEN`

### 3. `BELLDANDY_ALLOWED_ORIGINS` 只管 WebSocket Origin，不等于“整个服务都白名单化”

实际行为是：

- 本机监听且未配置白名单时，默认允许 `localhost/127.0.0.1`
- 公网监听且未配置白名单时，WebSocket 会全部拒绝
- 但 REST 入口仍然走各自的鉴权逻辑，不受这个变量直接控制

结论：

- 不要把它误当成全站 ACL
- 它主要是防 WebSocket 跨站连接

### 4. 已配对客户端拥有很高权限，因为它可以直接读写 `.env` 原文

这点很重要。

虽然 `config.update` 只有白名单字段可改，但 Web 前端实际还在用：

- `config.readRaw`
- `config.writeRaw`

这两条会直接读取和覆盖 `.env` 原文；它们受 pairing 保护，但一旦客户端完成配对，就具备原始配置编辑权限。

结论：

- Pairing 不是“轻权限登录”
- Pairing 通过后，基本就是高权限本地管理端
- 所以配对码和 allowlist 也属于安全边界的一部分

### 5. `BELLDANDY_DANGEROUS_TOOLS_ENABLED=false` 只会关闭 `run_command`

它不会关闭这些核心工具：

- `file_read`
- `file_write`
- `file_delete`
- `apply_patch`
- `web_fetch`

只要 `BELLDANDY_TOOLS_ENABLED=true`，这些工具仍然可用。

结论：

- `DANGEROUS_TOOLS_ENABLED` 不是“安全总开关”
- 真正的总闸门是 `BELLDANDY_TOOLS_ENABLED`
- 真正的细粒度边界要靠 `BELLDANDY_TOOLS_POLICY_FILE`

### 6. `BELLDANDY_TOOLS_POLICY_FILE` 是最关键的细粒度安全变量之一

它实际可以覆盖：

- `allowedPaths`
- `deniedPaths`
- `allowedDomains`
- `deniedDomains`
- `maxTimeoutMs`
- `maxResponseBytes`
- `exec.extraSafelist`
- `exec.extraBlocklist`
- `fileWrite.allowedExtensions`
- `fileWrite.allowDotFiles`
- `fileWrite.allowBinary`

结论：

- 只开工具、不配策略，安全边界会很粗
- 尤其是你准备开启 `run_command`、跨目录写文件、开放网页抓取时，策略文件几乎是必需品

### 7. `BELLDANDY_EXTRA_WORKSPACE_ROOTS` 会显著扩大文件工具和 Filesystem MCP 的可见范围

实际影响：

- `file_read / file_write / file_delete / list_files` 会把这些额外根目录当作合法范围
- Filesystem MCP 也会继承这些额外根目录

但它不会进入当前统一记忆建库主边界。

结论：

- 这是“扩大文件权限范围”的变量
- 不是“补充记忆源”的变量
- 最忌讳直接配成 `E:\`、`D:\` 这种大盘根目录

### 8. `BELLDANDY_WORKSPACE_DIR` 主要是 Docker 挂载变量，不是当前运行时文件权限总开关

它主要出现在 Docker 相关文件里，用于把宿主目录挂到容器内。

当前运行时实际主边界更看重：

- `BELLDANDY_STATE_DIR`
- `BELLDANDY_EXTRA_WORKSPACE_ROOTS`

结论：

- 不要误以为改了 `BELLDANDY_WORKSPACE_DIR` 就能精确收缩工具读写边界
- 单机 / 源码运行场景下，它不是最关键的安全变量

### 9. 浏览器 Relay 本身只监听 `127.0.0.1`，但浏览器工具默认不限制访问哪些网站

两层要分开看：

- Relay 服务自身只绑本机，风险相对可控
- 但浏览器工具如果启用，默认并不会限制站点

代码里确实支持：

- `BELLDANDY_BROWSER_ALLOWED_DOMAINS`
- `BELLDANDY_BROWSER_DENIED_DOMAINS`

但这两个变量当前没有写进 `.env.example`。

结论：

- 浏览器自动化一旦开启，建议同时配置浏览器域名白名单
- 否则就是“本机浏览器代理安全，但目标网站范围不受控”

### 10. `web_fetch` 的安全性比浏览器工具更强一些，但也要靠策略文件补足

`web_fetch` 自带：

- 仅允许 `http/https`
- 拒绝内网地址
- DNS 二次校验防 DNS rebinding
- 受 `allowedDomains / deniedDomains / timeout / response size` 限制

结论：

- 如果只是抓取网页内容，`web_fetch` 比浏览器自动化更容易控
- 推荐优先限制 `allowedDomains`

### 11. `BELLDANDY_ATTACHMENT_MAX_*` 控制的是上传体积，不是执行权限

它们的作用主要是：

- 限制单文件大小
- 限制单次消息附件总大小

结论：

- 这是“资源滥用 / 大文件注入”防线
- 不是权限隔离变量
- 但仍然建议在公网或多人环境中收紧

### 12. Webhook 的核心密钥不在 `.env`，而在 `webhooks.json`

环境变量只决定：

- `BELLDANDY_WEBHOOK_CONFIG_PATH`
- `BELLDANDY_WEBHOOK_IDEMPOTENCY_WINDOW_MS`

真正的 webhook token 在配置文件里。

结论：

- 配了 path 不等于配置了鉴权
- 真正的安全性取决于 `webhooks.json` 中每条 rule 的 token 质量

## 变量逐项评估

| 变量 | 实际作用 | 主要风险 | 建议 |
| --- | --- | --- | --- |
| `BELLDANDY_HOST` | 服务监听地址 | 暴露面扩大 | 默认 `127.0.0.1` |
| `BELLDANDY_AUTH_MODE` | WS/管理接口认证模式 | `none` 在公网不可接受 | 默认 `token` 更稳 |
| `BELLDANDY_AUTH_TOKEN` | 主认证 token | 泄露即高权限访问 | 强随机、不要复用 |
| `BELLDANDY_AUTH_PASSWORD` | 密码认证 | 易被弱口令拖垮 | 除非明确需要，否则不如 token |
| `BELLDANDY_ALLOWED_ORIGINS` | WS Origin 白名单 | 误以为覆盖全部入口 | 公网部署必须显式配置 |
| `BELLDANDY_COMMUNITY_API_ENABLED` | 开 `/api/message` | 增加 HTTP 攻击面 | 不用就关 |
| `BELLDANDY_COMMUNITY_API_TOKEN` | `/api/message` Bearer token | 若复用主 token，权限边界模糊 | 建议独立设置 |
| `BELLDANDY_TOKEN_USAGE_UPLOAD_ENABLED` | 向外上报 token 用量 | 额外出站网络 | 无需对接 office 就关 |
| `BELLDANDY_UPDATE_CHECK` | 启动时访问 GitHub 检查版本 | 多一条出站请求 | 极保守环境可关 |
| `BELLDANDY_TOOLS_ENABLED` | 工具总开关 | 开启后默认具备文件/网络工具 | 不需要工具就直接关 |
| `BELLDANDY_DANGEROUS_TOOLS_ENABLED` | 启用 `run_command` | 命令执行风险最高 | 默认保持 `false` |
| `BELLDANDY_TOOLS_POLICY_FILE` | 工具细粒度策略 | 不配置时边界过宽 | 只要开工具就值得配 |
| `BELLDANDY_STATE_DIR` | 主状态目录 | 配到共享盘或大目录会扩大暴露面 | 用独立专用目录 |
| `BELLDANDY_EXTRA_WORKSPACE_ROOTS` | 扩展文件工具根目录 | 横向扩大读写面 | 只配最小必要目录 |
| `BELLDANDY_WORKSPACE_DIR` | Docker 挂载工作区 | 容易被误当运行时安全变量 | 主要按 Docker 语义理解 |
| `BELLDANDY_MCP_ENABLED` | 开启 MCP 集成 | 引入额外外部工具与连接 | 只启用可信 MCP |
| `BELLDANDY_BROWSER_RELAY_ENABLED` | 开启浏览器自动化 relay | 浏览器控制能力上升 | 不用就关 |
| `BELLDANDY_BROWSER_ALLOWED_DOMAINS` | 浏览器白名单 | 未配置时浏览器站点范围过宽 | 开浏览器时建议配置 |
| `BELLDANDY_BROWSER_DENIED_DOMAINS` | 浏览器黑名单 | 默认空则不拦截危险域 | 可作为补充黑名单 |
| `BELLDANDY_WEBHOOK_CONFIG_PATH` | Webhook 规则文件路径 | 指向不受控文件会引入风险 | 放在 stateDir 下 |
| `BELLDANDY_WEBHOOK_IDEMPOTENCY_WINDOW_MS` | Webhook 幂等窗口 | 太短会重复执行，太长会压重试 | 保持 10 分钟左右 |
| `BELLDANDY_ATTACHMENT_MAX_FILE_BYTES` | 单文件附件上限 | 大文件拖垮内存/上下文 | 公网场景建议更低 |
| `BELLDANDY_ATTACHMENT_MAX_TOTAL_BYTES` | 单次附件总量 | 批量注入负载过大 | 多人环境建议更低 |

## 三挡安全配置建议

下面三挡不是“谁先进谁落后”，而是三种不同的风险接受度。

### A. 最保守档

适合：

- 只在自己电脑本机使用
- 不需要社区接入、Webhook、浏览器自动化、MCP
- 更在意收缩攻击面，而不是功能全开

建议配置：

```env
BELLDANDY_HOST=127.0.0.1
BELLDANDY_AUTH_MODE=token
BELLDANDY_AUTH_TOKEN=替换为强随机 token

BELLDANDY_ALLOWED_ORIGINS=http://127.0.0.1:28889,http://localhost:28889

BELLDANDY_COMMUNITY_API_ENABLED=false
BELLDANDY_TOKEN_USAGE_UPLOAD_ENABLED=false
BELLDANDY_UPDATE_CHECK=false

BELLDANDY_TOOLS_ENABLED=true
BELLDANDY_DANGEROUS_TOOLS_ENABLED=false
BELLDANDY_TOOLS_POLICY_FILE=~/.star_sanctuary/tools-policy.strict.json

BELLDANDY_MCP_ENABLED=false
BELLDANDY_BROWSER_RELAY_ENABLED=false
BELLDANDY_HEARTBEAT_ENABLED=false
BELLDANDY_CRON_ENABLED=false

BELLDANDY_STATE_DIR=~/.star_sanctuary
# BELLDANDY_EXTRA_WORKSPACE_ROOTS=

BELLDANDY_ATTACHMENT_MAX_FILE_BYTES=5242880
BELLDANDY_ATTACHMENT_MAX_TOTAL_BYTES=10485760
```

这一档的重点：

- 只监听本机
- 仍然要求 token，而不是 `none`
- 不开启额外 HTTP 入口
- 不给浏览器自动化和 MCP
- 不开放额外根目录
- 附件限制更紧

配套建议：

- `tools-policy.strict.json` 至少要限制可写目录和外网域名
- Pairing 通过的客户端也要视为高权限客户端

### B. 平衡推荐档

适合：

- 日常开发主力机
- 需要工具、方法论、文件编辑，但不想把面开太大
- 可能会用少量额外目录、定时任务、心跳，但不想开放社区 API

建议配置：

```env
BELLDANDY_HOST=127.0.0.1
BELLDANDY_AUTH_MODE=token
BELLDANDY_AUTH_TOKEN=替换为强随机 token

BELLDANDY_ALLOWED_ORIGINS=http://127.0.0.1:28889,http://localhost:28889,http://localhost:5173

BELLDANDY_COMMUNITY_API_ENABLED=false
BELLDANDY_TOKEN_USAGE_UPLOAD_ENABLED=false
BELLDANDY_UPDATE_CHECK=true

BELLDANDY_TOOLS_ENABLED=true
BELLDANDY_DANGEROUS_TOOLS_ENABLED=false
BELLDANDY_TOOLS_POLICY_FILE=~/.star_sanctuary/tools-policy.balanced.json

BELLDANDY_MCP_ENABLED=false
BELLDANDY_BROWSER_RELAY_ENABLED=false

BELLDANDY_HEARTBEAT_ENABLED=true
BELLDANDY_HEARTBEAT_INTERVAL=30m
BELLDANDY_HEARTBEAT_ACTIVE_HOURS=08:00-23:00
BELLDANDY_CRON_ENABLED=true

BELLDANDY_STATE_DIR=~/.star_sanctuary
BELLDANDY_EXTRA_WORKSPACE_ROOTS=E:\project\your-active-project

BELLDANDY_ATTACHMENT_MAX_FILE_BYTES=10485760
BELLDANDY_ATTACHMENT_MAX_TOTAL_BYTES=31457280
```

这一档的重点：

- 仍然坚持本机监听
- 允许常规工具和日常自动化
- `run_command` 仍然关闭
- 只给一个或少数明确的额外工作目录
- 保留心跳和 cron，但不开放对外 HTTP 入口

这是我最推荐的默认档位，因为：

- 功能完整度够高
- 风险面比“集成全开”小得多
- 配置复杂度也可控

### C. 受控开放档

适合：

- 需要接社区、Webhook、浏览器自动化、MCP 或跨目录执行
- 接受更多集成能力，但愿意显式补上约束
- 有明确的“可信来源域名 / 可信目录 / 可信客户端”模型

建议配置：

```env
BELLDANDY_HOST=0.0.0.0
BELLDANDY_AUTH_MODE=token
BELLDANDY_AUTH_TOKEN=替换为强随机 token

BELLDANDY_ALLOWED_ORIGINS=https://console.example.com,https://ops.example.com

BELLDANDY_COMMUNITY_API_ENABLED=true
BELLDANDY_COMMUNITY_API_TOKEN=替换为独立的 community token

BELLDANDY_TOKEN_USAGE_UPLOAD_ENABLED=true
# BELLDANDY_TOKEN_USAGE_UPLOAD_URL=https://office.example.com/api/internal/token-usage
# BELLDANDY_TOKEN_USAGE_UPLOAD_APIKEY=gro_xxx

BELLDANDY_TOOLS_ENABLED=true
BELLDANDY_DANGEROUS_TOOLS_ENABLED=true
BELLDANDY_TOOLS_POLICY_FILE=~/.star_sanctuary/tools-policy.open.json

BELLDANDY_MCP_ENABLED=true
BELLDANDY_BROWSER_RELAY_ENABLED=true
BELLDANDY_RELAY_PORT=28892
BELLDANDY_BROWSER_ALLOWED_DOMAINS=docs.example.com,github.com,developer.mozilla.org
BELLDANDY_BROWSER_DENIED_DOMAINS=mail.google.com,drive.google.com,onedrive.live.com

BELLDANDY_WEBHOOK_CONFIG_PATH=~/.star_sanctuary/webhooks.json
BELLDANDY_WEBHOOK_IDEMPOTENCY_WINDOW_MS=600000

BELLDANDY_STATE_DIR=~/.star_sanctuary
BELLDANDY_EXTRA_WORKSPACE_ROOTS=E:\project\project-a,E:\project\project-b

BELLDANDY_ATTACHMENT_MAX_FILE_BYTES=8388608
BELLDANDY_ATTACHMENT_MAX_TOTAL_BYTES=20971520
```

这一档的重点不是“全开”，而是“开了就补约束”：

- 公网监听必须配 token
- Community API 用独立 token
- 浏览器工具必须配域名白名单
- `run_command` 开启时必须配工具策略
- 额外工作区根目录必须是明确项目目录，不要是盘符根目录
- Webhook token 要在 `webhooks.json` 里单独维护

## 三挡配套策略建议

### A 档策略重点

- `allowedPaths` 只放：
  - `.`
  - `memory`
  - `methods`
  - `facets`
- `deniedPaths` 至少保留：
  - `.env`
  - `.git`
  - `node_modules`
  - `logs`
  - `sessions`
- `allowedDomains` 尽量只配文档站、代码托管站、你自己的接口域名

### B 档策略重点

- `allowedPaths` 可以覆盖当前项目目录
- `allowedDomains` 可以覆盖常用文档站和你自己的业务域
- `fileWrite.allowedExtensions` 可限制为：
  - `.md`
  - `.txt`
  - `.json`
  - `.yaml`
  - `.yml`
  - `.ts`
  - `.tsx`
  - `.js`
  - `.jsx`
  - `.css`
  - `.html`

### C 档策略重点

- `exec.extraBlocklist` 要显式补上你不接受的命令
- `allowedPaths` 要明确列目录，不要偷懒写整盘
- `allowedDomains` 和 `BELLDANDY_BROWSER_ALLOWED_DOMAINS` 要分别维护
- `fileWrite.allowBinary` 默认仍建议 `false`

## 一个最小的策略文件参考

只要你准备长期启用工具系统，我建议至少配一份类似下面的策略文件：

```json
{
  "allowedPaths": [
    ".",
    "memory",
    "methods",
    "facets"
  ],
  "deniedPaths": [
    ".git",
    "node_modules",
    ".env",
    "logs",
    "sessions"
  ],
  "allowedDomains": [
    "github.com",
    "raw.githubusercontent.com",
    "developer.mozilla.org"
  ],
  "deniedDomains": [],
  "maxTimeoutMs": 15000,
  "maxResponseBytes": 262144,
  "exec": {
    "extraBlocklist": [
      "powershell",
      "powershell.exe",
      "pwsh",
      "curl",
      "wget",
      "scp",
      "ssh"
    ]
  },
  "fileWrite": {
    "allowedExtensions": [
      ".md",
      ".txt",
      ".json",
      ".ts",
      ".tsx",
      ".js"
    ],
    "allowDotFiles": false,
    "allowBinary": false
  }
}
```

如果你后续真的要开 `run_command`，再按场景逐步放开，而不是一开始就全放。

## 我不建议依赖的“伪安全感”变量

### 1. `BELLDANDY_WORKSPACE_DIR`

它更多是 Docker 容器挂载入口，不是当前源码运行时的主要权限隔离变量。

### 2. `BELLDANDY_DANGEROUS_TOOLS_ENABLED=false`

它只能说明 `run_command` 没开，不代表文件写入、删除、抓网页、打补丁都被关了。

### 3. `BELLDANDY_ALLOWED_ORIGINS`

它只能说明 WebSocket Origin 受控，不代表 Community API / Webhook / 其他 HTTP 入口也自动受控。

## 默认推荐组合

如果你现在只要一套可以直接上手的默认值，我建议：

- 采用“平衡推荐档”
- `BELLDANDY_TOOLS_ENABLED=true`
- `BELLDANDY_DANGEROUS_TOOLS_ENABLED=false`
- `BELLDANDY_TOOLS_POLICY_FILE` 一定补上
- `BELLDANDY_EXTRA_WORKSPACE_ROOTS` 只给当前活跃项目目录
- `BELLDANDY_COMMUNITY_API_ENABLED=false`
- `BELLDANDY_MCP_ENABLED=false`
- `BELLDANDY_BROWSER_RELAY_ENABLED=false`

这套在当前实现里最稳：

- 功能够用
- 暴露面不大
- 不会因为“顺手开了一个集成”把安全边界一下子打开

## 额外说明

当前代码检查里，我认为最需要记住的不是“某个变量默认值是什么”，而是下面三件事：

1. Pairing 通过后的 Web 客户端权限很高，因为能直接编辑 `.env`
2. `EXTRA_WORKSPACE_ROOTS` 会真实扩大文件和 MCP 文件系统边界
3. 浏览器域名白名单变量已经有代码支持，但 `.env.example` 还没把它们写出来，容易被忽略

如果你下一步愿意继续，我建议直接补第二份配套文档：

- `tools-policy` 三挡 JSON 策略建议

这样安全变量和工具策略就能真正配成一套。
